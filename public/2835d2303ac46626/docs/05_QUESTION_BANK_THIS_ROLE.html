PART 1 — ML / LLM / NLP TECHNICAL (Stage 1 Priority)

1A. Transformer Architecture Deep-Dive

Q1. Walk me through the Transformer architecture. What are its key components, and why does each matter? [HF]

Model Answer:

The modern decoder-only Transformer (the standard for all major LLMs — LiLiuM, Llama, GPT) is built around five components:

  1. Token + Positional Embeddings. Each token maps to a learnable vector; position is encoded separately (see Q3 on RoPE). This separates token identity from position.

  2. Multi-Head Self-Attention (MHSA). For each head h, project input X into Q, K, V with separate weight matrices. Compute: Attention(Q,K,V) = softmax(QKᵀ / √d_k) · V Scaling by √d_k prevents logit explosion in high-dimensional spaces. Running H heads in parallel lets the model attend to multiple relationship types simultaneously (syntax, semantics, co-reference). Outputs are concatenated and projected.

  3. Feed-Forward Network (FFN). Two-layer MLP applied token-independently. In modern models (LiLiuM, Llama): SwiGLU activation (SwiGLU(x) = (W₁x ⊙ SiLU(W_gate x)) · W₂). Provides non-linearity and feature mixing within each position. The FFN is where most "stored facts" live — it acts as a key-value memory.

  4. Residual Connections + Pre-LayerNorm (or RMSNorm). Adding the input to each sublayer's output: prevents vanishing gradients in deep stacks. Pre-norm (normalize before the sublayer, not after) is now universal — it allows learning rates to be higher and avoids the post-norm instability that plagued BERT. LiLiuM uses RMSNorm (x / RMS(x), no mean-centering) — roughly half the compute of LayerNorm with no accuracy loss.

  5. Causal Mask. In decoder-only LMs: the attention mask prevents position i from attending to positions j > i, enforcing left-to-right generation.

Key strengths over RNN/LSTM: Transformers process the full sequence in parallel (no sequential hidden state), enabling O(n) computation in depth with O(n²) attention — but the parallelism is what makes GPU utilization efficient. Long-range dependencies require only one attention step; in RNNs they require n sequential steps.

Tony connection: The attention mechanism is the foundation for your LLM Router — you route based on model capability, which is itself learned from attention-based representations of query complexity.

Q2. What is multi-head attention, and why is it preferable to a single large attention head? [HF]

Model Answer:

Single-head attention with d-dimensional Q/K/V gives one weighted blend of all values. This conflates multiple types of relationships into one mixture.

Multi-head attention (MHA) projects into H lower-dimensional subspaces (each d/H dimensions), computes attention independently, then concatenates and projects back:

MultiHead(Q,K,V) = Concat(head₁, ..., headₕ) · W_O
head_i = Attention(Q·W_i^Q, K·W_i^K, V·W_i^V)

Why it helps: Different heads can specialize — empirically, some heads learn syntactic relations (subject-verb), others learn semantic similarity, others track coreference (important background for Versley, whose PhD was on bridging anaphora). This is not explicitly trained but emerges naturally. Total parameters are the same as one head at dimension d; inference cost is similar; quality is substantially better.

Tradeoff (important for Versley): Full MHA has KV cache size proportional to layers × H × seq_len × d_head. At long context or large batch, this dominates memory. Grouped Query Attention (GQA) — used in Llama 3, Qwen, and almost all 2024+ models — shares K/V across groups of Q-heads, reducing KV cache by factor H/G with negligible quality loss. eBay should be using GQA in any modern LiLiuM follow-up.

Q3. What is positional encoding? Compare sinusoidal, RoPE, and ALiBi. [HF]

Model Answer:

Transformers are permutation-equivariant by design — without position information, [cat sat mat] and [mat sat cat] are identical. Positional encoding breaks this symmetry.

Sinusoidal (Vaswani 2017): Fixed encoding added to embeddings before first layer: PE(pos, 2i) = sin(pos/10000^(2i/d)). Elegant, works, but encodes position as an additive bias to the embedding — subsequent layers see position mixed with content and cannot cleanly separate them. Cannot generalize beyond training length.

RoPE (Su et al. 2021) — used by LiLiuM, Llama, Qwen, Mistral, DeepSeek: Applies a rotation matrix of angle m·θ_j to Q and K vectors at position m. Key property: the inner product ⟨RoPE(q,m), RoPE(k,n)⟩ depends only on q, k, and m-n (the relative position). This means relative position emerges naturally in the dot product without explicit relative encoding. RoPE has no learnable parameters, works with Flash Attention, and extends gracefully to longer contexts via YaRN (scale rotational frequencies) or LongRoPE (layer-wise scaling). Llama 3.1 extended from 8k to 128k context this way.

ALiBi (Press et al. 2021) — used in MPT, BLOOM: Subtracts a linear penalty m·|i-j| from attention logits (proportional to key-query distance) rather than adding to embeddings. No learned parameters; generalizes reliably to longer sequences at inference than training (penalty just keeps growing). Better extrapolation than RoPE without extensions, but typically slightly lower absolute quality at known long contexts.

For the interviewer: LiLiuM uses RoPE (confirmed in the paper). The 4096-token context of LiLiuM-1B/7B/13B is a limitation — newer e-Llama work used 8192 context for continued pretraining. Tony can note: "RoPE with YaRN enables cheap context extension, which is how Meta got Llama 3.1 from 8k to 128k — an approach eBay could apply to LiLiuM for long listing descriptions or multi-document queries."

Q4. What is the computational complexity of self-attention, and how do efficient attention methods address the O(n²) problem? [HF]

Model Answer:

Standard attention: for sequence length n and dimension d: - Computing QKᵀ: O(n² · d) time, O(n²) memory (the attention matrix) - At n=128k: the attention matrix has 16B entries — infeasible to store in HBM

FlashAttention (Dao et al. 2022, v2 2023, v3 2025): Key insight: never fully materialize the n×n attention matrix. Use tiled computation: compute attention block-by-block, loading Q/K/V tiles from HBM to SRAM, computing partial softmax using online normalization (numerically stable), writing only O back. Same mathematical result. Memory: O(n) not O(n²). Speed: 2-8× faster than naive (memory bandwidth is the bottleneck). FlashAttention-3 adds FP8 support and hardware-level asynchrony (NVIDIA H100). The e-Llama pretraining used Flash-Attention-2.

Sparse attention (Longformer, BigBird): Each token attends to a local window w (linear complexity) plus a few global tokens. Changes the mathematical result — approximates full attention.

MLA (DeepSeek V2/V3): Projects K and V into a low-rank latent before the KV cache. Cache proportional to latent dim rather than n_heads × d_head. Achieves 131k context with manageable memory.

Practical for eBay: With product descriptions averaging 100-300 tokens, the O(n²) bottleneck is not acute for most listing tasks. But for RAG (retrieval over long documents, multi-listing context), FlashAttention is essential. LiLiuM almost certainly uses FlashAttention.

Q5. What is the vanishing gradient problem, and how do transformers solve it? [HF]

Model Answer:

In deep networks, gradients propagate backward through many layers via the chain rule. With activations like sigmoid (max gradient 0.25), gradient norms shrink exponentially with depth — very early layers receive near-zero gradient, so they barely learn.

Transformers solve this through:

  1. Residual connections: Each layer computes x_{l+1} = x_l + F(x_l). The gradient flows both through F and directly through the skip connection: ∂L/∂x_l = ∂L/∂x_{l+1} (1 + ∂F/∂x_l). Even if ∂F/∂x_l ≈ 0, the gradient is at least ∂L/∂x_{l+1} — a "gradient highway" ensuring early layers receive meaningful updates.

  2. Pre-norm (RMSNorm before sublayer): Normalizes activations before each sublayer, preventing scale explosion. In post-norm architectures (original Transformer), the normalization after the residual addition can cause problematic gradient dynamics. Pre-norm is more stable.

  3. Careful initialization: Xavier/Kaiming init ensures activations and gradients start at reasonable scale.

  4. QK-Norm (Gemma 2, Mistral Nemo, 2024): Explicitly normalizes Q and K vectors before dot product, preventing attention logit explosion (logits growing >1000 → numerical issues → loss spikes). This is a specific form of the vanishing/exploding gradient problem within attention.

Tony connection: Your pruning work (nnU-Net — >80% of weights removed with proxy Dice >0.95) had to be careful about gradient flow through sparse networks — unstructured sparsity can break residual paths if pruned aggressively. Similar issues arise in MoE where only 2/128 experts fire per token.

Q6. Compare encoder-only, decoder-only, and encoder-decoder architectures. When would you choose each? [HF]

Model Answer:

Architecture Examples Objective Best For
Encoder-only BERT, RoBERTa, DeBERTa MLM (bidirectional) Classification, NER, embeddings, retrieval
Decoder-only GPT-4, Llama 3, LiLiuM, Claude CLM (autoregressive, causal) Generation, chat, agents, code, MT with prompting
Encoder-decoder T5, mT5, BART, mBART Masked + autoregressive Seq-to-seq: MT, summarization, structured generation

Design choice logic:

  • Encoder-only: Every token attends to all other tokens (bidirectional). This gives richer contextual embeddings for understanding tasks. But generation is not natural (there's no sequential token prediction). Use for: retrieval encoders (eBERT at eBay), cross-encoders for reranking, classification heads.

  • Decoder-only: Causally masked attention — each token sees only past tokens. Enables in-context learning and few-shot generalization with zero architectural change. Scale has proven to unlock emergent capabilities (chain-of-thought, tool use). Use for: LiLiuM (generation, e-commerce understanding, MT). The dominant paradigm post-GPT-3.

  • Encoder-decoder: Separate bidirectional encoder (full context understanding) + causal decoder (generation). Naturally suited to seq-to-seq with explicit source/target. Prior SOTA for MT (mBART, NLLB). Now largely superseded by decoder-only for MT via prompting — though eBay's RAG for MT paper (2409.12880) uses Mixtral (decoder-only) with RAG examples rather than a dedicated MT encoder-decoder.

Versley nuance: His Amazon LINGUIST paper used AlexaTM 5B (seq2seq, 2022). He has real experience with both. He would appreciate noting that the shift to decoder-only came with tradeoffs — seq2seq can be more sample-efficient for structured output tasks because the full source is visible to every decoder step without positional overhead.

Q7. What are scaling laws, and what is the Chinchilla result? How does it affect how you'd plan LLM training at eBay? [HF]

Model Answer:

Scaling laws (Kaplan et al. 2020, Hoffmann et al. 2022) express loss as a power law over compute C, parameters N, and data D:

L(N,D) ≈ E + A/Nᵅ + B/Dᵝ

For a fixed compute budget C = 6ND (each token requires ~6 FLOPs for a forward+backward pass through an N-parameter model), loss is minimized when:

Chinchilla (Hoffmann et al. 2022): N ∝ D — scale model size and token count equally. Optimal tokens-per-parameter ≈ 20 tokens/param. Implication: GPT-3 (175B on ~300B tokens) was severely undertrained; a Chinchilla-optimal model at same compute = ~70B parameters trained on 1.4T tokens. Chinchilla-70B outperformed GPT-3-175B while using much less inference compute.

Beyond Chinchilla (2024-2025):

Inference-aware training (Sardana & Frankle 2024): If the model serves K inference tokens over its lifetime, optimal allocation shifts significantly toward smaller models trained longer. At K=10¹¹ tokens served, the optimal N is much lower than Chinchilla suggests — you amortize inference cost by overtraining smaller models. This is why Llama 3-8B was trained on 15T tokens (far above Chinchilla-optimal ~160B) — Meta optimized for serving, not training FLOPs.

eBay-specific consideration — tokenizer-compute interaction: LiLiuM was trained on 3T tokens with a custom 65k-vocabulary tokenizer. The standard Chinchilla formula C = 6ND counts tokens, but eBay's domain-specific tokenizer encodes more semantic content per token than a general tokenizer (e-commerce terms compressed from 4-6 generic tokens to 1-2 domain tokens). This means each eBay token carries higher information density than a CommonCrawl token. The 34% generation speedup Versley's team reports is the serving-side manifestation of this: the same information reaches the model in fewer autoregressive steps, so Chinchilla's training-compute estimate should be interpreted relative to information content, not raw token count. When planning LiLiuM-next or further CPT, this distinction matters: 3T domain tokens may represent substantially more semantic coverage than 3T generic tokens.

eBay context: LiLiuM (1B/7B/13B) trained on 3T tokens — well above Chinchilla-optimal for those sizes, consistent with the inference-optimization perspective. e-Llama (8B/70B) continued pretraining on 1T domain tokens is a domain adaptation approach, not from-scratch pretraining. Chinchilla doesn't directly apply to continued pretraining (the base model provides a strong prior; you need far fewer domain tokens than would be needed from scratch).

Tony bridge: My Diffusion Weights paper showed that fine-tuning can be accelerated by predicting future weight states — orthogonal to but consistent with the inference-optimal training philosophy: get to good weights faster by leveraging trajectory information.

Q8. What is masked language modeling vs. autoregressive language modeling? Compare BERT and GPT pretraining. [HF]

Model Answer:

Autoregressive (causal) LM (GPT, LiLiuM, Llama): Objective: predict the next token given all previous tokens.

L = -Σ log P(x_t | x_1, ..., x_{t-1})

Causal mask ensures no future leakage. Natural for generation — the pretraining objective matches the inference regime exactly. Scales to any sequence length at inference.

Masked Language Modeling (BERT): Randomly mask 15% of tokens; predict them from bidirectional context:

L = -Σ_{masked i} log P(x_i | x_{1..T, i≠masked})

Bidirectional attention — every token sees all other (unmasked) tokens. Richer contextual representations for understanding tasks. But the 15% masking creates a train-test mismatch: at inference, there's no masking — this can hurt some tasks and is why models like RoBERTa found removing NSP and increasing masking improved results.

NSP (Next Sentence Prediction): BERT's auxiliary task predicting if sentence B follows sentence A. Shown by RoBERTa/ALBERT ablations to be either trivially easy or harmful. Dropped in modern masked models.

Trade-offs: - MLM produces better embeddings for discriminative tasks (retrieval, classification) - Autoregressive produces better generation models; unified paradigm (one objective for all tasks) - Encoder-only MLM cannot generate naturally; decoder-only cannot do bidirectional encoding without tricks (fill-mask with diffusion-style decoding is one workaround)

Tony connection: Your Diffusion Weights paper is about fine-tuning dynamics — the training objective was standard autoregressive on MMLU/OpenBookQA/WMT16 (using LLaMA-style decoder-only models). The 3.2× speedup came from predicting weight evolution, not changing the LM objective.

1B. Pretraining, Tokenization, and Domain Adaptation

Q9. Compare BPE, WordPiece, and SentencePiece tokenization. Why did eBay build a custom tokenizer for LiLiuM? [HF]

Model Answer:

BPE (Byte-Pair Encoding): Iteratively merge the most frequent adjacent byte/character pair in the training corpus. Start from single characters (or bytes in byte-level BPE); add new tokens by merging. Greedy, deterministic. Used in GPT-4, Llama 3 (128k vocab), Qwen 3 (152k+ vocab).

Byte-level BPE (GPT-2): Starts from UTF-8 bytes — handles any language/script with zero OOV tokens. Critical for multilingual models.

WordPiece (BERT): Similar to BPE but merges based on likelihood improvement rather than raw frequency: choose merge that maximizes P(AB) / (P(A)·P(B)). Slightly different vocabulary from BPE on same corpus.

SentencePiece (Google) + Unigram LM: Operates directly on raw Unicode without pre-tokenization (no whitespace split assumption). Uses Unigram LM: start with large vocabulary, iteratively remove tokens that least harm corpus log-likelihood. Better for morphologically rich languages (German, Finnish), Asian scripts. Used in Gemma (256k vocab), T5, mT5.

The eBay LiLiuM tokenizer decision (core to Versley's work):

"Token fertility" = how many tokens a tokenizer requires to encode a given string. Standard tokenizers (LLaMA-2's SentencePiece, 32k vocab) were trained on general web text. E-commerce terms are systematically high-fertility: - "Polo Ralph Lauren" → 4-6 tokens in general tokenizer → 1-2 in custom tokenizer - Product codes ("ASML-L100/4M") → many tokens → potentially fewer - Category paths, item specifics, listing templates

Custom 65k vocabulary: LiLiuM trained a domain-specific vocabulary over eBay listing data alongside general text. Result: e-commerce sequences use fewer tokens for the same text. Since autoregressive generation cost scales linearly with token count, shorter sequences = fewer autoregressive forward passes = lower latency. eBay reports 34% generation speedup on e-commerce tasks vs. LLaMA-2 using this technique.

Versley's 2025 follow-up paper (arXiv 2509.26124) — vocabulary customization for adapted models:

The follow-up solves a different problem: how do you add domain vocabulary to an existing pretrained model (like Llama 3.1) without retraining from scratch? The algorithm extends an existing BPE merge list by appending new domain-specific merges at the end. Because BPE applies merges in order, appending ensures earlier merges are unaffected — which gives the key guarantee:

Monotone efficiency guarantee: Every input sequence is segmented into at most the same number of tokens as it was under the original tokenizer. This holds universally for all inputs (not just domain text), because appended merges can only reduce or maintain token counts — they can never increase them.

Embedding initialization: New vocabulary tokens have their embedding vectors initialized as the average of the embeddings of their constituent sub-tokens under the original tokenizer. For example, if "Makita" split as "Mak" + "ita" in the original tokenizer, the new "Makita" token embedding is initialized as the mean of the "Mak" and "ita" embedding vectors. This minimizes disruption to the residual stream and gives the model a meaningful starting representation for the new token without full retraining from scratch.

Results: Extending Llama 3.1 with 30k domain tokens reduced e-commerce input sequences by up to 20%, translating to 20-30% higher throughput (29.2% gain measured on 3000-word sequences). The model adopts the new tokenization in ~98% of cases for sequences longer than 15 words.

Why this matters beyond LiLiuM: LiLiuM's from-scratch custom tokenizer required training the whole model from scratch on that vocabulary. This paper shows you can post-hoc augment an adapted model (e-Llama) with vocabulary customization — combining CPT domain knowledge with tokenizer efficiency. The monotone guarantee is a production safety property: you can deploy this without regression-testing for throughput regressions, because no request can become slower.

Tony connection: Token count reduction has the same effect as speculative decoding or quantization — fewer operations per output. My LLM Router work reduces cost by routing to cheaper models; eBay's tokenizer approach reduces cost by reducing tokens per query. Complementary orthogonal axes of efficiency.

2025 research direction: SuperBPE (COLM 2025) learns cross-word "superword" tokens spanning whitespace — 33% fewer tokens and +4% benchmark accuracy. BoundlessBPE removes pre-tokenization constraints — 15% improvement in bytes-per-token. Vocabulary is now recognized as a scaling dimension alongside parameters and data.

Q10. What is continued pretraining / domain adaptation via continued pretraining? When do you use it vs. SFT vs. RAG? [HF]

Model Answer:

Continued pretraining (CPT): Take a pretrained base model and continue training on the next-token prediction objective on a domain-specific corpus — but without any instruction/preference data. This teaches the model new world knowledge and vocabulary patterns without changing its behavior format. Requires tens of billions to trillions of tokens to move the needle significantly.

e-Llama (arXiv:2501.09706, eBay): CPT of Llama 3.1 8B/70B on 1T tokens (50% e-commerce, 50% general). ~25% gain on English e-commerce benchmarks. Key decisions: - LR = 10% of original pretraining LR (3.0e-5 for 8B). Higher LR → catastrophic forgetting. See Q40 for the full ablation table. - 50/50 domain/general mix — more e-commerce doesn't help and degrades general capability. The 75% e-commerce condition in the ablation gave only +0.1 e-com score gain over 50% while general score began to degrade. - Model merging at end: average base Llama and fine-tuned e-Llama parameters — gives a smooth trade-off curve between general and domain capability at zero compute cost.

Why CPT wins over LoRA SFT for this use case: LoRA SFT trains only a low-rank adapter and optimizes on task-specific instruction-response pairs. It is efficient and good for formatting/behavior change, but it does not deeply alter the model's internal representations of domain vocabulary or domain-specific world knowledge — the frozen base weights constrain what the LoRA can learn. CPT with full gradient flow rewrites the representation space so that e-commerce concepts are natively understood at the residual stream level, not just surface-formatted. When downstream tasks are diverse (attribute extraction, MT, listing generation, QA) and the base model simply lacks the representational building blocks for the domain, CPT is the only technique that installs those building blocks. The 340,000 GPU-hours for e-Llama-70B is justified precisely because the resulting model is a general-purpose e-commerce foundation — not a task-specific adapter.

eBay uses all of these: - CPT: e-Llama, LiLiuM-from-scratch - SFT: instruction-tuning LiLiuM for chat/agents - RAG: product title translation (arXiv 2409.12880), Mercury recommendations - Model merging: e-Llama general/domain balance

Tony bridge: My LLM Router enables a different strategy: route easy e-commerce queries to a fine-tuned small model (LiLiuM-1B SFT'd) and hard/ambiguous queries to the full 70B — achieving cost savings comparable to domain CPT without the 340k GPU-hour training run.

Q11. What is LoRA, and how does it reduce trainable parameters while maintaining expressivity? Compare with full fine-tuning, QLoRA, and prefix tuning. [HF]

Model Answer:

LoRA (Hu et al., 2021): The hypothesis: weight updates during fine-tuning have low intrinsic rank. Instead of updating the full weight matrix W ∈ R^{d×k}, freeze W and add a trainable low-rank decomposition:

W' = W + ΔW = W + B·A    where B ∈ R^{d×r}, A ∈ R^{r×k}, r << min(d,k)

Initialize B=0, A~N(0,σ²) → zero perturbation at start. Scale by α/r for stability. After training, merge W' = W + α/r · BA for zero inference overhead.

Parameter reduction: For a 7B model, a typical attention projection is 4096×4096 = 16.8M params. LoRA with r=8 uses (4096×8 + 8×4096) = 65k params — 256× reduction. Apply to all linear projections: total trainable params ~0.5-1% of base model.

QLoRA (Dettmers et al. 2023): Quantize the frozen base model to 4-bit (NF4 — NormalFloat quantization, theoretically optimal for normally-distributed weights). LoRA adapters trained in BF16. Double quantization: also quantize the quantization constants. Paged optimizers: CPU-GPU paging for optimizer state. Result: fine-tune 65B on a single 48GB GPU. Quality within 1-2% of FP16 LoRA. Dominant for academic/resource-constrained settings.

Prefix tuning: Prepend k learnable "soft prompt" tokens to input — only these ~100-1000 parameters are trained. Works for classification; weaker for complex generation. Interpretability: the soft prompts have no discrete interpretation.

Comparison: | Method | Params Trained | Memory | Quality | Inference Overhead | |---|---|---|---|---| | Full FT | 100% | Very High | Best | None | | LoRA | 0.1-1% | Medium | Near-full for most tasks | None (merged) | | QLoRA | 0.1-1% | Low | -1-2% vs LoRA | None (merged) | | Prefix Tuning | <0.1% | Very Low | Weaker | +k tokens per inference |

eBay context: LoRA is almost certainly used for instruction-tuning LiLiuM on task-specific e-commerce data (product attribute extraction, listing generation, MT formatting) since full fine-tuning 13B is expensive and LoRA-merged models run identically fast.

Q12. Explain RLHF and DPO. Why was DPO introduced, and what are DPO's limitations? What is GRPO and how does it differ from PPO? [HF]

Model Answer:

RLHF (classic, 3 stages):

  1. SFT: Fine-tune base LM on high-quality demonstrations
  2. Reward Model (RM): Collect human pairwise preferences (prompt, chosen, rejected). Fine-tune a model with a Bradley-Terry head: P(y_w > y_l | x) = σ(r(y_w,x) - r(y_l,x)). Train with cross-entropy on human-preferred pairs.
  3. PPO: Optimize the SFT policy to maximize RM reward with KL penalty: max_π E[r_φ(y,x)] - β·KL[π(y|x) || π_sft(y|x)] Requires four models in GPU memory simultaneously (policy, reference, RM, value model). Sensitive to hyperparameters. Reward hacking: policy finds high-RM outputs humans would actually dislike.

DPO (Rafailov et al. 2023): Key insight: the optimal policy π* implied by the RLHF objective has a closed form:

r(y,x) = β log(π*(y|x) / π_ref(y|x)) + β log Z(x)

Substituting into the RM training loss and simplifying, the reward model and policy collapse into one optimization:

L_DPO = -E[(y_w,y_l)] [log σ(β · (log π/π_ref(y_w|x) - log π/π_ref(y_l|x)))]

Train the policy directly on preference pairs — no explicit RM, no PPO, no value model. Memory: just policy + frozen reference model. Stable, simple, now near-universal in open-source LLM fine-tuning.

DPO limitations: - Distribution mismatch: DPO uses offline preference data; the policy distribution at training time differs from data collection distribution. Can lead to suboptimal policy. - No exploration: PPO's online sampling finds high-reward continuations not in the dataset. DPO only uses existing pairs. - KL penalty is implicit: Cannot tune KL directly; β controls it indirectly. - Preference data quality is critical: Bad labels hurt more than in RLHF because there's no RM to smooth noise. - Doesn't directly enable verifiable reward: For math/code, GRPO with verifiable rewards is more principled.

GRPO (Group Relative Policy Optimization — DeepSeek-R1):

GRPO removes the critic/value model that PPO requires. Instead of a trained value function to estimate baselines, GRPO samples a group of G outputs for each question, then normalizes each reward relative to the group:

A_i = (r_i - mean({r_1, r_2, ..., r_G})) / std({r_1, r_2, ..., r_G})

This group-relative advantage replaces the per-token value function. The result: no critic model needed (the same size as the policy — a massive memory saving), training is more stable (no value function lag), and rewards can be fully verifiable (binary correct/incorrect for math, code execution success/failure) rather than relying on a potentially mis-calibrated reward model. DeepSeek-R1 achieved reasoning capabilities comparable to OpenAI o1 using GRPO with rule-based verifiable rewards, without a trained reward model.

Why Versley would care: GRPO is the technique behind the reasoning model wave of 2025. For eBay, verifiable rewards exist naturally in certain tasks: product attribute extraction correctness can be verified against a knowledge graph; MT quality can be verified via round-trip translation or reference comparison.

Modern variants: SimPO (removes reference model entirely), IPO (adds identity constraint), KTO (binary feedback, not pairwise).

Q13. What are the main failure modes when training LLMs? What causes loss spikes, and how do you mitigate them? [HF — Versley runs these jobs]

Model Answer:

Loss spikes = sudden jumps in training loss (2-10×), typically followed by slow recovery over hundreds of steps. At 480-H100 scale (e-Llama), a spike that requires 500 recovery steps costs significant compute and GPU time.

What the e-Llama paper actually says about spikes:

The e-Llama paper (arXiv:2501.09706) does not document a specific spike event in the published version — the training narrative focuses on hyperparameter choices validated by ablations, not an incident post-mortem. If Versley mentions a spike from an internal training run (separate from the published paper), the appropriate response is to ask him to describe what they observed before explaining your understanding of mitigations. Do not confabulate a specific event from the paper that is not there.

Root causes of loss spikes (mechanistic understanding):

  1. Attention logit explosion: QKᵀ / √d_k values become very large (>1000) → softmax becomes peaky/numerically unstable → gradient variance explodes. Arises from accumulated weight growth in Q and K projections. Fix: QK-Norm (normalize Q and K vectors before the dot product — used in Gemma 2, Mistral Nemo; prevents logits from growing unboundedly regardless of weight scale).

  2. LR × batch interactions: Adam's first/second moment estimates lag true gradient direction early in training or after data distribution shifts. If LR is too high relative to gradient noise, updates overshoot a loss basin.

  3. Data quality outliers: A single malformed batch (very long sequence, very noisy text, repeated content) causes spike. Fix: data quality filtering (deduplication, perplexity filtering), document packing with careful boundary attention masking so sequences from different documents don't attend across boundaries.

  4. MoE router instability: Softmax router logits can drift → unbalanced expert loading → tokens overflowing or dropped → gradient variance. Fix: auxiliary load-balancing loss (penalize deviation from uniform expert utilization), Z-loss (penalizes large softmax logits: ε · log²(Σ exp(logits))).

Mitigation toolkit:

  • Gradient clipping: g ← g · (max_norm / ||g||₂) if norm exceeds threshold (typically 1.0). Blunt instrument — clips all gradients proportionally.
  • ZClip (2025): Adaptive clipping using z-score anomaly detection on rolling gradient norm statistics. Clips only true outlier batches, not normal high-gradient steps.
  • SPAM (2025): Spike-Aware Adam — detects spikes, resets Adam momentum buffers to prevent optimizer reinforcing bad gradient direction.
  • Warmup schedule: Linear warmup over 1000-2000 steps prevents early LR being too aggressive before Adam moments stabilize.
  • BF16 instead of FP16: BF16 has larger exponent range (±3.4×10³⁸ vs FP16's ±65504) → handles larger activations without overflow. No loss scaling needed.
  • Checkpoint averaging: Average parameters across N checkpoints near training end — smoothes out optimization noise, typically 0.5-1% quality improvement. Used in LiLiuM.
  • Rollback: If spike is catastrophic, rollback to last healthy checkpoint. Requires frequent checkpointing (every 500-1000 steps at scale). This is the standard production response at large-scale training jobs — Megatron-LM supports this natively.

If you can predict where weights are going, you can potentially detect spike-prone trajectories before they materialize. An unexplored direction: use trajectory prediction to trigger early rollback.

Q13-NEW. Have you worked with tensor parallelism or pipeline parallelism? What are the communication bottlenecks in 3D parallel training? [HF — Versley leads this at 480 H100 scale]

Model Answer:

The e-Llama pretraining used Megatron-LM on 60 nodes × 8 NVIDIA H100 80GB = 480 GPUs with full 3D parallelism. This is the infrastructure Versley runs. Tony has not personally run 480-GPU jobs, but the conceptual understanding is critical.

3D parallelism = Tensor Parallelism (TP) + Pipeline Parallelism (PP) + Data Parallelism (DP):

Tensor Parallelism (TP): Shard individual matrix multiplications across GPUs within a node. For an FFN with weight W ∈ R^{d×4d}: split W column-wise across T GPUs. Each GPU holds W[:, d_i:d_{i+1}] and computes its shard of the output. An all-reduce recombines outputs. TP degree is typically bounded by GPUs per node (8 for H100 nodes) because all-reduce over NVLink (within node) is fast (~600 GB/s) but over InfiniBand (cross-node) would bottleneck. Communication: one all-reduce per transformer layer per TP group.

Pipeline Parallelism (PP): Assign different transformer layers to different nodes (node 0 handles layers 0-9, node 1 handles layers 10-19, etc.). During the forward pass, layer outputs (activations) are sent across InfiniBand to the next stage. During backward, gradients flow backward. Pipeline bubble: each stage must wait for its predecessor to finish the forward pass before it can start. Efficiency = (total micro-batches) / (total micro-batches + pipeline_stages - 1). At PP=8 and micro-batch count 16: efficiency ≈ 16/(16+7) = 70%. Larger micro-batch counts reduce bubble fraction. Interleaved scheduling (1F1B) helps further.

Data Parallelism (DP): Replicate the full model (or each TP/PP shard) across DP replicas, each processing different data. Synchronize gradients via all-reduce after each step. Communication: gradient all-reduce is the dominant inter-node communication for pure DP. Fully Sharded DP (FSDP / ZeRO-3) shards the optimizer state, gradients, and parameters across DP ranks — reducing per-GPU memory at the cost of more communication.

Why all three are needed at 480-GPU scale: - A 70B model requires ~140GB in BF16, which doesn't fit on a single 80GB H100 → need model parallelism (TP+PP) - TP within node (fast NVLink): shards weight matrices - PP across nodes (moderate InfiniBand): shards layers - DP across the remaining dimension: data throughput

Communication bottlenecks in practice: - TP all-reduce (NVLink): fast but requires all-reduce per layer — becomes expensive for large TP degree - PP activation transfer (InfiniBand): relatively small (one activation tensor per stage), manageable - DP gradient all-reduce (InfiniBand): expensive at large batch; ZeRO reduces this by distributing optimizer state - The interplay: Megatron-LM overlaps TP all-reduces with compute using CUDA streams; pipeline bubble is managed by micro-batch scheduling

Tony's honest answer if asked directly: "I haven't personally run 480-GPU training jobs, but I understand the 3D parallelism decomposition from Megatron-LM's design. The tradeoff between pipeline bubble fraction and micro-batch count is the key scheduling constraint, and I'd expect the InfiniBand cross-node communication for gradient synchronization to be the primary bottleneck at that scale rather than compute — which is consistent with the inference-optimal training philosophy of keeping batch sizes large to amortize communication cost."

Q15. What is catastrophic forgetting, and how do you mitigate it in continued pretraining for domain adaptation? [HF]

Model Answer:

Catastrophic forgetting: When a neural network is trained on new data/tasks, it overwrites (forgets) weights encoding old knowledge. In a sequential continual learning setting, performance on the original task can collapse entirely.

In CPT context: if you run e-commerce continued pretraining at too high an LR or with too much domain data, the model loses its general English/multilingual/reasoning capability gained during original pretraining.

Evidence from e-Llama ablation (Table 2, arXiv:2501.09706): - LR=3.0e-3 (10× original): General NLU score drops from 44.1 (baseline) to 34.5 → catastrophic forgetting - LR=3.0e-4 (1× original): General 40.5, modest forgetting - LR=3.0e-5 (0.1× original): General 42.0, minimal forgetting — optimal

Mitigation techniques:

  1. Low learning rate for CPT (primary): Use 5-10% of original pretraining LR. The pretrained model is already in a good basin; small steps stay nearby while absorbing domain signal.

  2. Data mixing (replay): Include general-domain data in CPT corpus (e-Llama: 50% general). Acts as "rehearsal" — model sees original distribution examples continuously and cannot forget them. Trade-off: you spend half your compute on non-domain data.

  3. Warmup/cooldown LR schedule: Cosine LR with warmup prevents initial high-LR damage. Some teams use constant LR followed by decay, others use cooldown at the end (annealing on high-quality domain subset).

  4. Model merging (post-CPT): Average parameters of original base and CPT-adapted model: θ_merged = α·θ_base + (1-α)·θ_cpt. At α=0.5 (equal weight), general capability largely recovered while domain gains are partially preserved. Tune α to hit target operating point. Zero additional training cost. Used in e-Llama with near-linear trade-off curve.

  5. EWC (Elastic Weight Consolidation): Compute Fisher information diagonal approximating which weights were most important for original tasks; penalize changes to those weights during CPT. Computationally expensive at scale; rarely used in practice.

  6. PEFT (LoRA for domain): Keep base model frozen; train LoRA adapters on domain data. By construction, base model is unchanged — no forgetting. Trade-off: adapter only has limited capacity; for heavy domain shift, CPT may work better.

Tony angle: My LLM Router work is relevant here. If catastrophic forgetting is a real risk, one strategy is: keep the base general-purpose model and route domain queries to a LoRA-adapted specialist, rather than adapting the base model itself. This avoids forgetting entirely and enables multiple domain specialists without separate full model copies (LoRA adapters are small).

1C. Inference, Efficiency, and Routing

Q16. What is KV cache? How does PagedAttention (vLLM) improve on naive KV cache management? [HF]

Model Answer:

KV Cache: During autoregressive generation, at each step t, the model needs K and V tensors from all previous positions 1..t-1 to compute attention. Without caching, you'd recompute all previous K/V values on every step — O(t²) total computation. The KV cache stores K and V for all processed tokens, so each new step only computes attention for the new token against the cached values. Cost per step: O(t) attention (over cache) + O(1) forward pass for the new token.

Memory cost: 2 × layers × heads × d_head × seq_len × batch_size × dtype_bytes. For a 70B model (80 layers, 8 GQA heads, 128 d_head): at 4096 tokens, batch 16: ~80GB just for KV cache. This is typically the binding memory constraint in LLM serving.

Naive KV cache problem: Systems pre-allocate a contiguous block of memory of size max_seq_len per request at request start. Problems: - Most requests don't use max_seq_len → 60-80% of KV memory is wasted - Requests of different lengths interfere with each other - No memory sharing between requests with shared prefixes

PagedAttention (vLLM, Kwon et al. 2023): Inspiration from OS virtual memory paging. Divide KV cache into fixed-size blocks (pages). Each sequence's KV cache is stored in non-contiguous pages. A block table maps logical page number → physical block.

Benefits: 1. No pre-allocation waste: Only allocate pages as they're actually needed 2. Prefix sharing: Multiple requests sharing the same system prompt or RAG context use the same physical KV pages — massive savings for RAG workloads (eBay: all MT requests share the same retrieval template) 3. Flexible memory management: Pages freed when request finishes, immediately available 4. Higher batch size: Less wasted memory → more concurrent requests → better GPU utilization

Continuous batching (iteration-level batching): Separately from PagedAttention, continuous batching allows new requests to join mid-batch when a sequence finishes, rather than waiting for the whole batch. Combined with PagedAttention: vLLM achieves 10-24× higher throughput than Hugging Face native inference.

Tony connection: My LLM Router work designs the dispatch strategy — which model gets which query. PagedAttention is the serving substrate that must execute those dispatched queries efficiently. The interaction is: if my router sends a query to LiLiuM-7B, the serving system must have efficient KV cache management; otherwise the router's cost savings are lost to memory overhead.

Q17. Explain speculative decoding. When does it help, and when does it not? [HF]

Model Answer:

Problem: Autoregressive LLM generation is sequential — each token requires one full forward pass through the model. Generation is typically memory-bandwidth-bound (loading model weights), not compute-bound. This means smaller models can generate tokens at nearly the same latency as larger models if memory bandwidth is the constraint.

Speculative decoding (Leviathan et al. 2022): 1. A small draft model (same tokenizer, much fewer parameters — e.g., 70M vs 70B) generates N candidate tokens in fast sequential generation 2. The target model (large, accurate) verifies all N tokens in a single parallel forward pass (since the draft tokens are already known, verification is a single batch forward pass, not sequential generation) 3. Accept/reject: Compare draft probabilities with target probabilities at each position. Accept token t_i if min(1, p_target(t_i) / p_draft(t_i)) passes a random threshold. Accept all tokens up to the first rejection; sample a correction token from the target at the rejection point.

Why it's correct: The joint distribution of accepted tokens equals exactly the target model's distribution — no approximation. It's pure latency optimization.

Speedup: If the draft model accepts on average k of N proposals, the effective throughput is roughly (k+1) tokens per target model forward pass instead of 1. Empirically 2-3× speedup for greedy generation when draft acceptance rate is high.

When it helps: - Large target model, memory-bandwidth-bound (high parameter count, small batch) - Draft and target models are strongly aligned (similar pretraining, same tokenizer) - Generation tasks where draft model makes high-quality predictions (repetitive, predictable outputs — product titles, structured listings)

When it doesn't help: - High batch sizes: memory bandwidth is amortized across the batch; target model is now compute-bound; speculative decoding doesn't help - Low draft acceptance rate: novel/complex domain text where small draft model guesses poorly - If draft model is still slow (e.g., 1B model still has meaningful latency overhead)

eBay angle: For structured e-commerce tasks (generating product attribute values in a fixed template), speculative decoding could yield high acceptance rates. For open-ended MT or search queries, acceptance rates are harder to predict. An alternative is using LiLiuM-1B as the draft model for LiLiuM-7B — same architecture family, likely high acceptance rate.

Q18. How would you design an LLM router to assign easy steps of complex tasks to cheaper models while maintaining overall quality? What are the key tradeoffs? [HF] (Tony's signature topic)

Model Answer:

This is the core of my LLM Router, TwinRouterBench, MERA, and Cost-Adaptive Routing work — I'll describe the full design space and then be direct about what Versley is most likely to probe.

Problem formulation: Given a set of models M = {m₁, ..., mₖ} with costs c₁ < c₂ < ... < cₖ and capabilities q₁ < q₂ < ... < qₖ, and a query distribution P(x), find routing policy π: x → m_i that minimizes expected cost subject to a quality constraint:

min_π E[c(π(x))]  s.t.  E[quality(π(x), x)] ≥ Q*

Routing signals (key design choice):

  1. Query complexity proxy: Embedding-based similarity to "hard" vs "easy" training examples. Works but suffers from distribution shift.
  2. Cascade routing: Query small model first; if confidence is high (max logit > threshold), use its answer; else query large model. Natural sequential structure. No extra routing model needed.
  3. Learned router classifier: Train a lightweight classifier (e.g., BERT-tiny) on query-level features to predict "does this query need the big model?". Training signal: cases where small model failed vs. succeeded on held-out labeled data.
  4. LLM-as-judge for routing: Query a small LLM to assess query difficulty and recommend routing. Meta-routing — expensive but accurate for ambiguous cases.

My LLM Router (NeurIPS'26 under review): Route individual steps of agentic tasks rather than full queries. Key insight: complex tasks have both hard and easy steps. Assigning easy steps (factual lookup, format conversion) to cheap models and hard steps (multi-hop reasoning, novel synthesis) to expensive models achieves >90% cost reduction vs. using SOTA for every step.

Numbers from my work: - LLM Router: >90% cost reduction vs. single SOTA per step, matched performance - TwinRouterBench: 53% cost reduction matching performance - MERA: 87.3% router accuracy at ~52% of always-large serving cost (router-to-label match, not task accuracy; 4.4% fallback)

What Versley will actually probe:

(a) Generalization to unseen task types:

"How does the router generalize to e-commerce task types not seen in any training distribution?"

Honest answer: a trained routing classifier will underperform on OOD tasks — it will default to whatever its prior is (likely the majority class). Mitigation: (1) use a conservative default — route to the strong model for tasks where the router is uncertain (routing loss is asymmetric: the cost of under-routing to a weak model on a hard task is higher than the cost of over-routing to a strong model on an easy task); (2) use a general-purpose difficulty embedding rather than task-specific features — embed the query with a general-purpose encoder and measure nearest-neighbor distance to known-hard queries; (3) my Cost-Adaptive Routing paper addresses this directly: cold-start on a new task type defaults to the strong model, then as trajectories accumulate, a specialist is fine-tuned and routing confidence grows.

(b) Error distribution at the routing margin:

"What actually happens at the boundary where the cascade router is uncertain?"

The cascade's uncertainty region is where the small model's confidence is near the threshold. In this region, the quality distribution is bimodal: some queries the small model handles fine, others it gets wrong. The mean cost reduction looks good, but the variance is high. In practice, we observed that margin cases cluster around the hardest sub-type of easy tasks — tasks the small model handles 60% of the time rather than 95%. Widening the confidence threshold (more conservative) improves quality at the boundary at ~10-15% additional cost. For eBay's production use, I'd recommend erring conservative at the margin and measuring empirical quality degradation at each operating point before setting the threshold.

(c) The "same performance" claim:

"Your NeurIPS paper claims >90% cost reduction at same performance — what is the exact quality metric and at what value?"

The quality metric is task-success rate on the agentic benchmark (Agentic Coding Bench and similar multi-step reasoning tasks), measuring whether the full task completes correctly end-to-end (not individual step accuracy). "Same performance" means within 1-2 percentage points of the full-SOTA baseline on task completion rate. The metric is intentionally end-to-end rather than per-step because individual step quality is not what users care about — task completion is. The 90%+ cost reduction is measured in API token cost (input + output tokens across all steps summed).

Q19. What are Mixture of Experts (MoE) models, and how does routing work? What are the key engineering challenges? [HF]

Model Answer:

Architecture: MoE replaces the dense FFN layer in a transformer with E parallel expert FFNs. A learned router (linear layer → softmax → top-K selection) assigns each token to the top-K experts:

output = Σᵢ gᵢ · Expertᵢ(x)    where g = top_K(softmax(W_r · x))

Only K/E fraction of expert weights activate per token → same inference FLOPs as a smaller dense model, but parameter count (and model capacity) scales independently.

Key parameters: E=8 (Mixtral), E=64 (switch), E=256 (DeepSeek-V3). K=1 (Switch Transformer) or K=2 (Mixtral, standard). DeepSeek-V3: 671B total / 37B active (5.5% active ratio).

Router challenges:

  1. Expert collapse: Without regularization, all tokens route to the same expert or two. The router settles into a degenerate mode. Fix: auxiliary load-balancing loss penalizes deviation from uniform distribution: L_balance = α × E × Σᵢ fᵢ × pᵢ where fᵢ = fraction of tokens routed to expert i, pᵢ = mean router probability for expert i.

  2. Token dropping: Fixed expert capacity (max tokens per expert per batch). If an expert is overloaded, excess tokens are dropped. Quality risk if tuned too aggressively. Mitigated by expert capacity factors > 1.0.

  3. Loss spikes in MoE: Particularly acute because router logits are unbounded before softmax. A sudden imbalance spike → experts see wrong distribution → gradient variance explosion. Fix: Z-loss on router logits + QK-Norm in the transformer layers + careful LR schedule.

  4. Communication overhead: In distributed training, different experts may be on different GPUs. Each token's expert routing requires an all-to-all dispatch (send each token to its expert's GPU) and all-to-all combine (gather results). At expert parallelism across nodes (InfiniBand), this can be the training bottleneck.

  5. Inference serving: All expert weights must be loaded even if only K/E are active per token. A 671B MoE model requires 671B worth of memory even though only 37B fires per query. Requires expert offloading or very large GPU pools.

Relationship to Tony's routing work: MoE routing is per-token, continuous, trained jointly with the model. My LLM Router routing is per-query or per-step, discrete, trained on outcome data, routes to entirely separate model instances. The mechanisms are analogous (both learn a routing policy) but operate at very different granularities and abstraction levels. Q20. Walk through a complete RAG pipeline. How does eBay use RAG for product title translation? [HF]

Model Answer:

RAG Pipeline (general):

  1. Offline indexing: - Split documents into chunks (fixed-size / semantic / paragraph-level) - Embed each chunk with a dense encoder (bi-encoder, e.g., sentence-transformers) - Store embeddings + metadata in a vector database (FAISS, Weaviate, Pinecone, Chroma)

  2. Online retrieval: - Encode incoming query with same encoder - ANN search: top-K chunks by cosine/dot-product similarity - Optional: hybrid retrieval — combine dense results with BM25 sparse retrieval via Reciprocal Rank Fusion (RRF) - Optional: reranking with a cross-encoder (compute per (query, chunk) pair for better accuracy)

  3. Augmented generation: - Prepend retrieved chunks to the prompt (or inject into specific positions) - LLM generates conditioned on both the query and retrieved context

  4. Output + monitoring: - Return generated text - Monitor faithfulness (is the answer grounded in retrieved content?) and relevance (did we retrieve the right chunks?)

eBay RAG for Product Title Translation (arXiv 2409.12880):

Problem: Product titles are short (avg 11.3 words), lack context, contain e-commerce jargon (brand abbreviations, product codes, material terms). Standard MT fails on these.

RAG approach: - Retrieval index: 10+ million bilingual product title pairs (e.g., English-Dutch, English-Polish) - Retrieval method: BM25 (Lucene 8.11.3 + OKAPI BM25) over title text — chosen over dense retrieval because product titles benefit from exact-match keyword lookup (brand names, model numbers must match exactly, not approximately) - Prompt: 1-shot or 5-shot examples of similar translated product titles prepended to the translation request - LLM: Mixtral-8x7B-Instruct - Index types tested: title-only (TTL), bullet points (BP), full description (PD), combined

Results (chrF improvement over zero-shot baseline): | Pair | 5-shot best gain | |------|----------------| | EN-PL | +15.3% | | EN-SE | +14.0% | | EN-NL | +12.0% | | IT-DE | +3.6% | | EN-DE | +2.0% |

Key insight: RAG helps most for language pairs where the LLM has limited proficiency (EN-PL: underrepresented in Mixtral's training; EN-SE: similar). For EN-DE (LLM's strong suit), RAG adds 2% — the LLM already handles well. RAG patches weak model coverage, not strong coverage.

Q21. What are the failure modes of RAG, and how do you evaluate a RAG system? [HF]

Model Answer:

Failure modes:

  1. Retrieval failure: Wrong chunks retrieved; relevant chunk not in index. Manifests as answer not grounded in documents. Root causes: embedding model doesn't capture the right similarity for this domain; query representation differs from document representation; chunk boundaries cut relevant context.

  2. "Lost in the Middle" (Liu et al. 2023): When many chunks are retrieved, LLM attends preferentially to the beginning and end of the context window. Chunks in the middle of long contexts are underweighted even if they're relevant. Fix: put most important chunks at beginning/end, or use IN2 training (train specifically on middle-content tasks).

  3. Faithfulness failure (hallucination): LLM generates plausible-sounding content that contradicts or goes beyond the retrieved chunks. Even with perfect retrieval, LLMs can override retrieved facts with parametric knowledge.

  4. Context length overflow: Too many retrieved chunks exceed the model's context window or cause attention degradation. Fix: reranker to select top-N most relevant; chunking strategy review.

  5. Staleness: Index built at time T; documents updated at T+1. Retrieved chunks may be outdated. Fix: incremental index updates, TTL on indexed content.

  6. Over-retrieval: Retrieved chunks that are topically related but not specifically helpful confuse the LLM ("noisy context" effect).

Evaluation framework:

Retrieval metrics: - Recall@K: fraction of truly relevant documents in top-K retrieved - Precision@K: fraction of top-K retrieved that are truly relevant - MRR (Mean Reciprocal Rank): 1/rank_of_first_relevant - NDCG@K: normalized discounted cumulative gain (rewards highly-relevant results ranked higher)

Generation/faithfulness metrics: - Faithfulness (RAGAS): is the answer entailed by retrieved context? - Answer relevance (RAGAS): does the answer actually address the question? - Context relevance (RAGAS): are the retrieved chunks relevant to the question? - ROUGE/BERTScore vs. reference answers: conventional generation quality

End-to-end: - Task-specific: chrF for MT, exact match for structured extraction, F1 for entity extraction - Human evaluation for open-ended generation quality - A/B testing: user engagement (click-through, dwell time, conversion) for product recommendations

1E. Classical NLP (Versley's Home Turf)

Q22. Walk through Hidden Markov Models and the Viterbi algorithm. How were they used in NLP, including machine translation alignment? (Reported directly in eBay interviews) [HF]

Model Answer:

HMM formalism: A generative model with hidden state sequence s₁, ..., sₙ and observed sequence o₁, ..., oₙ.

Three components: - Transition probabilities: P(sₜ | sₜ₋₁) — probability of moving to state sₜ given current state - Emission probabilities: P(oₜ | sₜ) — probability of observing oₜ from state sₜ - Initial probabilities: P(s₁) — prior over starting state

NLP applications: - POS tagging: states = POS tags (NN, VB, DT, ...), observations = words. Train P(tag | previous_tag) from treebank, P(word | tag) from counts. - NER with BIO tagging: states = B-PER, I-PER, B-ORG, ..., O. Same structure. - Speech recognition (historical): acoustic model (HMM per phoneme) + language model.

Viterbi algorithm (dynamic programming): Find the most probable state sequence argmax_S P(S|O) efficiently.

# Initialize
viterbi[t=1][s] = P(s₁=s) × P(o₁|s)
backpointer[1][s] = 0

# Recursion
for t in 2..T:
    for each state s:
        viterbi[t][s] = max_{s'} (viterbi[t-1][s'] × P(s|s')) × P(oₜ|s)
        backpointer[t][s] = argmax_{s'} (viterbi[t-1][s'] × P(s|s'))

# Termination
best_final_state = argmax_s viterbi[T][s]
# Traceback via backpointer

Complexity: O(T × N²) where T = sequence length, N = number of states. Feasible for NLP tag sets (N ~30-50 for POS, ~100 for BIO NER).

The MT connection — HMM word alignment (Versley's actual domain):

Before neural MT, phrase-based SMT (statistical machine translation) required word alignment: which source word corresponds to which target word in a parallel sentence pair? This is the key to extracting translation lexicons and phrase tables.

IBM Models 1-5 (Brown et al. 1993) are the foundational alignment models: - IBM Model 1: Each target word is independently generated from a mixture of all source words. EM algorithm on parallel corpus learns translation table t(f|e) = probability that foreign word f translates to English word e. No position model — all alignments equally likely given fertility. - IBM Model 2: Adds a distortion/position model: probability that source position j generates target position i in a sentence of length l and m. Still EM-trained. - IBM Model 3/4/5: Add fertility (a source word can generate 0, 1, 2... target words) and deficiency corrections. - HMM alignment model (Vogel et al. 1996): Between IBM 2 and 3 in complexity. Models alignment as an HMM where transitions are over relative alignment positions (short jumps more likely than long jumps). Viterbi decoding over alignment positions gives the MAP alignment. More linguistically motivated than IBM 2; less complex than IBM 3.

GIZA++ (Och & Ney 2003): The production tool that implemented IBM 1-5 + HMM alignment. Training: EM on parallel corpus. Decoding: Viterbi for MAP alignment. This was the backbone of all phrase-based MT systems (Moses, Joshua) through ~2015.

Why this matters for eBay: The attention mechanism in neural MT (Bahdanau et al. 2014) is a soft, differentiable relaxation of word alignment. The alignment weights αᵢⱼ = softmax(e(sᵢ, hⱼ)) play exactly the role that HMM alignment probabilities played in GIZA++. Versley's work at Amazon (LINGUIST, AlexaTM) built on this progression. Understanding HMM alignment → neural attention is the conceptual bridge from statistical NLP to modern LLMs that Versley will appreciate in a candidate.

Modern replacement: Neural MT (seq2seq + attention, 2014-2017) replaced HMM alignment-based SMT. The attention weights serve as implicit alignment. Modern LLMs do MT entirely via next-token prediction without explicit alignment, but the IBM/HMM alignment tradition directly informed how attention was conceived and analyzed.

Q23. What is coreference resolution? How has it evolved from rule-based to neural to LLM-based approaches?

Model Answer:

Coreference resolution: Identify all mentions in a text that refer to the same real-world entity and cluster them into coreference chains.

Example: [Alice]₁ went to the store. [She]₁ bought milk. [The woman]₁ forgot her wallet. → one chain for Alice.

Hard cases (Versley's specialty): - Bridging anaphora: I walked into a room. The window was open. — "the window" refers to a specific window inferrable from "a room" but not previously mentioned explicitly. Requires world knowledge. - Singleton mentions: mentions that don't corefer with anything (must not be over-clustered) - Generic references: Cats like tuna — "cats" refers to a class, not specific entities

Evolution:

  1. Rule-based (Hobbs 1978, Stanford CoreNLP): Deterministic, syntactic salience rules (subject > object; recent mention; pronoun agreement). Fast, interpretable. Fails on long-range dependencies and world knowledge requirements.

  2. Statistical / feature-based (CoNLL 2011/2012 shared tasks): Pairwise mention classifiers with hand-crafted features (distance, gender, number, head word, syntactic path). Lee et al. 2013. Required extensive feature engineering. ~70-75% F1 on CoNLL.

  3. End-to-end neural (Lee et al. 2017): BiLSTM + span representations. Joint mention detection + coreference linking in one model. Span representation = concatenation of endpoint BiLSTM states + attention-weighted span embedding. Train on all pairwise mention scores jointly. F1 improves to ~80%.

  4. BERT-based (SpanBERT 2020): Fine-tune SpanBERT (pretrained with span masking for better span representations) for end-to-end coref. ~85% F1 on CoNLL. The strong contextual representations from pretraining handle long-range dependencies well.

  5. LLM-based (GPT-4/Claude, 2023+): Prompt LLMs to identify coreference chains. Zero-shot GPT-4 achieves ~80% F1 without any fine-tuning. Fine-tuned Llama models approach SpanBERT. Advantage: world knowledge for bridging; disadvantage: slower, inconsistent output format.

Relevance to eBay: Coreference is implicitly important for multi-turn conversational search ("Show me blue jeans" → "Do they come in a 32 waist?" — "they" = "blue jeans"), product Q&A, and long product descriptions with anaphoric references to features. Modern LLMs handle this implicitly but imperfectly.

Q24. What is NER, and how has it evolved? What are the tradeoffs between CRF-based and transformer-based approaches?

Model Answer:

Named Entity Recognition: Identify and classify mentions of named entities (persons, organizations, locations, products, etc.) in text. Standard formulation uses BIO tagging (B-entity = beginning of entity, I-entity = inside, O = outside).

Evolution:

  1. Rule-based: Gazetteers (entity lists) + pattern matching. High precision on known entities; zero recall for novel ones.

  2. CRF (2003-2012): Conditional Random Fields model P(tags | words) using features (word shape, prefix/suffix, capitalization, surrounding words, POS tags, gazetteer membership). Lafferty et al. CRF++ toolkit. The CRF constraint: learns valid tag transition probabilities (e.g., I-PER cannot follow B-ORG). CoNLL 2003 English NER: ~88% F1.

  3. BiLSTM-CRF (2015-2018): Replace hand-crafted features with character embeddings + word embeddings → BiLSTM sequence model → CRF output layer. Learned representations eliminate feature engineering. ~91-92% F1.

  4. BERT-based (2019+): Fine-tune BERT with a classification head (or CRF on top) for token-level entity labeling. F1 >93% on CoNLL 2003 (near human-level ~97%). The bidirectional context from BERT handles long-range dependencies and ambiguous entities well.

  5. LLM with instructions (2022+): Prompt GPT-4/Llama to extract entities in JSON format. Strong zero-shot; can generalize to novel entity types without fine-tuning. Less reliable format-wise; more expensive; not suitable for real-time at scale.

For eBay: NER is directly useful for extracting structured attributes from unstructured product titles/descriptions (brand, model, material, color, size). A fine-tuned BERT-based tagger on e-commerce text (eBERT) is more appropriate than LLM prompting for this high-throughput task.

Q25. What is TF-IDF and when is it still useful over neural approaches?

Model Answer:

TF-IDF (Term Frequency - Inverse Document Frequency):

TF-IDF(t, d) = TF(t, d) × IDF(t)
TF(t, d) = count(t in d) / total_terms(d)
IDF(t) = log(N / (1 + df(t)))   where N=num docs, df(t)=docs containing t

Rare terms that appear in a specific document get high TF-IDF; common stopwords get low IDF weight and are downweighted.

When TF-IDF is still useful: 1. Exact-match keyword retrieval: Finding documents with specific product IDs, model numbers, brand names, or codes. Dense embeddings can hallucinate similar-sounding but wrong strings; TF-IDF matches exactly. 2. BM25 (TF-IDF variant): The backbone of Elasticsearch and production search systems. BM25 outperforms vanilla TF-IDF and is often competitive with dense retrieval on keyword-heavy queries. eBay's RAG MT paper uses BM25 for exactly this reason. 3. Low-compute environments: TF-IDF retrieval is much cheaper than embedding-based (no GPU needed, no neural inference). At billion-document scale, BM25 can be done with an inverted index; dense retrieval requires FAISS/GPU. 4. Interpretability: TF-IDF scores are human-readable — you can explain why a document was retrieved. Neural embeddings are opaque. 5. Hybrid search: BM25 + dense retrieval combined (Reciprocal Rank Fusion) consistently outperforms either alone — leveraging TF-IDF's exact-match strength.

When neural beats TF-IDF: Semantic similarity (finding paraphrases, synonyms), handling morphological variants (German compound words), multilingual matching, long-tail vocabulary not in the index.

Q26. What are Word2Vec, GloVe, and FastText? How do they compare to contextual embeddings like BERT?

Model Answer:

Word2Vec (Mikolov et al. 2013): Predict a word from its context (CBOW) or predict context from a word (Skip-gram). Trained via negative sampling. Produces a fixed vector per word — captures semantic similarity (king - man + woman ≈ queen). Limitations: one embedding per word regardless of context; "bank" (financial) = "bank" (river).

GloVe (Pennington et al. 2014): Factorize the global word-word co-occurrence matrix. Ratios of co-occurrence probabilities encode relational meaning. Similar quality to Word2Vec but training is more stable and interpretable. Produces static context-free embeddings.

FastText (Bojanowski et al. 2017): Extends Word2Vec by representing words as bags of character n-grams. "apple" = sum of embeddings for "app", "ppl", "ple", "apple", etc. Handles rare words and morphological variants naturally (German compound words, product codes, brand names with misspellings). Produces OOV embeddings by summing n-gram embeddings. Excellent for e-commerce product search where product codes and variant spellings are common.

Contextual embeddings (BERT): Each token's representation depends on the full context via bidirectional attention. "bank" in "river bank" and "bank account" produce different vectors. Dramatically better for NLP tasks requiring disambiguation and rich semantics. Cost: much larger models, much higher inference cost.

When to use each: | Embedding | Use Case | |---|---| | FastText | Real-time embedding of rare/OOV tokens; product code search | | TF-IDF + GloVe | Lightweight classifiers; approximate semantic search | | BERT (eBERT) | High-quality understanding; reranking; classification; entity extraction | | Full LLM | Generation; complex reasoning; MT |

eBay angle: eBay built eBERT (BERT pretrained on 3B item titles) — a domain-adapted static embedding model for product similarity. This is the retrieval backbone for recommendation before the LLM era. The evolution is: FastText → eBERT → LiLiuM, each layer adding context and capability at increasing cost.

1F. Vision-Language Models

Q27. What is a VLM? How does CLIP work, and how do more recent models like LLaVA and Flamingo differ? [HF]

Model Answer:

CLIP (Contrastive Language-Image Pretraining, OpenAI 2021): Train a vision encoder (ViT) and text encoder (Transformer) jointly on 400M (image, text) pairs scraped from the web. Objective: for a batch of N pairs, maximize similarity between correct (image, text) pairs and minimize similarity to all N-1 incorrect pairs (contrastive/InfoNCE loss):

L = -Σ_i log(exp(sim(v_i, t_i) / τ) / Σ_j exp(sim(v_i, t_j) / τ))

Result: a shared embedding space where matching image-text pairs have high cosine similarity. Zero-shot classification: compare image embedding to class name embeddings. Excellent feature extractor; no generative capability.

Limitation of CLIP: Cannot generate text or answer questions — just compare embeddings.

LLaVA (Liu et al. 2023) — connector-based VLM: - Freeze CLIP vision encoder; freeze (or lightly fine-tune) a large LLM (LLaMA/Vicuna) - Train only a projector (simple 2-layer MLP) to map visual patch embeddings → LLM token space - Then visual instruction tuning: fine-tune projector + LLM on multimodal instruction data - Visual tokens are prepended to text tokens; LLM generates autoregressively - Cheap, modular. LLaVA-1.5 (2023): 13B model outperforming much larger contemporaries on VQA benchmarks

Flamingo (DeepMind 2022) — cross-attention fusion: - Interleave new cross-attention layers (attending to image features) with frozen LLM self-attention layers throughout the model - Visual conditioning at every LLM layer — more expressive integration - More parameters than connector-based; requires architecture modification

InstructBLIP (Q-Former): - Lightweight Querying Transformer with N learnable query tokens that cross-attend to image features - Q-Former compresses variable-length image patches → fixed N query representations - These N representations are input to the LLM — more flexible than linear projection, more compact than all patches

eBay VLM (arXiv:2602.11733, 2026) — Versley is a co-author: - LLaVA-OneVision architecture (updated connector-based approach with dynamic resolution) - Vision encoders: SigLIP2-SO400M-Patch14-384 or Qwen2.5-ViT - Text decoders: e-Llama-8B, LiLiuM-1B/4B/8B, Qwen3, Gemma3 - Three-stage training: Vision-Language Alignment → Mid-stage → Visual Instruction Tuning on 4M e-commerce examples - Key finding: replacing generic Llama-3.1-8B with e-Llama-3.1-8B gave these specific gains on e-commerce tasks (Table 1): - Aspect Prediction: 37.7 → 44.4 (+6.7 points) - Deep Fashion Understanding: 46.0 → 52.8 (+6.8 points) - Dynamic Attribute Extraction: 51.9 → 60.4 (+8.5 points)

Efficiency result (Table 3): Fine-tuned Gemma3-4B achieves F1 = 50.5 vs zero-shot Gemma3-27B F1 = 44.8 — the smaller fine-tuned model outperforms the 6.75× larger general model, with a 3.8× inference speedup. This is Tony's routing argument in a different form: domain adaptation beats brute-force model scale.

1G. General ML Fundamentals

Q29. Explain the bias-variance tradeoff. Give concrete examples in the context of LLMs/NLP. [HF]

Model Answer:

Formal decomposition: For a regression model f trained on sample D, predicting target y at input x:

E_D[(y - f(x))²] = Bias²[f(x)] + Var[f(x)] + Noise
Bias = E_D[f(x)] - y_true        (systematic error)
Var  = E_D[(f(x) - E_D[f(x)])²]  (sensitivity to training data)

High Bias (underfitting): Model is too simple to capture the true pattern. Same wrong answer regardless of training data. Example: linear classifier on non-linear data; n-gram language model on complex language → perplexity plateau.

High Variance (overfitting): Model memorizes training data; predictions vary wildly across different training samples. Example: small BERT fine-tuned on 100 examples for classification — perfectly fit training set, random on test set.

The tradeoff: Increasing model complexity reduces bias but increases variance. The optimal model capacity lies where total error is minimized.

In LLMs/NLP: - Underfitting: A 1B model on a complex e-commerce reasoning task may have too little capacity (high bias) - Overfitting: Fine-tuning 70B on 1k examples with high LR → test accuracy low, training accuracy 100% - Good calibration: The e-Llama CPT strategy — pretrain large to reduce bias; control LR/data ratio to prevent variance increase - Regularization: Dropout in fine-tuning; early stopping; LoRA (low rank = implicit regularization) - Data augmentation: Synthetic data generation (Tony's intent dataset, eBay's VLM data pipeline) reduces variance by increasing effective training set size

Double descent (2019): In very large models (overparameterized regime), variance surprisingly decreases after an initial peak. The reason: gradient descent on overparameterized models finds flat minima that generalize well. This explains why LLMs with billions of parameters on internet-scale data don't overfit catastrophically.

Q30. Compare L1 vs. L2 regularization. When do you prefer each? [HF]

Model Answer:

L2 (Ridge / Weight Decay): Add λ Σ wᵢ² to loss. Gradient: -2λwᵢ added to weight update → weights shrink proportionally to their magnitude. Weights approach zero but never reach it exactly (geometric decay). In practice via weight decay: w ← w(1-α·λ) - α·∇L.

L1 (Lasso): Add λ Σ |wᵢ| to loss. Subgradient: λ·sign(wᵢ). Constant force toward zero regardless of weight magnitude → weights can be driven to exactly zero (sparse solutions). L1 = automatic feature selection.

Geometric intuition: - L2 constraint: sphere (all weights penalized equally) - L1 constraint: diamond (corners align with axes → solutions tend to land on corners where some weights = 0)

When to prefer: | Use L2 | Use L1 | |---|---| | Most weights matter; don't want zeros | Explicit feature selection needed | | Continuous input features | High-dimensional sparse features (bag-of-words, TF-IDF) | | Neural network training (weight decay) | Linear models with sparse priors | | Want stable gradients | Want exact zeros (model compression) |

In LLMs: L2 weight decay is standard in optimizer configuration (AdamW weight decay λ=0.1 typical). L1 is uncommon in neural training (non-differentiable at zero; sparse gradients are inefficient on GPU). Pruning (Tony's work) is closer to L0 regularization — explicitly setting weights to zero — which achieves sparsity even more aggressively than L1.

AdamW vs Adam: Adam with L2 adds λw to gradient before scaling by second moment — doesn't achieve the same regularization as true weight decay. AdamW applies weight decay directly to weights (decoupled from gradient): w ← w(1-αλ) - α·m/√(v+ε). AdamW is the correct and standard approach for LLM training.

Q31. Precision, Recall, F1, AUC-ROC — when do you use each metric? How do you evaluate a domain-adapted LLM on e-commerce tasks? [HF]

Model Answer:

Precision = TP / (TP + FP)    — of predicted positives, how many are correct
Recall    = TP / (TP + FN)    — of all true positives, how many did we find
F1        = 2·P·R / (P+R)     — harmonic mean of precision and recall

AUC-ROC: Area under the ROC curve (TPR vs. FPR at all classification thresholds). AUC=0.5 = random; AUC=1.0 = perfect. Threshold-independent.

When to use each:

Metric Use When
Precision Cost of false positive is high: spam filter (don't block real email); ad targeting (don't waste budget)
Recall Cost of false negative is high: cancer screening (don't miss cancer); fraud detection (don't miss fraud)
F1 Balanced classification; imbalanced classes where accuracy is misleading
AUC-ROC Comparing model quality independently of threshold; works well when classes are balanced
AUC-PR (Precision-Recall curve) Better than AUC-ROC for highly imbalanced classes — ROC can look good even with many FPs when negatives >> positives
chrF/BLEU MT evaluation (character/n-gram overlap)
NDCG Ranking / recommendation (ordered results)
Perplexity LM quality; lower = better — but domain-specific and doesn't capture instruction-following quality

Tests whether the model has internalized e-commerce ontology. - Item Selection: Given a buyer query and a set of candidate items, select the most relevant. Tests ranking and understanding of buyer intent relative to product characteristics.

These exist because existing benchmarks don't capture the competence that matters in production. An e-Llama that scores identically to Llama on MMLU but +25% on Aspect Prediction is better for eBay even if a benchmark leaderboard treats them as equivalent.

Q32. What is backpropagation? Walk through the chain rule and gradient computation. [HF]

Model Answer:

Backpropagation is automatic differentiation applied to neural networks — efficiently computing ∂L/∂w for every parameter w using the chain rule.

Chain rule: For a composition L = f(g(w)):

∂L/∂w = (∂L/∂f) × (∂f/∂g) × (∂g/∂w)

Forward pass: Compute and cache intermediate activations z₁, z₂, ..., zₙ, and final loss L. These are needed in the backward pass.

Backward pass: Starting from L, compute gradients backward through the computation graph:

For a simple 2-layer network: x → z₁=W₁x → a₁=ReLU(z₁) → z₂=W₂a₁ → L

∂L/∂W₂ = ∂L/∂z₂ · a₁ᵀ
∂L/∂a₁ = W₂ᵀ · ∂L/∂z₂
∂L/∂z₁ = ∂L/∂a₁ ⊙ 1(z₁>0)    [ReLU gradient = indicator function]
∂L/∂W₁ = ∂L/∂z₁ · xᵀ

Key insight: Each gradient reuses the downstream gradient (already computed) multiplied by the local Jacobian of that operation. This avoids redundant computation — total cost is ~2× the forward pass.

In practice (PyTorch): Automatic differentiation builds a computation graph during the forward pass; loss.backward() triggers the backward pass traversal.

Numerical stability concern: Attention gradient involves softmax(QKᵀ/√d) — the softmax Jacobian has special structure. FlashAttention computes the backward pass in a numerically stable way without materializing the full n×n attention matrix.

Q33. What is discourse parsing, and how does it relate to LLM performance on complex tasks?

Model Answer:

Discourse = how sentences/clauses are related to form coherent text. Discourse parsing builds a tree (RST — Rhetorical Structure Theory) or graph (PDTB — Penn Discourse Treebank) over a document showing relations like: Elaboration, Contrast, Cause-Effect, Concession, Restatement.

Example: "The product arrived damaged [Cause]. Therefore, I returned it [Effect]." The "therefore" is an explicit discourse connective marking the relation.

Versley's background: Published on discourse relations, connective annotation, discourse for parsing — this is his linguistic home territory.

Connection to LLM performance: - LLMs that understand discourse structure (causal chains, concessive structure) summarize and reason better - Chain-of-thought prompting implicitly forces the model to make discourse structure explicit ("therefore", "however", "as a result") - Coreference (Versley's PhD) is intimately tied to discourse — entity mentions across a discourse chain maintain referential identity across discourse segments - Product descriptions use implicit discourse: "This jacket features... However, note that... Therefore, it is best for..." — understanding the discourse helps attribute extraction

Modern LLM status: Large models implicitly learn discourse relations from pretraining but don't explicitly parse RST trees. Encoder models fine-tuned on RST corpora still outperform LLMs on explicit discourse parsing. But for downstream tasks (summarization, QA), the implicit discourse understanding in LLMs is often sufficient.

Q34. What are n-gram language models, and what were their key limitations that neural models resolved?

Model Answer:

N-gram LM: Approximate the probability of a sequence by the product of conditional probabilities using a Markov assumption:

P(w₁, ..., wₙ) = Π P(wₜ | wₜ₋₍ₙ₋₁₎, ..., wₜ₋₁)

Estimate from corpus counts: P(w_t | w_{t-2}, w_{t-1}) = count(w_{t-2}, w_{t-1}, w_t) / count(w_{t-2}, w_{t-1}).

Smoothing: Unseen n-grams would have P=0. Kneser-Ney smoothing is the classic solution: recursively back off to (n-1)-gram counts with a continuation probability component. Kneser-Ney remains competitive for specialized small-corpus tasks.

Limitations: 1. Sparse data / high n: With vocabulary V, an n-gram model has V^n possible n-grams. At n=5, most are unseen. Practical n≤5. 2. No semantic generalization: "the dog bit the man" and "the canine bit the man" are completely different to an n-gram model — no notion that "dog" and "canine" are synonyms. 3. No long-range dependencies: Markov assumption limits to n-1 preceding tokens. Understanding "The customer who complained last week about the missing lens cap called again about the same issue" requires tracking the subject across multiple clauses — impossible for n≤5. 4. No sub-word generalization: "internationalization" and "internationalize" are unrelated tokens.

Neural resolution: Word embeddings (Word2Vec, GloVe) gave semantic generalization; RNNs gave theoretically unlimited context (in practice ~50-100 tokens effective); Transformers gave full-sequence attention (attention weights are a useful diagnostic, NOT reliable explanations - Jain & Wallace 2019; Wiegreffe & Pinter 2019).

Why know this: Versley did serious work in statistical NLP (HMMs, discriminative parsers, kernel methods) before deep learning. He will appreciate that you know the historical baseline and can articulate exactly what neural models gained.

Q36. What is PCA? How would you use it in an NLP/ML pipeline? [HF] (Reported at eBay interviews)

Model Answer:

PCA (Principal Component Analysis): Find the directions (principal components) of maximum variance in high-dimensional data. Formally: compute covariance matrix Σ = XᵀX/(n-1), find eigenvectors/eigenvalues, project data onto the top-k eigenvectors.

The k-th principal component is the direction that captures the k-th most variance, orthogonal to all previous components.

Algorithm (practical): 1. Center data: X ← X - mean(X) 2. Compute SVD: X = UΣVᵀ — columns of V are principal components 3. Project: X_reduced = X · V[:, :k]

When to use in NLP/ML: - Visualization: Project 768-d BERT embeddings to 2D (via PCA or t-SNE) to inspect cluster structure, domain gaps - Noise reduction: NLP features (TF-IDF vectors) may have ~100k dimensions; PCA reduces to 100-500 without losing much signal - Speeding up downstream classifiers: Reduce feature dimensionality before SVM or logistic regression - Whitening: Decorrelate features before training; sometimes helps shallow models - Intrinsic dimension estimation: How many principal components are needed to capture 95% variance? Tells you about data complexity

When NOT to use: Deep learning — models learn their own representations end-to-end; PCA on raw features discards non-linear structure and is generally counterproductive.

SVD connection: PCA is SVD on the data matrix (or eigendecomposition of covariance). Truncated SVD (used in LSA for documents) is exactly this applied to term-document matrices.

Q37. What is SVM, and what is the kernel trick? [HF] (Reported at eBay interviews)

Model Answer:

SVM (Support Vector Machine): Binary classifier that finds the hyperplane maximizing the margin (distance) to the nearest training points from each class.

Hard-margin formulation (linearly separable):

min_{w,b} ½||w||²    s.t.  yᵢ(w·xᵢ + b) ≥ 1  ∀i

The decision boundary is w·x + b = 0; margin = 2/||w||. Maximizing margin = minimizing ||w||².

Soft-margin (non-separable): Introduce slack variables ξᵢ ≥ 0:

min ½||w||² + C Σ ξᵢ    s.t.  yᵢ(w·xᵢ+b) ≥ 1 - ξᵢ

C controls the bias-variance tradeoff: large C = low slack = low bias/high variance; small C = more slack = high bias/low variance.

Kernel trick: SVMs only use inner products ⟨xᵢ, xⱼ⟩ in the dual formulation. If we replace the inner product with a kernel function K(xᵢ, xⱼ) = ⟨φ(xᵢ), φ(xⱼ)⟩, we implicitly work in a high (or infinite) dimensional feature space without explicitly computing φ(x).

Common kernels: - RBF/Gaussian: K(x,z) = exp(-γ||x-z||²) — infinite-dimensional feature space; captures non-linear similarity - Polynomial: K(x,z) = (x·z + c)^d - Linear: K(x,z) = x·z — equivalent to standard SVM; fastest

Modern NLP use: SVMs were SOTA for text classification before deep learning (LibSVM with linear kernel + TF-IDF features). Now largely replaced by BERT fine-tuning, but SVMs on BERT embeddings are still occasionally used for low-data regimes (the margin maximization provides a stronger inductive bias than logistic regression when training set is small).

Q38. Explain A/B testing for ML models. What is statistical power, and how do you determine sample size? [HF]

Model Answer:

A/B testing: Randomly split traffic — control (A, current model) vs. treatment (B, new model). Measure the difference in the primary metric (CTR, GMV, conversion rate) over a sufficient period.

Statistical framework: - Null hypothesis H₀: Treatment effect = 0 (models are equivalent) - Significance level α (Type I error rate): P(reject H₀ | H₀ true). Standard: α=0.05 - Statistical power 1-β (1 - Type II error rate): P(reject H₀ | H₁ true, i.e., effect is real). Standard: 1-β=0.8

Sample size formula (two-sample proportion test):

n = (z_{α/2} + z_β)² × (p₁(1-p₁) + p₂(1-p₂)) / (p₁ - p₂)²

For α=0.05 (z=1.96), power=0.8 (z=0.84), p₁=0.05 (baseline CTR), p₂=0.055 (5% lift):

n ≈ 2.8 × (0.05×0.95 + 0.055×0.945) / (0.005)² ≈ 50,000 per arm

Common mistakes: - Peeking: Stopping the test early when significance is reached inflates Type I error (Bonferroni correction or sequential testing helps) - Multiple comparisons: Testing 10 metrics at α=0.05 means 0.4 expected false discoveries; use Bonferroni or Benjamini-Hochberg - Network effects: In recommendation systems, changing what A sees might affect B's options (interference). Use holdout cells or cluster-level randomization - Novelty effect: Users engage more with anything new; wait for behavior to stabilize

eBay context: eBay A/B tests ML models constantly (LiLiuM vs Llama for listing generation, new ranking features, RAG for MT). Primary metrics: listing creation rate, sold items, GMV per lister. The 50%+ increase in listing creation from Magical Listing is the result of large-scale A/B testing.

Q41. What does eBay's vocabulary customization paper (Versley et al., Sept 2025 (arXiv preprint)) contribute beyond LiLiuM's original tokenizer? [HF]

Model Answer:

Problem it solves: LiLiuM was trained from scratch with a custom 65k vocabulary — but what about adapting an existing pretrained model (like Llama 3.1) to have an efficient domain vocabulary without full retraining from scratch?

Token fertility problem: A general tokenizer (Llama's 128k BPE) has high fertility on domain text: - "HDMI" → might be 3-4 tokens in a general tokenizer → 1 token in domain tokenizer - "EU 42" (shoe size) → 2-3 tokens → 1 token - "Makita" (brand) → multiple tokens → 1 token - Each unnecessary token costs one full forward pass during generation → direct inference latency cost

Method: The algorithm appends new domain-specific BPE merges to the end of the existing merge list. Because BPE applies merges in order (earlier merges have priority), appending new merges means all existing tokenizations are preserved exactly — appended merges can only create additional compressions on top of the existing result, never undo them.

The monotone efficiency guarantee: Every input sequence is segmented into at most the same number of tokens as it was under the original tokenizer. This holds universally for all inputs — not just domain-specific text, not just sequences containing the new tokens. The reason: appended merges cannot affect the outcome of earlier merges, so the existing tokenization is always a valid upper bound, and new merges can only reduce it.

Embedding initialization for new tokens: New vocabulary tokens are initialized with the average of the embedding and projection vectors of their constituent sub-tokens under the original tokenizer. For example, if "Makita" was previously tokenized as ["Mak", "ita"], the new "Makita" token embedding is initialized as the mean of the "Mak" and "ita" embedding vectors. This gives the model a meaningful starting representation in the residual stream without disrupting existing token embeddings, minimizing the amount of continued pretraining needed to integrate the new tokens.

Results (verified from paper): - Extending Llama 3.1 with 30k domain tokens: up to 20% input sequence compression on e-commerce tasks - This translates to 20-30% higher throughput (29.2% throughput gain measured on 3000-word sequences) - The model adopts new tokenization in ~98% of cases for sequences longer than 15 words

Why this matters beyond LiLiuM: LiLiuM's from-scratch custom tokenizer required training the whole model from scratch on that vocabulary. This paper shows you can augment an adapted model (e-Llama) with vocabulary customization as a post-hoc step — combining CPT domain knowledge with tokenizer efficiency. The monotone guarantee makes this a production-safe optimization: no request can become slower after deployment, only the same or faster. You can ship this without regression-testing throughput on every edge case.

Tony's angle: "This is the tokenizer-level version of my routing work — both are about getting more output per dollar by reducing unnecessary computation. The monotone guarantee is elegant because it transforms the deployment risk from 'will this regress some cases?' to 'this categorically cannot regress any case.' My LLM Router has an analogous property: the fallback to the strong model is always available, so quality can only be equal to or better than strong-model-only, not worse. Both are safe-by-design optimizations, which is what production systems need."

Q43. How does eBay use LLMs for search ranking and advertising? Walk through the distillation pipeline. [HF]

Model Answer:

Context: eBay search processes billions of queries; a ranking model must score millions of candidate items per query in real time (<100ms budget). LLMs are accurate but slow; fast models (bi-encoders) are less accurate. The solution: knowledge distillation.

LLM-to-BERT distillation for search relevance ("From Reasoning LLMs to BERT", eBay 2025):

  1. Teacher (LLM): A large reasoning LLM (GPT-4 or similar) assesses relevance of (query, item) pairs. Produces rationale + binary/scalar relevance score. This is the "LLM-as-judge" approach for labeling — critically, it replaces or supplements click-through signal which is biased (users can only click what they can see, and existing ranking affects what they see).
  2. Assistant (cross-encoder): Fine-tuned BERT-large cross-encoder trained to predict the teacher's relevance scores. Cross-encoder: takes (query, item) concatenation → single relevance score. Slower than bi-encoder but more accurate than pure embedding similarity.
  3. Student (bi-encoder): Small BERT model trained via knowledge distillation on cross-encoder outputs. Bi-encoder: query and item encoded separately → cosine similarity. Can be indexed (item embeddings precomputed) → fast approximate nearest-neighbor at serving time.

Serving pipeline: - Retrieval stage: Bi-encoder student embeds query → ANN search over pre-indexed item embeddings → top-100 candidates (ms latency) - Reranking stage: Cross-encoder assistant scores top-100 pairs → rerank → top-20 (tens of ms) - Final ranking: Business-rule layer (diversity, price, seller rating)

Advertising — LLMDistill4Ads (arXiv:2508.03628):

The core contribution of LLMDistill4Ads is replacing biased click signal with LLM-generated relevance signal for training ad keyphrase ranking models. Click data has two fundamental problems at eBay's scale: (1) 95% of items receive no clicks (extreme sparsity), and (2) middleman bias — search filters and existing ranking determine which items are observed, so click signal only measures quality conditioned on being shown. An LLM judge (Mixtral 8x7B teacher) evaluates (keyphrase, product) pairs directly, providing debiased, dense relevance labels.

The operational pipeline to deploy LLM judgment at scale uses the same three-tier distillation hierarchy as the search ranking work: - LLM teacher (Mixtral 8x7B) → relevance labels for (keyphrase, product) pairs - Cross-encoder assistant (microBERT) → intermediate calibration layer - Bi-encoder student (microBERT) → production EBR retrieval model

The two-stage distillation (teacher → cross-encoder → bi-encoder) outperforms direct distillation (teacher → bi-encoder), with two-stage Pearson loss achieving F1=0.88 vs direct distillation F1=0.83.

To Judge or not to Judge (ECAI 2025): Related paper evaluating when LLM-as-judge is warranted for keyphrase relevance — addresses the question of when the LLM judgment cost is justified vs. when click signal is sufficient.

def wordBreak(s, wordDict):
    words = set(wordDict)
    n = len(s)
    dp = [False] * (n + 1)
    dp[0] = True
    for i in range(1, n+1):
        for j in range(i):
            if dp[j] and s[j:i] in words:
                dp[i] = True
                break
    return dp[n]

def hasPathSum(root, target):
    if not root: return False
    if not root.left and not root.right:
        return root.val == target
    return hasPathSum(root.left, target - root.val) or \
           hasPathSum(root.right, target - root.val)

def letterCombinations(digits):
    phone = {'2':'abc','3':'def','4':'ghi','5':'jkl',
             '6':'mno','7':'pqrs','8':'tuv','9':'wxyz'}
    result = []
    def backtrack(idx, current):
        if idx == len(digits): result.append(current); return
        for c in phone[digits[idx]]:
            backtrack(idx+1, current+c)
    if digits: backtrack(0, "")
    return result

import numpy as np
def kmeans(X, k, max_iter=100):
    centroids = X[np.random.choice(len(X), k, replace=False)]
    for _ in range(max_iter):
        # assign
        dists = np.linalg.norm(X[:,None] - centroids[None,:], axis=2)  # (n,k)
        labels = dists.argmin(axis=1)
        # update
        new_centroids = np.array([X[labels==i].mean(0) for i in range(k)])
        if np.allclose(centroids, new_centroids): break
        centroids = new_centroids
    return labels, centroids

Logistic Regression with gradient descent:

def sigmoid(z): return 1 / (1 + np.exp(-z))

def logistic_regression(X, y, lr=0.01, epochs=1000):
    w = np.zeros(X.shape[1]); b = 0
    for _ in range(epochs):
        y_hat = sigmoid(X @ w + b)
        dw = X.T @ (y_hat - y) / len(y)
        db = (y_hat - y).mean()
        w -= lr * dw; b -= lr * db
    return w, b

TF-IDF:

import math
def tfidf(corpus):
    # TF: term frequency per doc
    tf = [{term: doc.count(term)/len(doc) for term in set(doc)} for doc in corpus]
    # IDF
    N = len(corpus)
    df = {}
    for doc in corpus:
        for term in set(doc): df[term] = df.get(term, 0) + 1
    idf = {term: math.log(N / (1 + df[term])) for term in df}
    # TF-IDF
    tfidf_vecs = [{term: tf_doc.get(term, 0) * idf.get(term, 0)
                   for term in idf} for tf_doc in tf]
    return tfidf_vecs

Q51. Design an LLM router for cost-efficient inference at eBay scale. [HF — Tony's home territory]

System Design Outline:

Architecture:

Query In
   ↓
[Feature Extraction] — VERY fast (<5ms)
  - Query length, complexity proxy (perplexity under 1B model)
  - Task type classifier (MT, attribute extraction, listing generation, QA)
  - Urgency signal (real-time vs. batch)
   ↓
[Router Model] — lightweight (BERT-tiny / rule-based hybrid)
  - Outputs: {1B, 7B, 13B, external-API}
  - Trained on: (query, model, outcome) triples — which model was good enough for this query?
   ↓
[Dispatch]
  - 1B: structured, repetitive tasks (format conversion, simple extraction)
  - 7B: standard listing generation, single-domain MT
  - 13B: complex multi-attribute extraction, ambiguous queries
  - External: novel task types, high-stakes edge cases (low volume)
   ↓
[Monitoring & Feedback Loop]
  - Per-model quality metrics (BLEU for MT, attribute F1 for extraction)
  - Router accuracy (compare router choice vs. oracle — cheapest model that would have been correct)
  - Cost dashboard: $ per million queries per task type
  - Drift detection: if quality drops for a task type, route up-tier automatically

Key tradeoffs: - Router adds latency (5-20ms overhead); must be much faster than model inference - Router training requires labeled data (which model was "good enough"?) — requires human eval or LLM-as-judge - Cold start: new task types default to 13B until enough data accumulates - Cascade routing alternative: try 1B → if confidence low → upgrade to 7B → etc. Adds latency; avoids needing explicit router training

Deployment: - Router is stateless, horizontally scalable - A/B test routing policy changes against quality metrics before full deployment - Log all routing decisions for offline analysis

Q52. Design eBay's machine translation system for cross-border product titles.

System Design Outline:

Scale: 100M+ product listings; ~40 language pairs; real-time at listing creation + batch for existing catalog

Architecture:

  1. Request intake: Seller submits product title in source language → route to MT service
  2. RAG retrieval: Query bilingual title index (BM25 over 10M+ aligned pairs) → retrieve 5 similar translated examples per language pair
  3. LLM translation: LiLiuM-7B or e-Llama-8B with 5-shot prompt: [Example 1: EN→DE] ... [Translate:] {title} → translated title
  4. Quality filter: Automatic chrF score against a baseline MT model (mBART/NLLB) — if large divergence → flag for human review or fallback to NLLB
  5. Post-processing: Title length normalization (eBay title limits: 80 chars); forbidden content filter; brand name preservation check
  6. Feedback loop: Seller edits, buyer engagement data (do buyers click translated titles?) → training signal for continuous improvement

Evaluation: - Automatic: chrF (per eBay MT paper), BLEU, COMET (neural MT metric) - Human: bilingual quality raters for sample of 1k titles/language pair monthly - Business: CTR and conversion rate for translated listings in target market

Efficiency levers: - Cache translations for exact-match titles (many eBay listings are duplicates) - Batch mode for catalog-wide translation overnight; real-time for new listings - LiLiuM-1B for high-volume low-complexity pairs (EN-DE: established model); LiLiuM-7B for low-resource pairs (EN-PL, EN-SE: where RAG helps most)
