ML System Design Playbook — eBay MLE Amsterdam

Priority / How to use: This is your Stage 2 (and late Stage 1) reference. Read Part 1 once to internalize the framework skeleton. Then drill each of the 5 eBay-flavored designs until you can narrate the full end-to-end story in 30–40 minutes, pushing back on constraints and proposing iterations. Part 3 gives you compact answers to curveballs — memorize the numbers.

PART 1: REUSABLE ML SYSTEM DESIGN FRAMEWORK

This 14-step skeleton applies to every ML system. In an interview, you adapt and trim. Always start by clarifying requirements before diving into model choices — interviewers penalize candidates who jump to architecture before understanding constraints.

Step 0: Problem Framing and Clarification

Before touching any design, ask:

  1. What business objective does this serve? (Revenue, engagement, cost reduction, trust/safety)
  2. Who are the users and what is the interaction pattern? (Buyer, seller, internal tool, API consumer; one-shot vs. session-based)
  3. What is the scale? (Queries per second, catalog size, user base, data freshness)
  4. What are the latency constraints? (p50 / p99 SLAs; synchronous vs. async; streaming vs. batch)
  5. What is the cost budget? (GPU/hour, API cost ceiling, inference cost per query)
  6. What data do we already have? (Logs, labels, proprietary catalog, third-party signals)
  7. What does "success" look like in 6 months? (Quantified)
  8. Are there regulatory or fairness constraints? (PII, GDPR in EU, bias, cross-border rules)

eBay-specific defaults to assume unless told otherwise:

  • Scale: 2B+ active listings, 135M monthly active users, hundreds of QPS on core ranking/search paths
  • Latency: 200–500 ms total page load; ML component budget ~50–150 ms
  • Multilingual: EN/DE/FR/ES/IT as primary markets
  • Cost sensitivity: very real — eBay invests heavily in efficiency (LiLiuM custom tokenizer 34% speedup is operational, not academic)

Step 1: Success Metrics

Define metrics in two layers before designing anything.

Classification (relevance, intent) AUC-ROC, F1 PR-AUC when positives sparse
Generation (MT, listing) chrF (MT), ROUGE (summarization) BERTScore, human eval

Metric selection rationale you must articulate:

  • NDCG@K is position-aware — top slots matter more on a search results page
  • chrF is character n-gram F-score; better than BLEU for short e-commerce titles (BLEU 4-gram statistics are too sparse for 11-word titles — eBay found this empirically, arXiv 2409.12880)
  • AUC-ROC is threshold-agnostic; prefer PR-AUC under heavy class imbalance (rare positives = relevant items)

Online Metrics (production A/B test signal)

Search relevance Click-through rate (CTR), GMV per search Bounce rate, pogo-sticking
Recommendations CTR, add-to-cart rate, conversion Session length, diversity

Key principle: Never optimize a single metric. eBay's Mercury platform explicitly balances relevance + business KPIs. Always name at least one guardrail metric to show you understand side effects.

Step 2: Data and Labels

Data Sources (eBay-specific)

  • Implicit behavioral signals: clicks, purchases, dwell time, add-to-cart, saves, messages to sellers — massive volume, but biased toward popular items and position
  • Explicit editorial labels: human relevance judges (expensive; used sparingly for offline eval gold sets)
  • LLM-generated labels (LLM-as-judge): eBay uses LLM judgments for advertiser keyphrase relevance (arXiv 2505.04209); scales to billions of pairs but inherits LLM biases
  • Product knowledge graph: structured attributes (category, price, condition, item specifics) — ground truth for attribute extraction tasks
  • Seller-provided data: titles, descriptions, images, price history — noisy but comprehensive
  • Cross-lingual parallel data: eBay's MT training corpus (included in LiLiuM pretraining)

Label Quality Issues and Mitigations

Issue Manifestation Mitigation
Position bias Higher-ranked items get more clicks regardless of true relevance Inverse Propensity Score (IPS) weighting; randomized exploration experiments
Selection bias Only shown items can be clicked Counterfactual evaluation; bandit logging
Sparse positives <1% of catalog appears in any given session Hard negative mining; BPR loss
Stale labels User preferences shift; yesterday's popular = today's outdated Time-decayed weighting; recency features
Feedback delay Purchase signal lags click by days/weeks Multi-task learning with click proxy + purchase target

Data Volume / Freshness Requirements

  • Pretraining-scale: 3T tokens for LiLiuM; 1T for e-Llama continued pretraining
  • Fine-tuning scale: 4M instruction examples for eBay VLM (arXiv 2602.11733); 10k–1M for task-specific SFT
  • Online serving: most ranking models need <24h feature freshness; real-time models need <1min (Mercury's near-real-time queue)

Step 3: Features

Feature Categories

Query features (search/ranking):

  • Raw query text, tokenized, character n-grams
  • Query embeddings (eBERT or LiLiuM encoder)
  • Query category prediction (soft taxonomy labels)
  • Historical query CTR, session context

Item features:

  • Title, description, category path (structured)
  • Visual embeddings (product images via ViT / SigLIP2)
  • Structured attributes: price, condition, seller rating, shipping speed, return policy
  • Historical item CTR/CVR, GMV, recency of listing

User features:

  • Browsing history embeddings (sequence of item IDs)
  • Purchase history, categories of interest
  • Demographic proxies (inferred from behavior, not stored directly for GDPR)
  • Session context: device, time of day, geo (country/language)

Cross features:

  • Query–item semantic similarity (cosine of embeddings)
  • Query–item category match score
  • User–item co-occurrence in sessions

Feature Engineering Principles:

  • Prefer learned representations (embeddings) over hand-crafted for semantic similarity
  • Keep exact-match signals (BM25 term overlap, category exact match) as separate features — neural does not always beat keyword matching for product IDs and brand names
  • Log-transform count features (CTR, views) to normalize skew
  • Normalize continuous features per-query for ranking models

Step 4: Candidate Generation (Two-Stage Architecture)

For any system with a very large item space (eBay: 2B listings), a two-stage retrieve-then-rank architecture is mandatory — you cannot run expensive models over the full catalog at query time.

Stage 1: Candidate Retrieval

Goal: Recall@K as high as possible from N candidates (K ≈ 100–1000, N = 2B)

Approaches:

  1. BM25 / Sparse retrieval (Lucene/Elasticsearch): fast, no GPU, excellent for exact-match (brand names, product IDs, UPC codes). eBay uses Lucene 8.11.3 with OKAPI BM25 for MT RAG retrieval.
  2. Dense retrieval (bi-encoder ANN): query and item encoded independently to fixed-dimension vectors; cosine similarity via ANN (FAISS HNSW or Vertex AI Vector Search). eBay uses Vertex AI Vector Search for recommendations (Google Cloud blog). Enables semantic matching.
  3. Hybrid: RRF (Reciprocal Rank Fusion) over BM25 and dense candidate lists. Typical eBay-scale setup. Dense catches semantics; BM25 catches exact product codes.
  4. Item-to-item collaborative filtering: given user's historical items, retrieve similar items via co-purchase matrix factorization (WALS / LightFM / two-tower trained on co-engagement).

ANN index choices:

  • FAISS HNSW: in-memory, low latency (~5ms), good for <100M vectors
  • Vertex AI Vector Search (ScaNN): scalable to billions, managed, integrates with GCP (eBay is on GCP/Azure hybrid)
  • Cost of reindexing: embeddings must be re-computed when the encoder model changes — plan incremental indexing for real-time listings

Stage 2: Ranking

Goal: NDCG@K precision over the retrieved candidate set

Model choices:

Model Type Latency Quality Notes
Gradient Boosted Trees (LightGBM/XGBoost) <10ms Good on structured features Baseline; interpretable; handles missing values
Linear model (logistic regression) <1ms Baseline Great for rapid iteration, low latency
Deep CTR (Wide & Deep, DeepFM, DCN) 20–50ms Better cross-feature learning Standard for production rankers
Cross-encoder (BERT on query+item) 100–500ms High quality Too slow for L1 ranking; use as L2 reranker on top-20
LLM reranker (RankZephyr, eBay LLMDistill4Ads) 200ms–2s Highest quality Teacher for distillation; not direct serving

eBay's distillation pipeline (from LLMDistill4Ads paper, arXiv 2508.03628): LLM teacher (high quality, slow) → cross-encoder assistant (medium) → bi-encoder EBR student (fast, production). This is the standard 3-stage distillation for production NLP ranking at eBay.

Loss Functions

Training Infrastructure

  • eBay pretraining scale: 480 H100 GPUs, 60 nodes, NVLink + InfiniBand; Megatron-LM with 3D parallelism; Flash-Attention-2 (from e-Llama paper, arXiv 2501.09706); well-tuned runs achieve ~40–47% MFU (model FLOP utilization) on H100 clusters
  • Fine-tuning scale: 8–64 A100/H100 GPUs; FSDP or DeepSpeed ZeRO-2/3; LoRA for PEFT
  • Ranking models: typically CPU-trainable with Spark/distributed gradient boosted trees for feature-heavy models; GPU for deep CTR

Key Training Decisions (with rationale)

Learning rate for continued pretraining: Use 10% of original pretraining LR (e-Llama: 3.0e-5 for 8B). Higher LR causes catastrophic forgetting of general capabilities. eBay ablated this explicitly: LR=3.0e-3 dropped general score from 43.3 to 34.5 while e-commerce score barely moved.

Data mixing ratio for domain adaptation: 50% domain / 50% general is optimal in eBay's ablation. Above 50% domain shows diminishing e-commerce returns with increasing general degradation. The intuition is not just "general data as regularizer" — it's mechanistic: the model needs exposure to general-format instruction-following, reasoning, and in-context learning examples that only exist in general-domain corpora. A 100%-domain model loses those capabilities and becomes brittle on zero-shot tasks. See the Pretraining Data Curation section below for deeper treatment.

Batch size: LiLiuM / e-Llama used ~11.8M tokens per batch (critical for stable gradient estimates at pretraining scale). Larger batch = less noise but needs LR scaling (linear scaling rule or square-root rule).

Checkpoint averaging: Average parameters from last N checkpoints before deployment. Smooths out oscillations from late-stage training. Free quality boost, used in both LiLiuM and e-Llama.

Step 5B: Pretraining Data Curation — Versley's Home Turf

This section is specifically high-yield because Versley leads LLM pretraining at eBay. Expect depth here.

Why Data Curation Matters at Scale

At 3T tokens, even a 1% contamination rate means 30B tokens of low-quality data. A 1% duplication rate means 30B wasted compute tokens. At eBay's e-Llama scale (340k H100 GPU-hours), bad data is expensive.

Quality Filtering Pipeline

Standard pipeline for web-scale pretraining data:

Raw web crawl
  ↓
1. Language detection (fastText, <1ms per doc)
   └─ Filter to target languages; proportional weighting for multilingual

2. Heuristic cleaning
   ├─ Min/max document length thresholds
   ├─ Repetition filters (remove docs where N-gram repetition > threshold)
   ├─ Symbol/punctuation ratio (remove docs that are mostly code fragments or tables)
   └─ URL/email density filter

3. Perplexity-based quality filtering
   ├─ Train a small KenLM 5-gram model on a high-quality seed corpus (Wikipedia, curated books)
   ├─ Score each document: low-perplexity = high quality (fluent, well-structured text)
   ├─ Set language-specific thresholds via manual inspection of perplexity percentiles
   └─ Remove very high-perplexity outliers (garbled text, spam) and very low-perplexity documents
      (templated machine-generated text that would cause memorization)

4. PII scrubbing
   └─ Named-entity detection → replace emails, phone numbers, credit card numbers

5. Deduplication (most critical for model quality)
   ├─ Exact deduplication: hash-based (MD5 over normalized text)
   └─ Fuzzy deduplication: MinHash + LSH
       ├─ Extract character or word n-grams from each document
       ├─ Compute MinHash signature (B bands × R rows → target 75–85% Jaccard similarity threshold)
       └─ LSH groups documents with similar signatures → remove near-duplicates

Why deduplication matters beyond storage:

  1. Memorization: Models memorize training examples, especially those seen many times. Duplicated data = training the model to recite verbatim text, not generalize. This is a privacy risk (membership inference) and a quality risk.
  2. Train/test leakage: If benchmark tasks were web-scraped and are duplicated in training data, the model appears to generalize when it is actually memorizing. This is the benchmark contamination problem (see Step 6).
  3. Compute waste: Duplicated tokens contribute no new information gradient; they are pure compute waste at eBay's scale.

eBay-specific considerations for LiLiuM's 3T tokens:

  • Heavy e-commerce data (product listings, seller descriptions) has extreme duplication — many sellers list identical products with copy-pasted descriptions. MinHash fuzzy deduplication at product-description level is essential.
  • Cross-lingual duplicates: the same product title in EN and DE are not duplicates — do not over-deduplicate multilingual data.
  • The tension: you want enough domain data for terminology coverage but not so much that the model memorizes catalog entries.

Data Mixing Strategy

The 50/50 domain/general split (eBay's empirically optimal ratio for e-Llama) balances:

Concern Too much domain data Too much general data
E-commerce vocabulary Good Terminology gaps
In-context learning Degraded Preserved
Reasoning chains Degraded Preserved
Instruction following Degraded Preserved
Machine translation OK (MT-parallel in domain) Good (more linguistic variety)
Generalization Brittle on OOD tasks Robust

Multilingual data weighting: LiLiuM explicitly weights multilingual data to cover EN/DE/FR/ES/IT/NL proportionally to eBay's cross-border trade volume — not proportional to web crawl frequency (which would under-represent non-English markets).

Benchmark Contamination Risk

LiLiuM trained on 3T tokens — how do you know benchmark tasks weren't in the corpus?

Detection methods:

  1. N-gram overlap: check whether test-set answer strings or question preambles appear in training data (exact or fuzzy)
  2. Canary injection: insert known synthetic strings in training data; test whether model completes them — measures memorization rate
  3. Temporal splitting: prefer benchmarks whose test sets were published after the training data cutoff
  4. Dynamic benchmarks: LiveBench-style benchmarks release new questions monthly from post-cutoff sources; decontamination is built in by design

Practical response to "how do you know?": "We can't fully rule it out. We use temporal cutoffs so our training corpus predates the benchmark release dates where possible. For internally curated benchmarks like eComMMMU, the test set was held out from pretraining and only used for evaluation — no feedback loop. For public benchmarks, we report results with contamination caveats and prefer LiveBench-style dynamic evaluation for production quality tracking."

Step 6: Offline Evaluation

Evaluation Principles

  1. Use a held-out test set that mirrors production distribution — not random split; temporal split is better (train on older data, test on newer) to avoid leakage
  2. Evaluate the full pipeline, not just the model — a great ranker on a bad candidate set fails in production
  3. Use the metric that corresponds to the serving scenario — if you serve top-5 results, measure NDCG@5, not NDCG@100
  4. Compare against a strong baseline — random, BM25, or the existing production model

eBay-Specific Evaluation Benchmarks

  • Aspect Prediction, AP Multiple Choice, Price Multiple Choice, Most Common Aspects, MCA Multiple Choice: e-commerce NLU benchmark tasks from eBay's e-Llama evaluation suite; baseline ~36–61 for Llama-3.1 8B; e-Llama 8B achieves ~54–75 (+25% average)
  • eComMMMU: public multimodal e-commerce understanding benchmark; the eBay VLM (e-Llama-8B based) achieves 58.3%
  • chrF for MT: eBay RAG-for-MT paper shows up to +15.3% chrF improvement for EN-PL pair with 5-shot RAG
  • LLM-as-judge: eBay uses LLM judges for advertiser keyphrase relevance at scale; must account for position bias (judge prefers answer A over B regardless of quality when A comes first), verbosity bias (longer answers rated higher), and self-enhancement bias (model prefers its own outputs)

LLM Evaluation: Why Benchmarks Lie and What You Do About It

The problem: Static benchmarks give a point estimate that conflates:

  • Genuine capability generalization
  • Memorization of training examples that overlap with test set
  • Format-matching to the benchmark's particular framing
  • In-context learning from the few-shot prompt, not model knowledge

What you do about it:

  1. Contamination audit: before reporting a number, check n-gram overlap between training data and benchmark. If >5% test questions appear in training, caveat the result.
  2. Multiple benchmark agreement: if a model improves on MMLU but degrades on a held-out eBay internal benchmark, trust the internal benchmark — it's not contaminated.
  3. Capability elicitation vs. benchmark performance: a model may know the answer but fail the benchmark's specific format. Use chain-of-thought prompting and compare zero-shot vs. few-shot to disentangle.
  4. LLM-as-judge with calibration: when using LLM judges at scale, run human spot-checks on 2–5% of judgments to measure judge accuracy. If judge accuracy < 80%, the judge is not reliable enough for training signal.
  5. Dynamic evaluation: for production quality monitoring, use a rolling held-out set drawn from recent eBay queries — never recycled into training. This mirrors real distribution drift.

└─ Total: 300ms–2s depending on model and output length

Inference Optimization Techniques

KV Cache: Stores past K and V tensors during autoregressive generation. Memory formula:

KV cache memory = 2 × num_layers × num_KV_heads × d_head × seq_len × batch_size × bytes_per_element

The leading 2 accounts for K and V tensors at each layer. Critically, use KV heads (not query heads) — this distinction matters enormously under GQA/MQA. A model using GQA with 8 KV heads vs. 32 query heads has 4× less KV cache memory than the naive formula would suggest. For LiLiuM-7B, the actual KV cache size depends on the KV head configuration which is not publicly documented — do not quote "30GB at 128k context" without confirming eBay's GQA configuration. A rough upper bound assuming full MHA (no GQA) would be higher; with GQA-8 it would be ~4× smaller. Always frame this: "the exact number depends on whether LiLiuM uses GQA and how many KV heads — I'd confirm the architecture spec before quoting a number."

Use GQA (Grouped Query Attention) or MQA (Multi-Query Attention) to reduce KV size by 4–8×. vLLM's INT8 KV quantization halves memory again with ~0.5% quality loss.

PagedAttention (vLLM): Divides KV cache into pages (like OS virtual memory). No pre-reservation → eliminates 60–80% memory waste. Enables KV sharing for common prefixes (system prompt caching). Prefix caching is especially valuable for eBay's RAG scenarios where the product catalog retrieval result is identical across many queries.

Speculative Decoding — Correct Math:

The mechanism: a small draft model generates K draft tokens; the target (verifier) model checks all K+1 tokens in a single forward pass. If draft token i is accepted with probability α (the acceptance rate), the expected number of tokens produced per verification pass is:

E[tokens per pass] = (1 - α^(K+1)) / (1 - α)

This follows from a truncated geometric distribution. The theoretical speedup formula is:

speedup ≈ (1 - α^(K+1)) / ((1 - α) × (1 + K × cost_draft / cost_target))

where cost_draft / cost_target is the ratio of compute cost per forward pass between draft and target models.

Concrete example for eBay (LiLiuM-1B draft → LiLiuM-7B target, K=7):

  • Assume α=0.75 (reasonable for domain-matched draft; higher for in-distribution queries, lower for OOD)
  • E[tokens per pass] = (1 - 0.75^8) / (1 - 0.75) = (1 - 0.1) / 0.25 ≈ 3.6
  • cost_draft/cost_target ≈ 1/7 (roughly proportional to parameter count)
  • speedup ≈ 3.6 / (1 + 7 × 1/7) = 3.6 / 2.0 = 1.8×

At α=0.85 and K=7: E[tokens] ≈ 5.3, speedup ≈ 5.3 / 2.0 ≈ 2.6×. This matches the empirically observed "2–3× speedup" range.

Critical constraint: Speculative decoding only improves single-sequence throughput when the target model is memory-bandwidth-bound (small batch size, ≤4 sequences). At large batch sizes (e.g., batch=64 in bulk translation), the target model is compute-bound and speculative decoding provides little benefit — you are better off with larger batches for throughput. This is the key operational insight: use speculative decoding for the real-time path (Magical Listing, agentic shopping, batch=1–4) and not for the bulk batch path (indexing translation, batch=512).

Both models must share the same tokenizer — LiLiuM's custom tokenizer is an architectural advantage here since all LiLiuM sizes use it.

Continuous Batching: Dynamically insert new requests mid-batch. Eliminates head-of-line blocking. Standard in vLLM, TGI, SGLang. Critical for mixed-priority queue (agentic Magical Listing is async; product search is real-time).

Chunked Prefill: Split long prompts into chunks interleaved with decode steps. Reduces tail latency for streaming applications (streaming listing generation to seller UI).

Disaggregated Prefill/Decode: Separate GPU pools for prefill (compute-bound) and decode (memory-bandwidth-bound). Emerging eBay-scale pattern; prefill pool can be bursty, decode pool scales more smoothly.

Quantization:

  • AWQ 4-bit: production standard for throughput-critical paths; minimal quality loss, no per-task calibration
  • GPTQ 4-bit: good alternative; needs 128-sample calibration set
  • FP8 (H100 native): increasingly common for large models; better than INT8 for weight matrices
  • KV cache quantization: INT8 KV reduces memory 2× with ~0.5% quality loss on most tasks

Custom Tokenizer Speedup: LiLiuM's 65k vocabulary (vs. LLaMA-2's 32k) encodes e-commerce terms as fewer tokens → fewer autoregressive steps → 34% generation speedup on downstream tasks. The vocabulary customization follow-up paper (arXiv 2509.26124, Versley co-author) formalizes extending existing tokenizers with domain-specific BPE merges while guaranteeing sequences are never longer than with the original vocabulary — the "monotone efficiency guarantee." See the Tokenizer Design section below for full mechanism.

Step 7B: Tokenizer Design — Versley's Co-authored Work

This is high-yield because Versley co-authored arXiv 2509.26124. Know the mechanism, not just the number.

The Problem: Vocabulary Mismatch

A general-purpose tokenizer (LLaMA-3.1: 128k vocab, but tuned for English web text) handles e-commerce terms poorly:

  • "Damenschuhe" (German, "women's shoes") → multiple subword tokens
  • "iPhone15ProMax" → fragmented into many pieces
  • Product codes like "B087J4GRCL" → character-level splits

Result: more tokens per sequence → more autoregressive steps → higher latency and cost → lower context utilization.

The Algorithm: BPE Vocabulary Extension with Monotone Efficiency

Standard BPE (Byte-Pair Encoding) starts with a character-level vocabulary and iteratively merges the most frequent adjacent pair. To extend an existing tokenizer with domain vocabulary:

1. Collect domain corpus (eBay product titles, descriptions, bilingual pairs)
2. Tokenize corpus with EXISTING tokenizer → get current segmentation
3. Count frequency of adjacent token pairs in the domain corpus
4. For each candidate new merge operation:
   a. Add merged token to vocabulary
   b. VERIFY: does this merge ever INCREASE token count for any string?
      (It cannot, by construction — merging two adjacent tokens into one
       can only reduce or maintain token count, never increase it)
5. Select top-K most frequent domain-specific merges
6. Initialize new token embeddings (random or from average of constituent tokens)
7. Continue pretraining with updated tokenizer (short warm-up phase)

Monotone efficiency guarantee (formal): For any string s, if the extended tokenizer applies a new merge operation, the resulting token count |tok_new(s)| ≤ |tok_old(s)|. This is guaranteed because new merges can only combine adjacent tokens, never split them. Corollary: the extended vocabulary never makes any sequence longer than the original vocabulary.

Result for eBay: up to 20% sequence length reduction for e-commerce text (arXiv 2509.26124). Combined with the original LiLiuM vocabulary advantage over LLaMA-2 (65k vs. 32k vocab → 34% speedup), the vocabulary-extended model further reduces latency.

Multilingual Tokenizer Fertility

A natural Versley follow-up: "How does vocabulary coverage differ across scripts?"

Token fertility = average tokens per word (lower = better vocabulary coverage):

Language Latin script (EN) Germanic (DE) Romance (FR/ES) Korean Arabic
LLaMA-2 32k ~1.3 ~1.8 ~1.5 ~3.5 ~2.8
LiLiuM 65k (estimated) ~1.2 ~1.4 ~1.3 ~2.5 ~2.2

High fertility means more tokens per word → more autoregressive steps → higher cost and worse attention span (the model "sees" fewer words in the same context window). For eBay's primary markets (DE/FR/ES/IT), LiLiuM's explicit inclusion of European language vocabulary in its 65k vocab directly addresses this. For expansion into Korean, Turkish, or Arabic, vocabulary extension via the arXiv 2509.26124 algorithm is the right intervention.

Step 7C: Alignment Pipeline — SFT → Reward → RLHF/DPO

Versley leads pretraining, but alignment is a natural adjacent topic for an ML/LLM research interview. A two-page treatment here.

Why SFT Alone Is Insufficient

Supervised Fine-Tuning (SFT) on instruction-response pairs teaches the model to produce responses in the right format. It does not teach the model to prefer better responses over worse ones — because SFT training signal is identical regardless of response quality (all targets receive equal cross-entropy loss weight).

Problem in e-commerce: a model SFT'd on "generate a product listing" might produce technically valid listings that are:

  • Misleading (exaggerated condition claims)
  • Keyword-stuffed (not readable)
  • Generic (same boilerplate for very different products)

You need a preference signal to distinguish good from bad outputs.

Reward Model Training

1. Collect prompts (e.g., "Generate a listing for this product photo")
2. Sample multiple completions per prompt from the SFT model (e.g., N=4)
3. Human raters (or LLM judges) rank completions: A > B > C > D
4. Train a reward model on preference pairs:
   loss = -log(σ(r(prompt, A) - r(prompt, B)))  # Bradley-Terry model
   where r(·) is the scalar reward output
5. Reward model predicts "how good is this response for this prompt"

eBay-specific reward signal design:

  • Seller satisfaction proxy: do sellers accept the AI output or heavily edit it? (edit rate per field)
  • Listing quality: does the generated listing lead to a sale within 30 days?
  • Compliance: does the output violate prohibited item policies?
  • Truthfulness: do generated attributes match what is visible in the product image?

PPO vs. DPO

PPO (Proximal Policy Optimization):

  • Online RL: sample from current policy, get reward from reward model, update policy
  • Requires 4 models in memory simultaneously: policy, reference policy (KL constraint), reward model, value model → high GPU memory requirement
  • Strong but complex; standard in GPT-4, Claude alignment pipelines
  • KL regularization: R(x,y) = r(x,y) - β × KL(π_θ(y|x) || π_ref(y|x)) prevents policy from drifting too far from SFT initialization

DPO (Direct Preference Optimization):

  • Offline: re-uses preference pairs directly; no reward model needed
  • Collapses reward model + RL into a single cross-entropy loss over preference pairs
  • L_DPO = -E[log σ(β log(π_θ(y_w|x)/π_ref(y_w|x)) - β log(π_θ(y_l|x)/π_ref(y_l|x)))] where y_w = preferred response, y_l = rejected response
  • Simpler to implement, no value model, lower memory footprint
  • Weakness: relies on offline preference data; cannot explore new responses during training

GRPO (Group Relative Policy Optimization — used in DeepSeek-R1):

  • Removes the value model from PPO by using group-normalized rewards within a batch
  • Each prompt: sample G responses, compute rewards, normalize within group → no critic needed
  • 2× less memory than PPO; increasingly popular for reasoning-heavy fine-tuning

eBay's likely path: SFT on curated instruction data → DPO on preference pairs collected from seller edit behavior and LLM-judge annotations → evaluate on internal listing quality benchmark. PPO reserved for agentic tasks where online exploration is needed (agentic shopping where reward = task completion).

Reward Hacking in E-commerce

Reward hacking is when the model learns to maximize the proxy reward without improving the true objective.

E-commerce specific examples:

  • Seller gaming: sellers might provide feedback that inflates listing quality for items they want to push (adversarial preference data)
  • Keyword stuffing: model learns that listings with more keywords get higher automated quality scores → produces unreadable keyword soup
  • Fake confidence: model learns that definitive attribute claims get higher ratings than hedged ones → states attributes it cannot see in the image
  • Length bias: LLM judge rewards longer responses → model learns to pad descriptions

Mitigations:

  • Use multiple independent reward signals (seller edit rate + sold rate + image-attribute consistency) — harder to simultaneously game all three
  • Ensemble reward models trained on different data splits — high variance across ensemble = reward model uncertainty → fall back to conservative response
  • Constrain output format via structured JSON schema — limits degrees of freedom for gaming
  • Monitor reward distribution during RL training: if reward increases faster than downstream quality metrics, stop training earlier (early stopping on held-out quality eval)

Step 8: Scaling Laws — Versley's Pretraining Lens

Chinchilla Compute-Optimal Training

Hoffmann et al. 2022 ("Chinchilla") established that for a fixed compute budget C (measured in FLOPs), the optimal allocation is roughly equal between model size N and training tokens D:

N_opt ∝ C^0.5
D_opt ∝ C^0.5
D_opt / N_opt ≈ 20 tokens per parameter

This means a 7B parameter model should be trained on ~140B tokens for compute-optimality. LiLiuM-7B trained on 3T tokens is trained on ~21× the Chinchilla-optimal token count.

Inference-Optimal Training: Why LiLiuM Is "Overdense"

Chinchilla optimizes for one-shot training cost. When you account for inference cost (serving N requests), the optimal recipe shifts toward smaller models trained longer:

Total cost = training_cost + inference_cost × num_requests
For large num_requests: minimize model size while maintaining quality
→ Train smaller model on more tokens (inference-efficient)

This is why Llama-3-8B was trained on ~15T tokens (D/N ≈ 1875) and why LiLiuM-7B on 3T tokens (D/N ≈ 428) makes sense for eBay: the 7B model is cheap to serve, and training it longer improves quality without increasing inference cost. The 1B model at D/N ≈ 3000 is even more extreme — it is inference-optimal for eBay's highest-throughput use cases.

Connection to Tony's routing work: Overdense training of small models + intelligent routing is exactly the right strategy for eBay. The 1B model at 3T tokens may match the quality of a 7B model at 140B tokens on domain-specific tasks — and at 4× lower inference cost.

Reading Loss Curves

Pretraining loss curves tell you about data quality and model capacity:

  • Smooth monotonic decrease: training is healthy; data is diverse and well-mixed
  • Loss spike followed by recovery: numerical instability (LR too high, bad batch); common trigger for gradient norm explosion → fix with gradient clipping
  • Plateau after early steep drop: model has saturated capacity of the data — either add more data diversity or scale model
  • Loss oscillation (high variance): batch size too small; increase batch or accumulate gradients
  • Sudden loss increase: data pipeline bug introduced corrupted batches; investigate immediately

Evaluation during pretraining: eBay monitors both training loss and a set of held-out domain benchmarks (e-commerce NLU tasks) at regular checkpoints. If benchmark scores plateau while training loss continues to improve, the model is memorizing rather than generalizing.

MFU as a Training Efficiency Metric

Model FLOP Utilization (MFU) = actual tokens processed per second / theoretical peak tokens per second:

MFU = (tokens/s × FLOPs/token) / (num_GPUs × peak_FLOPs/GPU)

For a well-tuned H100 pretraining run with FlashAttention-2 and tensor parallelism: 40–47% MFU is typical. Below 30% indicates pipeline inefficiency (data loading bottleneck, poor TP placement, excessive PP bubble). Above 50% is excellent.

Why it matters: MFU is how Versley's team knows whether their 480-H100 cluster is being used efficiently. A 10% MFU improvement = 10% more model capacity or 10% less training cost for the same quality.

Step 10: Feedback Loops and A/B Testing

Feedback Loop Design

User interaction → behavioral log → feature store update → model retrain trigger → new model → A/B test → gradual rollout

Frequency: Daily incremental training for fast-moving signals (CTR); weekly full retraining for deep features; quarterly pretraining/domain adaptation for LLMs.

Feedback lag: Purchase signal has 3–7 day lag. Use surrogate metric (click-to-cart) for faster iteration; validate correlation with purchase on a subset.

Selection bias in feedback: Only shown items can be clicked. Mitigate via:

  • Inverse Propensity Score (IPS) weighting in training
  • Bandit exploration (epsilon-greedy or Thompson sampling on a fraction of traffic)
  • Counterfactual evaluation using propensity models

IPS in depth: The propensity p(shown | item, position) must itself be estimated — it is not a number you simply look up. In practice, eBay logs the score and rank from the existing production ranker for each displayed item. This logging policy defines the propensity. If the logging policy changes, historical propensities become stale. More robust: use doubly robust (DR) estimation, which combines a direct reward model with IPS:

DR_estimator(item) = direct_model(item) + (observed_reward - direct_model(item)) / propensity(item)

The DR estimator is unbiased if either the direct model or the propensity estimate is correct — more robust to misspecification in either alone. Exploration policy (e.g., epsilon-greedy at 5% of traffic) ensures some propensities are uniform and well-estimated, breaking the dependence on the logging policy.

A/B Testing

Setup:

  • Randomize at user level (not session level) to avoid spillover
  • Pre-compute required sample size: n = 2σ²(z_α + z_β)² / δ² where δ = minimum detectable effect, σ = metric standard deviation
  • Run for minimum 2 full weeks to capture weekly seasonality
  • Report lift, confidence interval, and p-value (prefer Bayesian A/B if speed of decision is critical)

eBay-specific considerations:

  • Buyer-seller marketplace: intervention on buyer side can affect seller supply; test both
  • Cross-border: run separate analysis by market (DE, FR, ES, IT, US) — effects differ by country
  • New model rollout: start at 1% → 10% → 50% → 100% (canary pattern)

Key Model Choices and Tradeoffs

Why two-stage (L1 + L2) instead of one-stage cross-encoder over all candidates? A cross-encoder on 2B items at 50ms each = 100B ms = physically impossible. Two-stage: L1 is fast (GBDT or lightweight deep model) to narrow to 100; L2 applies expensive cross-attention only to manageable set. This is standard LTR practice, not just eBay's — same pattern at Google, Amazon, Airbnb.

Why BM25 + dense hybrid instead of dense-only? Dense captures semantic similarity but struggles with exact product codes, brand names, and model numbers (e.g., "Nike Air Max 90 size 10 Obsidian"). BM25 excels at exact token overlap. Hybrid consistently outperforms either alone on e-commerce benchmarks.

Why eBERT / LiLiuM encoder instead of generic BERT? Domain vocabulary — product titles use jargon, abbreviations, and specialized terms that general BERT often misrepresents. LiLiuM's 65k custom vocabulary encodes e-commerce terms as single tokens rather than multi-subword sequences, improving embedding quality and reducing encoding latency.

Router Design (Deep Dive — Tony's Core Work)

Complexity Estimation and the Labeling Problem

The router must decide in <5ms which tier to use. But before discussing the classifier, address the labeling problem honestly:

How do you get router training labels?

The naive approach is: label a query "hard" if the small model produced a low-quality output on it and "easy" if the small model matched the large model. But this is circular — you need to run the large model on all queries to generate labels, which defeats the cost-saving purpose.

Practical solutions:

  1. Offline label collection with amortized cost: run a one-time sweep of both models on a representative sample (e.g., 100k queries drawn from the full distribution). Use large-model output as reference. This is a one-time cost that pays off over months of inference savings.
  2. Continuous refinement with logging: in production, route according to the current policy, but log a small random sample (1%) to both small and large model and compare outputs. Use these comparisons as new training labels. The router improves incrementally without re-running the large model on everything.
  3. Synthetic difficulty features: use proxy features that correlate with difficulty without requiring large model inference: query length, syntactic complexity (parse tree depth), vocabulary rarity score, feature tag (agentic tasks are harder than structured attribute extraction). These features are imperfect but give the router a head start.
  4. MERA-style skill clustering (Tony's work): cluster queries into skill domains (e.g., "fashion attribute extraction," "electronic specification parsing," "multilingual title translation") using unsupervised clustering on query embeddings. For each skill cluster, measure small-model vs. large-model quality on a labeled subsample. Route based on cluster membership × per-cluster quality thresholds. This is more interpretable than a black-box classifier and easier to update when new skill clusters emerge.

Router calibration: Track expected vs. actual small-model success rate. If the router predicts 85% success rate for "easy" queries but actual quality check (edit rate, verifier score) shows only 70%, the router is over-confident. Calibrate using Platt scaling or isotonic regression on held-out data.

Distribution Shift: When the Router Goes Stale

The router is trained on historical query distributions. eBay's distribution shifts seasonally (holiday products, new product categories, new language pairs):

  • New product category launch: queries for new electronics models have no router training examples → model uncertainty → default to conservative routing (send to larger model)
  • Seasonal events: queries surge for seasonal items (Halloween costumes, Christmas gifts) → difficulty distribution may shift because domain-specific vocabulary coverage differs
  • Monitoring signal: track the router's "hard tier" fraction by feature tag. If the fraction of queries routed to the large model spikes unexpectedly, it means the router is seeing OOD queries and escalating them correctly — but it also signals a retraining opportunity.
  • Trigger-based retraining: PSI on query embedding distribution. If PSI > 0.2 vs. router's training distribution → trigger router retraining within 48 hours using the continuous logging data.

Cascade Routing: Honest Cost Model

The cascade approach is the most production-ready, but it has a critical cost that the naive model ignores: double inference on misclassified queries.

True cost model (cascade):
E[cost] = P(easy_correct) × cost(1B) 
        + P(easy_misclassified → escalated) × (cost(1B) + cost(7B))   # double inference
        + P(medium_correct) × (cost(1B) + cost(7B))                    # cascade down from cascade start
        + P(hard) × (cost(1B) + cost(7B) + cost(13B))

If 5% of "easy" queries are misclassified and need escalation (the "escape rate"), and 10k QPS × 5% = 500 escalated queries/sec, each paying cost(1B) + cost(7B) instead of just cost(7B), the wasted tokens are significant. At eBay scale:

  • 500 escalated queries/s × (1B call ≈ 0.5M tokens/s wasted)
  • Over 24 hours: ~43B extra tokens from escape rate alone

Tony's tracking metric: "I'd define the 'escape rate' as the fraction of small-model calls that fail the quality verifier and escalate. I'd track this per feature tag and per query cluster, and tune the quality verifier threshold to balance cost savings against escape rate. A 1% escape rate increase from 5% to 6% is acceptable; a jump to 15% signals either router degradation or distribution shift."

Critical distinction (Versley will know this precisely):

Three-Stage Training Recipe (with the Constraint Versley Will Ask About)

Stage 1: Vision-Language Alignment
  ├─ Freeze language model; train only vision projector (MLP bridge)
  ├─ Data: BLIP-LAION 558k general image-caption pairs
  ├─ Goal: map visual tokens into language model's embedding space
  └─ Duration: short (vision projector has <<1% of total parameters)

Stage 2: Mid-Stage Training
  ├─ Unfreeze both vision encoder and language model
  ├─ Data: filtered mixture of LLaVA-OneVision mid-stage data (general VQA, etc.)
  ├─ Goal: develop multi-modal reasoning; avoid over-fitting to e-commerce format
  └─ Data selection: redundant subsets removed to avoid degenerate solutions

Stage 3: Visual Instruction Tuning
  ├─ Full model fine-tuning
  ├─ Data: ~4M internal e-commerce instructions
  │   ├─ VQA: 45% (free-form, yes/no, image-only questions)
  │   ├─ Dynamic Attribute Extraction: 30%
  │   ├─ Precise Instruction Following: 12.5%
  │   └─ Product Listings: 12.5%
  └─ Multi-image items: 100k items sampled, median 5 images/item

The key architectural question (Versley will ask): How does multi-image work if training used single images?

The answer is mechanistic, not hand-waving:

  1. Visual token representation: SigLIP2 encodes each image independently into a fixed-length sequence of visual tokens (e.g., 256 tokens per image for a 336×336 image at ViT-L patch size 14).
  2. Multi-image concatenation: for a 5-image item, the visual tokens are simply concatenated: [img1_tokens | img2_tokens | img3_tokens | img4_tokens | img5_tokens | text_tokens]. This is appended before the text tokens in the input sequence.
  3. Positional encodings: within each image, 2D positional encodings are applied by SigLIP2 before projection. Across images, there is no explicit cross-image positional encoding — the sequence position in the transformer handles ordering. This means the model has no explicit mechanism to reason about "this is image 3 of 5"; it just sees a longer visual token sequence.
  4. Context length pressure: 5 images × 256 tokens/image = 1280 visual tokens, plus the text prompt. For a context window of 8192 tokens, this is 15-20% of the window on visual tokens alone. At >5 images, you risk exceeding context limits or degrading attention quality due to token dilution.
  5. Generalization mechanism: the model trained on single images learns to extract attributes from visual context. When presented with multiple images of the same item (different angles, different lighting), the cross-attention mechanism naturally aggregates information across the concatenated sequence — this is not new capability, it is the same visual reasoning extended to a longer input.
  6. Observed quality: the eBay VLM paper shows the model generalizes to multi-image input without multi-image training, but with some caveats: the model may fixate on the first image in the sequence (ordering matters) and may duplicate attributes seen in multiple images rather than synthesizing a single consolidated description.

Production implication: For Magical Listing, eBay likely applies a simple greedy strategy: select the best 2–3 images from the seller's upload (highest quality, least blur, most distinct angles) rather than feeding all 20 photos a seller might provide.

RAG for MT — The Key Insight with Honest Tradeoffs

Standard MT treats each title independently. E-commerce MT benefits enormously from retrieval:

Title to translate: "Nike Air Max 90 Retro OG White/Black Men Size 10"
BM25 retrieval returns:
  - "Nike Air Max 90 Retro OG Blanc/Noir Homme Taille 44" [EN→FR existing]
  - "Nike Air Max 95 OG White/Black Men Size 11 → Weiß/Schwarz Herren Größe 44" [EN→DE]
These few-shot examples teach:
  - "White" → "Blanc" (not "Blanche")
  - "Men Size 10" → "Homme Taille 44" (shoe size conversion + gender term)
  - Brand names stay untranslated
  - "OG" stays as "OG" (not localized)

Why BM25 over dense retrieval for MT examples? For retrieval-augmented MT, you want Translation Memory (TM) behavior: near-exact surface matches for consistent terminology. Dense retrieval finds semantically similar items, which for MT can give you the wrong translation variant — "Red Evening Gown" and "Burgundy Cocktail Dress" are semantically similar but use different color terms. BM25 with lexical overlap finds "Red Evening Gown → Robe de soirée rouge" which is exactly the example you need for "Red Maxi Dress."

When BM25 fails (honest tradeoff): For a brand-new product category or a unique brand name with zero lexical overlap in the index, BM25 returns nothing. Dense retrieval degrades gracefully in this case by finding semantically similar items (even if different brand). Recommendation: hybrid retrieval with BM25 primary + dense fallback, where the few-shot examples are annotated with retrieval confidence — if BM25 confidence is low (few exact matches), fall back to zero-shot MT rather than using low-quality dense examples that might mislead the model.

Custom Tokenizer for MT (Full Mechanism)

LiLiuM's 65k vocabulary benefits MT via:

  • "Damenschuhe" (German, "women's shoes") = single token, not "Damen" + "##schuhe"
  • Fewer decode steps → faster MT → lower cost at billions-of-requests scale
  • Better attribute-level fidelity: if "Damenschuhe" is a single token, the model learns clean alignment to "Women's Shoes"; fragmented subwords create alignment ambiguity

The vocabulary customization paper (arXiv 2509.26124) extends this further: BPE merges specific to e-commerce terminology in all target languages, with the monotone efficiency guarantee ensuring no string becomes longer than before the extension.

Multilingual Tokenizer Fertility Across eBay Markets

For expansion to Korean or Turkish, the existing 65k vocab (even with European language emphasis) shows higher fertility for those scripts. The vocabulary extension algorithm (arXiv 2509.26124) is the right intervention: collect a sample of Korean product listings, run the extension algorithm, add the most frequent Korean n-gram merges. This reduces Korean token fertility and improves both quality and serving cost for any Korean market expansion.

Evaluation

chrF explained: Character n-gram F-score between hypothesis and reference. Better than BLEU for short texts:

  • Character n-grams are denser → non-zero recall even for short titles
  • Naturally handles morphological variation in DE/FR/ES (inflected forms)
  • Correlates better with human judgments for product titles

COMET: Neural MT metric (trained on human judgment data); better than chrF for overall quality of longer descriptions but requires GPU to compute. Use chrF for fast iteration, COMET for final evaluation.

Tony's Insight

"The fundamental challenge in e-commerce MT is that product titles are not natural language sentences — they're sequences of keywords with implicit structure. Standard MT models try to translate grammatically, which breaks down for keyword sequences. The RAG approach is elegant: instead of teaching the model all e-commerce terminology, you show it similar titles that have already been translated correctly. It's in-context learning applied to a specialized domain, with retrieval doing the heavy lifting."

2. Feedback Loops and Exposure Bias

Q: Your recommendation model is trained on clicks, but clicks are biased by what's already shown. How do you handle this?

The problem: Feedback loop: model shows item A → A gets clicked → model learns A is good → shows A more → A gets more clicks → ... Meanwhile item B, equally good but never shown, has zero training signal.

IPS Weighting (with the real engineering detail):

# p_shown must be ESTIMATED from the logging policy, not looked up
# eBay logs the score and rank from the current production ranker for each displayed item
# propensity = P(item shown at position k | query, user)

# Doubly Robust estimator (more robust than IPS alone):
def dr_loss(direct_model_pred, observed_reward, propensity):
    # If propensity model is correct: unbiased even if direct_model is wrong
    # If direct_model is correct: low variance even if propensity is wrong
    return direct_model_pred + (observed_reward - direct_model_pred) / propensity

# Exploration policy ensures some propensities are uniform (epsilon-greedy):
# 5% of impressions are randomly selected → propensity = 1/catalog_size (known exactly)
# These provide clean, unbiased training signal

Exploration budget:

  • Reserve 5–10% of impressions for exploration (Thompson sampling or epsilon-greedy)
  • Thompson sampling: sample from posterior over item quality → naturally balances explore/exploit
  • UCB: score = estimated_CTR + exploration_bonus(1/sqrt(impressions))

Diversity enforcement:

  • MMR: penalize items too similar to already-selected items in output set
  • Category cap: maximum 3 items from same subcategory in homepage feed
  • Gini coefficient monitoring: if Gini of impressions across items > 0.7, diversity alarm

5. Cost Ceilings

Q: GenAI feature is over budget. What do you cut?

Cost reduction levers in priority order (least quality impact first):

  1. Semantic cache: 0 quality impact for hit; 20–40% cost reduction for repetitive queries
  2. Prefix caching (vLLM): 0 quality impact; 30–50% KV cache savings for common system prompts
  3. Quantization (AWQ 4-bit): <0.5% quality impact; 2× throughput
  4. Speculative decoding (real-time path only): 0 quality impact; 1.8–2.6× throughput (memory-bandwidth-bound single-sequence; not effective for large batches)
  5. Vocabulary customization: 0 quality impact; 20–34% fewer decode steps
  6. Routing: small model for easy queries: ~5% quality impact on edge cases; 50–90% cost reduction
  7. Batch async processing: larger batch sizes → better GPU utilization; 2–4× throughput
  8. LoRA specialist: train small model on strong-model trajectories; 70–90% cost reduction post-training
  9. Reduce generation length: cap max tokens; structured JSON output vs. free text
  10. Off-peak scheduling: batch non-critical tasks to off-peak hours; spot instances

Example cost reduction waterfall for Magical Listing:

  • Baseline: LiLiuM-13B BF16, no cache → $0.005/listing
  • After AWQ 4-bit: $0.0025/listing (2× throughput)
