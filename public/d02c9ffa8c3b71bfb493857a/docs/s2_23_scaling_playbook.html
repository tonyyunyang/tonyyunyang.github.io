The Scaling Playbook

Your reusable answer shape for any "now 10× it / how does this scale?" question. Round 1 already threw one at you, and eBay grades systems thinking. Read it until the six axes are muscle memory, then drill the worked walk out loud.

The one move that wins the question

When an interviewer says "now 10× it" or "how does this scale?", they are not asking for scaling laws (Chinchilla/MFU/inference-optimal live in the ML System Design Framework — reach for those only if the question is about training compute-optimality). They are testing whether you reason about a system under load like an operator. So don't free-associate. Walk six axes, in order, out loud — and for every axis land the same two-part punch:

The meta-move — end each axis on: "here's what breaks first, and here's the cheapest lever that fixes it."

That phrase is the whole grade. It shows you know where a system actually fails (not where it's flashy) and that you reach for the low-quality-cost lever before the expensive one. And because the cheapest lever is almost always efficiency — cache, quantize, route, shard — every scaling answer flows straight back to your thesis: the systems race, not the scaling race.

The six axes, in order:

DATA → MODEL → TRAINING → SERVING → COST → ORG

Mnemonic: you build a model from data, you train it, you serve it, you pay the cost, and someone runs it at 3am. Same order every time.

Describe the machinery in the abstract and pivot to efficiency — the exact wording lives in The Honest MLOps-Gap Script. Say the playbook, own the one clause, move on.

The worked walk to drill — scale routing itself (your edge)

This is the one to steer probes toward. It maps directly onto your project (cross-ref Project Walkthrough: Cost-Efficient LLM Routing) and eBay's real model zoo (LiLiuM 1B/7B/13B, e-Llama 8B/70B, frontier fallback).

  • DATA — routing decisions + per-skill outcomes. Breaks first: the traffic distribution drifts (new categories, seasonal spikes, new listing flows), so yesterday's routing policy silently misroutes; cheapest lever: log the decision mix and retrain on a cadence, don't assume stationarity.
  • MODEL — the router must be tiny relative to the models it routes to, or it eats its own savings. Breaks first: a heavy router adds latency to every request; cheapest lever: a small classifier/encoder kept sub-10ms.
  • TRAINING — learn the (query → best-model) policy per skill. Cheapest lever: the benchmark-first approach (TwinRouterBench) so every retrain is measured, not guessed — 53% (trained router vs. unrouted Opus, matched resolve, dynamic track).
  • SERVING — the router sits on the hot path in front of the fleet. Breaks first: the router itself becomes a throughput bottleneck; cheapest lever: make it horizontally shardable — partition by request type/entity so it scales out linearly.
  • COST — the whole point. MERA: 87.3% router-decision accuracy at 51.8% serving cost; the router-first design gets >90% cost reduction at iso-quality on favorable traffic. Cheapest lever: route the routine listing-title request to LiLiuM-1B, the cross-border translation to e-Llama, and only genuinely hard reasoning to 70B/frontier.
  • ORG — operate the router like a model. Breaks first: drift on the decision distribution — the router's output mix shifts before any accuracy label lands; cheapest lever: an SLA-aware knob (operator sets "hold quality at X" directly) + drift monitoring on the decision mix, not just model accuracy.

Say it like this (the pocket version): "At one QPS the interesting number is accuracy; at eBay QPS the interesting number is cost-per-quality under drift — with a router cheap and shardable enough that it doesn't spend the savings it creates."

Where to plant scaling (so it doesn't sound bolted on)

Scaling should feel native to how you think, not like a checklist you produce on demand. Plant it in both rounds.

In a system-design answer (Round 2)

  • Open by naming scale — the moment you clarify requirements, say the magnitudes ("2B listings, ~100M searches/day, p99 <500ms, 7 markets"). That frames every later choice as scale-driven.
  • Use scale as the reason for a decision, not an afterthought. "I'd shard the embedding tables because at 2B IDs they're the memory hog, not the MLP" is far stronger than tacking "and it scales" onto the end.
  • When they say "now 10× it," run the six axes explicitly — narrate the walk, land breaks-first/cheapest-lever each time, and close on the cost axis (your edge). Full worked designs are in Flagship: Search Ranking & Recsys and System Design Skeletons Bank.

In the project walkthrough (Round 3)

Plant scaling at these anchors so it reads as a spine, not an add-on (full script in Project Walkthrough: Cost-Efficient LLM Routing):

  1. In the hook — end the first sentence with "…and does that survive scaling up?" so the whole project is framed as a scaling question from word one.
  2. In a key decision — justify learned-router-over-cascade by scale: "cascades' escalation tax dominates at volume." Scale as the reason for a fork is the strongest form.
  3. In results — translate the % into eBay-scale money and GPU/carbon footprint; the number only bites at scale.
  4. In the retrospective/scale beat — the explicit trio: router sub-10ms + horizontally shardable; drift on the decision mix + retraining cadence; an SLA-aware knob. Map targets onto the real zoo (LiLiuM 1B/7B/13B, e-Llama 8B/70B, frontier).
  5. In probe redirects — when pushed on an ops area you're thin on, answer honestly then pivot to a scaling strength (batching, shardable routing, cost-per-quality as the north star).

⚠ [CONFIRM TONY: in the walkthrough, keep the eBay-scale dollar/GPU translation directional ("at real serving volume, halving cost on routable traffic is a seven-figure line item") — do not assert a specific eBay infra number you can't defend.]

Pocket lines (say these under pressure)

  • The shape: "DATA → MODEL → TRAINING → SERVING → COST → ORG — and for each, what breaks first and the cheapest lever."
  • The meta-move: "Always end on what breaks first and the cheapest lever that fixes it."
  • The recsys trap: "The hog is the embedding table, not the MLP — so I shard before I add layers."
  • The serving trap: "The p99 tail breaks before the average, so I split real-time from batch."
  • The cost ladder: "Least-quality-hit first: cache, quantize, then route small-first."
  • The routing one-liner: "At eBay QPS the number that matters is cost-per-quality under drift, with a router cheap enough not to spend its own savings."
  • The thesis anchor: "Every scaling question ends at efficiency — that's the systems race, not the scaling race."
