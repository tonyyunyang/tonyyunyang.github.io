Flagship: Search Ranking & Recs

The one worked design to have cold. Read this the night before the ML System Design round — it's eBay's bread-and-butter and Tony's flagged weak spot, so it must be airtight. One end-to-end story, deep exactly where the general playbook is thin.

How to use this. Narrate the 60-second spine (§13) first, then expand any node when probed. Everything here is ONE design: "Design product search ranking / personalized recommendations at eBay scale (~2B listings)." Search is the spine; recommendations are the same machine with the user as the query — I promote that fork to its own section (§4A). For the 14-step framework and the latency-budget ASCII, see Design Framework & How to Drive; for the five other worked designs and the LLM serving/routing gateway, see the Design Skeletons Bank. This doc does not repeat those.

1. Frame it in one breath

Say it like this: "Retrieve-then-rank over 2B listings. I'll spend my time on the three things that actually decide quality: how I train the two-tower retriever — because sampling bias is the whole game — how I make the ranker multi-objective and calibrated, and how I prove a win offline → interleaving → A/B before it touches GMV. Everything else is plumbing I'll wire up but not dwell on."

That opener does two things: it shows I know where the hard parts are (not the architecture diagram, which everyone draws), and it sets up the deep dives so I control the conversation.

2. Requirements & metrics — name the bundle, not one number

eBay defaults (state, don't re-derive): ~2B active listings, ~135M MAU, ~100M searches/day, p50 200ms / p99 <500ms, 7 language markets (EN/DE/FR/ES/IT/NL/PL), DSA/GDPR constraints.

North-star + guardrails, said together. The single most common junior mistake is optimizing CTR. CTR rewards clickbait. State the bundle out loud:

Layer Metric Why
North-star GMV per search session The business metric; resists clickbait
Guardrails purchase-rate, latency p99, null/low-recall query rate catch the ways GMV can be gamed
Fairness guardrail seller-impression Gini (no concentration increase) marketplace health; DSA exposure
Offline proxy NDCG@10 on graded relevance (also MRR, recall@100) cheap, fast — but only a proxy

The one-liner that shows judgment: "The offline metric is a proxy I periodically re-validate against the online north-star. Proxy/target divergence — the model climbs NDCG while GMV goes flat — is the classic failure, so I never treat NDCG as the goal, only as a fast gate."

Two label regimes, not one (this is the whole hinge of §3):

  • Graded editorial relevance — small, clean, expensive. Used for offline NDCG and as the final arbiter.
  • Behavioral engagement (clicks/carts/purchases) — huge, biased. Used for training.

Say it like this: "Clicks train the model; graded relevance judges it. I never let the biased signal be the final arbiter."

3. Labels & the click-bias problem (the part everyone hand-waves)

Behavioral logs are the only signal at scale, and they're biased three ways. Name all three and a fix for each — this is where you separate yourself from a candidate who just says "we use clicks."

Bias Mechanism Fix I'd use
Position bias Top slots get clicked regardless of true relevance IPS-weight each click by 1/propensity(position); estimate the propensity via periodic randomization (RandPair / RandTop-k on a small traffic slice) or a position-bias model. Modern refinements (doc-similarity-enhanced IPS, control-function estimators) give real but modest gains (~1.3%).
Selection bias Only shown items can be clicked; the unshown are unlabeled, not negative Counterfactual / off-policy estimation; treat unshown as missing, not negative; run an exploration budget so some impressions have known uniform propensity.
Trust / presentation bias Rich snippets, images, badges inflate clicks independent of relevance Model it jointly with position (mixture-based correction), or accept it as noise and lean on editorial labels for the final eval.

The propensity is estimated, not looked up. p(shown | item, position) comes from the logging policy — eBay logs the score and rank the production ranker assigned each displayed item. If the logging policy changes, historical propensities go stale. More robust than raw IPS is the doubly-robust (DR) estimator, which is unbiased if either the propensity model or a direct reward model is correct:

DR(item) = direct_model(item) + (observed_reward − direct_model(item)) / propensity(item)

The 5% exploration slice (§10) is what makes some propensities uniform and clean, breaking the dependence on the logging policy.

The 2026 twist worth dropping (signals I'm current): teams are testing LLM-generated relevance annotations as a partial replacement for click labels — cheaper than human editors, less biased than clicks — but you must validate the judge against a human gold set first (position/verbosity/self-enhancement bias; gate any judge below ~80% agreement). One sentence, big signal.

4. Candidate generation — two-tower training done right

This is where retrieval quality actually lives. "Two-tower, in-batch negatives" is one line in most answers; that one line is the whole ballgame.

Architecture (fork point).

  • Query tower encodes [query text, query context]; item tower encodes [title, category, attrs, image embedding]. Both → L2-normalized d-dim vectors; score = cosine. Item embeddings are pre-computed and ANN-indexed.
  • Search fork: query tower = the text-query encoder.
  • Recs fork: "query" tower = a user tower over the behavior sequence — last-N item IDs through a GRU for the long horizon plus a small Transformer over the recent ~20 interactions. Everything below is identical.

The training problem is negative sampling, and it has a specific trap:

  1. In-batch negatives (reuse the other items in the batch as negatives) is the efficient default — but it biases toward popular items, because popular items show up as negatives more often and get pushed down disproportionately.
  2. logQ correction is the fix: subtract log Q(item) (the item's sampling / appearance probability) from its logit before the softmax. This de-biases the in-batch approximation of the full softmax. Naming this unprompted is a strong signal — it's the industry-standard trick. (If pushed: RecSys 2025's "Correcting the LogQ Correction" shows even the correction has known failure modes.) Q is estimated, not known — for an established item it's just its historical appearance frequency, but for a streaming / cold item you have almost no count, so I'd keep a running frequency (a count-min sketch over the impression stream) with a smoothing prior / floor so a brand-new item doesn't get a near-zero Q and an exploding correction. That estimate goes stale when the logging policy shifts (the appearance distribution moves), which is another reason to tie it to the 5% exploration slice (§10) where propensities are clean and uniform.
  3. Hard negatives on top of in-batch: mine items that are close but wrong — same category, semantically near, but never co-engaged. Without them the retriever only learns to separate obvious junk and collapses on fine distinctions. Mix easy (in-batch) + hard (mined) at a tuned ratio; too many hard negatives destabilizes training.
  4. Temperature τ on the contrastive loss controls how peaked the distribution is — tune it, it materially moves recall@k.

Say it like this: "The failure mode of a lazily-trained two-tower is that it retrieves popular, generically-relevant items and misses the long tail — which is exactly eBay's inventory. logQ correction plus hard-negative mining is how you fix that, and I'd watch recall@100 on tail queries specifically, not just aggregate recall — because aggregate recall stays high while the tail quietly dies."

Retrieval is multi-channel, and fusion is itself a learning problem. eBay won't have one retriever — it'll have dense ANN + BM25 (exact product codes, brands, UPCs) + item-to-item co-engagement + maybe a trending channel. Say RRF (Reciprocal Rank Fusion) as the v1 baseline, then: "I'd learn the channel weights per query intent — treat multi-channel fusion as a query-dependent learning-to-rank problem over heterogeneous sources rather than a fixed RRF weight" as v2. That progression shows you know both the pragmatic start and the ceiling.

⚠ [CONFIRM TONY]: If asked "which channel dominates at eBay," don't assert a split — eBay's exact retrieval mix isn't public. Say "I'd measure per-channel recall contribution and tune from there."

4A. The feed fork — recommendations as a first-class surface

The recs fork is not "search with the user as query and everything else identical." Four things genuinely change, and volunteering them shows I've actually built a feed, not just relabeled a search box.

  1. Candidate generation is query-less. There's no text query to anchor recall, so I generate from three sources and fuse: the user tower (behavior-sequence embedding → ANN over the item index), item-to-item co-engagement ("users who bought/viewed X also engaged Y" — cheap, high-precision, and the workhorse for the "More like this" rail), and a trending / fresh channel so a cold-start user or an exhausted session still has supply. Same RRF-then-learned-fusion progression as §4, but the mix skews toward co-engagement and trending because there's no query intent to lean on.

  2. Whole-page / slate ranking is a first-class objective, not a post-hoc guard. In search, MMR-style diversity is mostly a collapse-guard on top of a pointwise sort. In a feed the slate is the product: the value of item 4 depends on items 1–3 already shown (redundancy, category balance, seller spread), so I optimize page-level expected value — a listwise / slate objective (e.g. a slate re-ranker or a generative list model) — rather than sorting independent pCVR scores and hoping MMR patches it. Diversity moves from a penalty term to part of the objective.

  3. User-embedding serving freshness is the real serving constraint. You cannot run a full user-Transformer over each user's behavior sequence at request time, across ~135M MAU — the tail latency and cost are untenable. (Say "135M monthly active users," not "135M-parameter model" — a request-time session Transformer over the last ~20 events does run in production; the untenable thing is a full-history forward per request at that user count.) So I split the user embedding: a long-horizon component precomputed on a cadence (batch, hours-old, captures durable taste) plus a recent-interaction component refreshed near-real-time (the last ~N events streamed in, so a click 30 seconds ago actually moves the next feed). Request time is then a cheap combine (concat / small head) over two mostly-precomputed vectors, not a full Transformer forward. This is the recs analogue of the feature-freshness tiers in §6.

  4. Position effects and feed staleness are structural. Infinite scroll means position bias is severe and depth-dependent (engagement decays down the scroll, so IPS propensities must be modeled per feed-depth, not just per top-k slot), and a feed that doesn't refresh its candidate pool re-shows the same items and goes stale within a session — I'd cap per-session repeats and rotate the trending/fresh channel so scrolling keeps surfacing new supply.

Say it like this: "The feed is the same retrieve-then-rank machine, but three things flip: candidate generation is query-less so I lean on the user tower plus item-to-item co-engagement plus trending; ranking becomes a slate problem where diversity is in the objective, not an MMR patch; and I can't run a full user-Transformer over every user's behavior sequence at request time across ~135M monthly actives, so the user embedding is a slow precomputed long-horizon vector plus a near-real-time recent-interaction vector combined cheaply per request."

5. Ranking — multi-task + calibration

Why multi-task, concretely. The business cares about a chain: click → add-to-cart → purchase → (no return). Each step is sparser and closer to what you actually want. Single-objective on clicks over-serves clickbait. So the ranker predicts several heads jointly.

Architecture ladder — say them in order, pick one, justify the pick:

Model What it is When it breaks / wins
Shared-bottom one trunk, per-task heads simplest; suffers negative transfer when tasks conflict (click vs. purchase want different things)
MMoE shared expert pool, per-task gating network picks its mix reduces negative transfer; the standard production choice
PLE (Progressive Layered Extraction) explicitly separates shared vs task-specific experts in stacked layers beats MMoE when tasks are heterogeneous (click is dense/easy, CVR is sparse/hard); current large-e-commerce default

Naming PLE and why — task-specific experts stop the shared experts from being dominated by the easy, dense click task — is a strong signal. Under the trunk, feature interactions still matter: DCN-V2 (explicit bounded-degree crosses) is the workhorse; DeepFM as a baseline.

Blend to a single score:

score = w1·pClick + w2·pCart + w3·pPurchase + w4·pRelevance − w5·pReturnRisk

How the weights actually get set (don't hand-wave "tuned to GMV" — that's the junior answer). It's a three-stage funnel matching §7: (1) offline search — coordinate descent or Bayesian optimization over the weight vector against a business-proxy objective on replayed logs (expected GMV / session under the new order), because you can't A/B every candidate weight set; (2) the shortlist that survives goes to interleaving for a fast, sensitive ranking-preference read; (3) survivors graduate to an A/B on realized GMV. Crucially I frame this as constrained optimization, not free maximization: guardrails like seller-impression Gini and return-rate are hard constraints, so the search is "maximize the GMV proxy subject to Gini ≤ threshold and return-rate ≤ threshold" — a Lagrangian with the guardrails as penalty/constraint terms, not soft nice-to-haves. And I'd flag the multiple-comparisons trap: sweeping many weight vectors on a single 2-week A/B window will surface a "winner" by chance, so I fix the candidate set before the test, hold out a final confirmation split (or correct the significance threshold), and treat the offline/interleaving stages as the place to prune — not the A/B. Fork: for ads this becomes an eCPM auction score (bid × pCTR), and calibration stops being optional (see below).

Calibration is load-bearing — and the general playbook barely mentions it. The moment you blend heads or feed a score into an auction / business rule / threshold, the probabilities must mean what they say: a predicted 0.1 pCVR must convert ~10% of the time. Rankers trained on imbalanced, biased data are systematically mis-calibrated.

  • Measure: Expected Calibration Error (ECE), plus reliability diagrams per segment (new vs. established items, per market — calibration that's fine in aggregate can be badly off for cold items).
  • Fix: Platt scaling / isotonic regression / temperature scaling on a held-out set; recalibrate on a schedule, because calibration drifts even when ranking AUC doesn't.
  • Calibrated heads do not compose into a calibrated blend. Even if pClick, pCart, and pPurchase are each individually well-calibrated, the weighted blend w1·pClick + … − w5·pReturnRisk is not automatically a calibrated estimate of anything — the weights, head correlations, and the minus term destroy the per-head calibration. So I calibrate at two levels: the individual heads, and a final calibration map fit on the blended score against the realized downstream outcome (e.g. realized purchase / GMV), then I monitor ECE on the served quantity itself — the number the auction / threshold actually consumes — not just on the component heads.

Say it like this: "AUC tells you the order is right; calibration tells you the numbers are right. For a pure sort you only need order. The instant a downstream consumer — an auction, a budget pacer, a pCVR × price blend, a threshold — reads the actual value, you need calibration, and I monitor ECE as a first-class metric, not an afterthought."

6. Features — feature store & point-in-time correctness

A feature store exists to prevent two failure classes. Name both.

  1. Train/serve skew — the same feature computed by two code paths (batch SQL for training, an online service for serving) drifts apart. Fix: a single feature-definition source materialized for both offline and online.
  2. Label leakage via point-in-time joins — the subtle, high-impact one. When you build a training row for (query, item, t), every feature must be joined as of time t (feature_ts ≤ t), not at its latest value. Accidentally join the current item-CTR into a row from last week and you've leaked the future.

Trap: This leakage inflates offline AUC by 5–20% while online collapses — it is the single most common "great offline, dead online" bug. Feast/Tecton-style time-travel joins exist precisely for this. When someone describes that symptom, audit point-in-time correctness first (§12).

Freshness tiers — say you'd tier features, not treat them uniformly:

Tier Example Path Freshness
Static item attributes, category batch hours OK
Near-real-time item CTR/CVR micro-batch minutes
Streaming session/context features Flink/Kafka seconds

The streaming path is where the cost and the on-call pain live.

Say it like this (own the gap gracefully): "I know the point-in-time correctness contract cold — it's the highest-leverage correctness bug in the whole system. The streaming ops at 100k events/sec is where I'd pair with the platform team; I've run this at research and shadow-eval scale, not a live 3am rotation, and I'd rather be precise about that."

7. Evaluation bridge — offline → interleaving → A/B

The general playbook jumps from offline NDCG straight to A/B. The in-between is where modern teams get 10–100x faster. Present it as a three-phase funnel:

  1. Offline — NDCG@10 / MRR / recall@k on graded relevance + replayed logs. Cheap, fast, weakly predictive. Gate: don't ship a regression.
  2. Interleaving — for candidates that pass offline, mix results from ranker A and ranker B into one list per user and attribute each click to whichever ranker sourced it. Because every user sees both rankers, it removes between-user variance → ~10–100x more sensitive than A/B, so it calls a winner on a small fraction of the traffic and runtime an A/B would need. Published interleaving pipelines (Netflix, Bing) report the same 10–100x sensitivity win. This is the tool that lets a research team iterate weekly instead of monthly.
  3. A/B — only interleaving winners graduate to a real A/B on the business metric (GMV/session). User-level randomization (not session — spillover), ≥2 weeks for weekly seasonality, per-market splits (DE/FR/ES/IT effects genuinely differ). Canary 1% → 10% → 50% → 100% with automated rollback on guardrail breach.

Name interleaving's limit unprompted (shows judgment): "Interleaving measures ranking preference, not downstream marketplace effects — seller supply response, long-term retention. So it's a fast filter, not a replacement for A/B on GMV."

Two-sided-marketplace caveat — the A/B itself is biased (SUTVA violation). A plain buyer-randomized A/B silently assumes one buyer's treatment doesn't affect another's outcome (SUTVA). On a marketplace that's false: ranking reallocates seller impressions and supply — a treatment that surfaces different sellers changes the impressions, prices, and stock that control buyers then see, so the two arms interfere and the buyer-side readout is contaminated (and the seller side is invisible to buyer randomization entirely). I'd measure the seller side with a design that randomizes at the interference boundary: cluster / region randomization (whole markets or seller-cohorts in one arm), a budget-split / two-sided design, or a switchback (flip the whole system on a time cadence) so supply-side effects are inside the experimental unit, not leaking across it.

8. Serving & scale — the two facts most candidates miss

Don't re-draw the 200ms latency budget (it's in the general playbook). Add the two scale decisions that reveal real experience:

ANN index choice is a scale/cost decision, not a default.

Index Use when Cost profile
HNSW recall is king, RAM is plentiful (≤~100M vectors in-memory) ~5–20ms, memory-heavy
IVF-PQ / DiskANN billion-scale, memory-bound compressed, partly on disk, slower
FAISS-GPU / managed ScaNN (Vertex) billions with dynamism throughput-oriented

Do the arithmetic out loud — it forces the design. 2B vectors × 768 dims × 4 bytes (fp32) ≈ 6.1 TB of raw embeddings. That does not fit in one box's RAM, and you're not paying to keep 6 TB hot. Product quantization (say 768 dims → ~96 bytes/vector) compresses it to ~190 GB — now it fits across a handful of memory nodes. That single calculation is why the index is IVF-PQ / DiskANN, not flat HNSW, and why it's sharded.

Shard / replica topology + scatter-gather. Partition the index across N shards (by hash, or by a coarse IVF/semantic-ID bucket so related items co-locate), each shard replicated R times for QPS and availability. A query scatters to all N shards (or to the shards its coarse bucket selects), each returns its local top-k, and a broker gathers and merges to a global top-k. Then a two-stage retrieve: stage 1 does a cheap coarse-quantized / PQ scan to get a large candidate set fast, stage 2 re-ranks that shortlist against fuller-precision vectors (or the raw fp32 for the survivors) to recover the recall PQ threw away. Tail latency is set by the slowest shard, so replicas + hedged requests matter as much as the index itself.

eBay's 2B listings + constant new-listing churn → you need incremental/streaming indexing: a separate hot index for <24h listings, merged at query time, because a full re-embed + re-index on every encoder update is untenable.

Embedding version-sync — the failure mode nobody mentions until it bites. The query tower and the item index must be the same tower version: cosine similarity is only meaningful between vectors from the same embedding space. So a tower retrain is not a model swap, it's a full 2B item re-embed — every item vector in the index is now from the old space and incompatible with the new query tower. You can't do that in place while serving. The pattern: build the re-embedded index as a shadow index offline, validate it, then atomically dual-cutover the query tower and the item index together (flip both at once behind the router). This is a different mechanism from the hot-index-for-fresh-listings trick above: the hot index adds new items under the current tower version; the shadow-index cutover replaces the entire index because the tower version itself changed. Conflating the two is the classic "why did recall fall off a cliff after we shipped the new encoder" incident.

The ranking model's real memory hog is embedding tables, not the MLP. Item/seller/category ID embeddings dominate: at web scale they set the memory-capacity ceiling and the bulk of the all-to-all communication in RecSys training/serving — not the FLOPs (the lookups are memory-/bandwidth-bound, the MLP on top is tiny). So "scaling the model" here means sharding embedding tables (model-parallel across GPUs, or hybrid CPU-hosted embeddings + GPU top-MLP), not adding layers. (Industry RecSys is trending toward LLM-scale via embedding-table engineering, not deeper nets.)

9. Cold start — lean on content until behavior exists

The content side of the two-tower is the cold-start solution. Say that explicitly.

  • New item: its content/visual embedding places it in the two-tower space on day one — it inherits its neighbors' priors. Add a small, decaying exploration boost until ~100 impressions accumulate; transfer seller reputation as a prior (a seller with 10k clean transactions starts with higher trust).
  • New user: session-bootstrap — the first ~3 interactions update real-time features immediately — plus a popularity/geo prior. Optional cold-start UI (ask 3 preference questions at signup) to seed.
  • New market (MT-adjacent): zero-shot multilingual capability day 0 → RAG bootstrap from existing bilingual inventory → LoRA once ~1k examples exist.

Say it like this: "I don't treat cold start as a bolt-on. The two-tower's content tower already embeds a brand-new listing into the same space as established items, so it gets reasonable neighbors immediately. The only thing missing is behavioral signal, and a decaying exploration boost buys that in ~100 impressions."

10. Feedback-loop collapse — instrument it, don't hope

A recommender trained on its own impressions concentrates: it shows popular items → they get clicked → it learns they're good → shows them more. Left unchecked for ~48h without exploration, the served distribution can concentrate to a ~10x narrower item set than the catalog. Make it measurable:

  • Impression Gini alarm: track the Gini coefficient of impressions across items/sellers, daily. Alarm if it rises past ~0.7.
  • Exploration budget: non-negotiable 5–10% of impressions. Prefer Thompson sampling over ε-greedy/UCB — it adapts the exploration rate to the model's uncertainty, not just to item age.
  • Diversity enforcement: MMR (penalize items too similar to already-selected) + category/seller caps (e.g. max 3 from one seller in the top 10).

Say it like this: "Feedback-loop collapse isn't a philosophical risk, it's a number I watch — impression-Gini. If it crosses ~0.7 that's an alarm. The fix is a fixed exploration budget with Thompson sampling, and it doubles as the clean, uniform-propensity signal that de-biases my training data. Same mechanism solves two problems."

11. Monitoring — three layers

Layer What Signal / threshold
Input feature drift, null-rate PSI/KS per feature (alarm PSI > 0.2), null-rate spikes
Model calibration + quality ECE creep; online NDCG proxy on a fresh held-out slice
Business end-to-end health GMV/session, seller-Gini, latency p99

Seasonality is the trap: e-commerce drifts hard around holidays and end-of-season. Use time-aware features and expect scheduled retrains — a drift alarm that fires every December isn't a bug, it's the calendar.

12. Failure modes I'd volunteer

Volunteering these unprompted is a strong-senior signal — it shows I've been burned and know where to look.

Failure Tell Mitigation
Great offline, dead online AUC↑ offline, flat/down A/B Audit point-in-time leakage (§6) first — it's the usual culprit
Retriever returns popular-generic, misses tail tail-query recall low, GMV flat on long-tail sessions logQ correction + hard negatives (§4); track tail-recall
Calibration drift ECE creeps up while AUC steady scheduled recalibration; per-segment reliability diagrams
Ranker gamed keyword-stuffed titles rank separate exact-match features; seller-trust signals; editorial audit
Index staleness new listings invisible for hours hot near-real-time index merged with main (§8)
Feedback collapse impression-Gini↑ exploration budget + diversity caps (§10)

"Great offline, dead online" is a differential diagnosis, not one bug — I'd rank the causes (cheapest/most-common first) rather than name a single culprit:

  1. Point-in-time leakage (§6) — a future-valued feature joined into a past row inflates offline AUC 5–20% and vanishes online. Most common, cheapest to check, so it's #1.
  2. Train/serve skew (§6) — the feature is computed by two code paths (batch SQL vs. online service) that quietly diverge, so the model sees different inputs offline vs. in production.
  3. Offline candidate-set / incumbent selection bias — the subtle one. Offline eval replays logs that only contain items the incumbent ranker chose to show. A better retriever surfaces previously-unshown, unlabeled items, which score as misses on logged labels — so it looks worse offline while being better online. The fix isn't to trust offline; it's off-policy / counterfactual evaluation with exploration-propensity labels from the 5% slice (§10), so unshown-but-relevant items aren't silently counted as negatives.
  4. Proxy-metric divergence — NDCG@10 climbs but GMV/session stays flat (§2's proxy/target gap). The offline number improved; the thing you actually sell didn't.
  5. Train-window vs serving distribution shift — the model was trained on a window that no longer matches live traffic (seasonality, catalog turnover, a new market), so honest offline numbers on stale data don't transfer.

Say it like this (the meta-move): "When someone tells me a model was great offline and died online, I don't reach for the architecture — I run a ranked differential: point-in-time leakage first because it's the most common and cheapest to check, then train/serve skew, then whether offline eval is penalizing the retriever for surfacing items the incumbent never showed, then proxy-vs-GMV divergence, then plain train/serve distribution shift."

13. The 60-second spine (open with this, then expand on probing)

"Retrieve-then-rank over 2B items. Query understanding normalizes the query and produces an embedding. Candidate generation is multi-channel — a two-tower dense retriever trained with in-batch negatives plus logQ correction plus hard negatives, because sampling bias is the whole game, alongside BM25 for exact codes and item-to-item CF — fused by a learned, query-dependent ranker rather than fixed weights. Ranking is a multi-task PLE/MMoE model predicting click/cart/purchase, blended to a GMV-aligned score and calibrated — ECE-monitored — because that score feeds business rules. Features come from a store with point-in-time joins so I don't leak the future. I prove wins offline (NDCG) → interleaving (fast, ~100x more sensitive) → A/B (GMV, per market). Cold start rides the content side of the two-tower; feedback-loop collapse I watch via impression-Gini with a Thompson-sampling exploration budget. And if you ask where I'd take it — generative retrieval with semantic IDs, which is where recsys and foundation models converge."

That last clause is the bait for §14 — it invites the frontier question onto Tony's turf.

14. The frontier fork — generative retrieval / semantic IDs

If they push "what would you build in two years / where is the field going," this is the high-signal answer and it's Tony's lane (recsys × foundation models).

The idea. Replace embed → ANN → rank with a single generative model that emits the item-ID sequence directly. Items are tokenized into semantic IDs — a hierarchy of codebook tokens from an RQ-VAE over content embeddings, so similar items share ID prefixes. A Transformer (encoder-decoder) then generates the next item's semantic ID given user history, collapsing retrieval and ranking into one model.

A real production instance to cite (not just a benchmark): Kuaishou's OneRec (2025) — encoder-decoder + sparse MoE, session-wise list generation, DPO preference-alignment on top. Deployed to hundreds of millions of DAU, retrained daily, and it beat the classic cascade in production — which TIGER didn't; TIGER was a benchmark result.

The honest trade-offs (this is what earns the point — don't sell it as a free lunch):

Concern Two-tower cascade Generative / semantic-ID
Cold start content tower embeds new item day 0 new item has no learned ID token → need content-derived semantic IDs to place it
Serving one ANN lookup (~ms) autoregressive decoding, pricier — but unifies two systems into one
