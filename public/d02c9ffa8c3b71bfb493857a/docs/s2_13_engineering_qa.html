Engineering Q&A — Build & Train

Spoken-ready Q&A for Interview 1 (Live Coding & Engineering) covering Docker, testing, CI/CD, and — going deep — distributed training and infra optimization. Read the blockquotes out loud until they're your own words; there's no AI in the room. For anything production-ops (canary, live drift, on-call), see "The Honest MLOps-Gap Script".

How to use under pressure: don't recite. Internalize the shape of each answer — the one number, the one trade-off, the one story. This loop deep-probes, so if they push past the script, fall back to first principles and derive rather than recall. Training (Topic 4) is your home turf — go there confidently.

TOPIC 1 — Docker & Containers for ML

Q1.1 — Why containerize an ML application at all?

Say it like this:

The core problem in ML is reproducibility. A model that trains fine on my laptop breaks in production over a different CUDA version, a different NumPy, or a missing system library.

A container pins the entire environment — OS libraries, Python deps, CUDA and cuDNN — into one immutable image, so "works on my machine" becomes "works everywhere."

That matters twice for ML: once for training reproducibility, and again for serving, where I need the exact same preprocessing and library versions the model was trained against. It's the same instinct as fixing random seeds — I'm removing hidden sources of variance from the system.

Q1.2 — How do you get GPU access inside a container?

Say it like this:

You install the NVIDIA Container Toolkit on the host — not inside the container — and it bridges Docker to the host's NVIDIA driver. The container ships CUDA and cuDNN, but the driver always lives on the host; you never install drivers inside the image. Then you run with docker run --gpus all, or --gpus '"device=0,1"' to pin specific GPUs, and inside the container nvidia-smi should list your devices.

Trap: the CUDA toolkit version in the image must be compatible with the host driver. The driver is backward-compatible with older CUDA, not forward — so a too-new CUDA image on an old driver fails.

Q1.3 — Your ML image is 15 GB. How do you shrink it?

Say it like this:

First, multi-stage builds — build wheels or compile custom kernels in a heavy -devel base with compilers and nvcc, then COPY --from=builder only the artifacts into a slim -runtime base, dropping the whole build toolchain. Second, pick the right base: nvidia/cuda:xx-runtime instead of -devel, or a slim Python base, and skip a full Conda if pip wheels suffice. Third, order layers by change frequency so dependency layers stay cached, use a .dockerignore to keep datasets and checkpoints out of the build context, and collapse RUN commands so apt-cache deletions actually land in the same layer.

# builder stage: heavy, has compilers
FROM nvidia/cuda:12.4.0-devel AS builder
COPY requirements.txt .
RUN pip wheel --wheel-dir /wheels -r requirements.txt

# runtime stage: slim, only the built artifacts
FROM nvidia/cuda:12.4.0-runtime
COPY --from=builder /wheels /wheels
RUN pip install --no-index --find-links=/wheels /wheels/*
COPY src/ /app/src/          # code last: only this layer rebuilds on edits

Q1.4 — Walk me through a good Dockerfile layer strategy.

Say it like this:

Layers cache top-to-bottom and invalidate on the first change, so I put the least frequently changing things first: base image, then system packages, then Python deps, then finally the application code. That way editing a training script rebuilds only the last layer instead of re-installing PyTorch every time. I pin versions explicitly — torch==2.x, a fixed base-image digest — because a floating latest tag silently breaks reproducibility. And large data stays out entirely: datasets and weights get mounted as volumes or pulled from object storage at runtime, never baked into the image.

Q1.5 — Docker Compose or Kubernetes for serving?

Say it like this:

Depends on scale and who operates it. Compose is single-host, near-zero learning curve — great for local dev, integration tests, or a small service where one box carries the load; you get 80% of the value for 20% of the complexity. Kubernetes earns its complexity when you schedule across many nodes, autoscale on custom signals, do GPU scheduling via device plugins, or hit strict availability SLAs — that's where Compose hits a wall, especially for GPU endpoints. For a research-to-prod handoff I'd prototype the serving container in Compose and hand K8s, or a managed layer like KServe, to the platform team — which is candidly also where my hands-on production-ops experience ends, and I'd say so.

Q1.6 — How do you handle secrets and config in containers?

Say it like this:

Never bake secrets into the image — anything in a layer is recoverable by anyone who pulls it. I pass config through environment variables or mounted config files, and secrets through the orchestrator's secret store — K8s Secrets, Docker secrets, or a vault — injected at runtime. Twelve-factor style: config lives in the environment, not the code, so the same image promotes cleanly from staging to prod with only the env changing. For ML that also means model version, feature-store endpoints, and thresholds are config, not hardcoded constants.

Q1.7 — Container vs VM, briefly?

Say it like this:

A VM virtualizes hardware and runs a full guest OS with its own kernel, so it's heavy — gigabytes and slow to boot. A container shares the host kernel and isolates just the process, filesystem, and network namespaces, so it's megabytes of overhead and starts in milliseconds. For ML that's why containers won — I can pack many inference replicas onto one GPU host cheaply and spin them up fast for autoscaling. The trade-off is weaker isolation: containers share a kernel, so it's process isolation, not a security boundary as hard as a VM.

Connect it to my work (Docker): My compression work points at image and serving footprint. In my nnU-Net pruning paper I removed >80% of a trained model's weights at proxy Dice >0.95 — but I'm careful about what that buys: it's a weight-REDUNDANCY result, and it's UNSTRUCTURED, so I don't claim a speedup, and the container/replica payoff — packing more replicas per GPU — needs the STRUCTURED follow-up to actually land. I frame containerization as the delivery mechanism for efficiency work: routing cuts the compute a request needs, and a tight multi-stage image is how that saving reaches production. It ties to eBay's own results too — a finetuned Gemma3-4B beating a zero-shot Gemma3-27B at ~3.8x faster inference is exactly the kind of model that packs into a lean GPU container.

TOPIC 2 — Testing ML Systems

Q2.1 — How is testing an ML system different from testing normal software?

Say it like this:

In normal software the logic is deterministic — same input, same output — so tests are straightforward. ML adds two headaches: non-determinism (random init, stochastic optimization, and data shuffling mean two runs differ), and the fact that behavior is defined by data, not just code — so a passing unit test doesn't mean the model is good. So I test on three axes — code, data, and model — and I kill non-determinism by fixing seeds across NumPy, PyTorch, and Python, and pinning the environment in a container. The mental model I like: assert on properties and tolerances, not exact float equality.

Q2.2 — What would you actually unit-test in an ML pipeline?

Say it like this:

The deterministic pieces, which is more than people think: data transforms and feature engineering, tokenization, the shape and dtype of tensors flowing between stages, and any custom loss or metric. For a feature like "days since last purchase," I write a unit test with a hand-crafted input and the exact expected value — that's pure logic, fully deterministic. I also test the plumbing: that a forward pass runs, that loss drops on a single-batch overfit test, and that a checkpoint saves and reloads to identical weights. These run fast and catch the majority of real bugs before any training happens.

Q2.3 — How do you handle non-determinism so tests don't flake?

Say it like this:

First, seed everything — torch.manual_seed, numpy.random.seed, Python's random, and PYTHONHASHSEED; for full GPU determinism I'd add torch.use_deterministic_algorithms(True), accepting it may be slower. Second, assert with tolerances — torch.allclose with an atol/rtol instead of exact equality — because floating-point and hardware differences make bit-exactness unrealistic. Third, pin the environment in a container so CI runs on the same library and CUDA versions I developed against. And where I genuinely can't remove variance, I test statistical properties — accuracy above a threshold, output distribution in range — rather than a single value.

def seed_everything(seed=42):
    random.seed(seed); np.random.seed(seed)
    torch.manual_seed(seed); torch.cuda.manual_seed_all(seed)

# assert on tolerance, never == on floats
assert torch.allclose(out, expected, atol=1e-5, rtol=1e-4)

Q2.4 — Unit vs integration tests here — what does each look like?

Say it like this:

A unit test isolates one deterministic function — a feature transform, a metric — with a tiny hand-made input; it's milliseconds and needs no services. An integration test wires real stages together — read data, engineer features, write to the store; or load model then serve a prediction — to catch interface mismatches the unit tests can't see, so it's slower because it touches real services and sample data. In ML I add an end-to-end pipeline test on a small fixture dataset: does the whole train-eval loop run and produce a model artifact without crashing? I gate every commit on the fast unit tests and run the heavier integration tests pre-merge or on a schedule.

Q2.5 — How do you validate the data, not just the code?

Say it like this:

Code tests won't catch schema drift or a column that's suddenly 40% nulls — that needs data validation as a first-class gate. I'd use something like Great Expectations to declare expectations — column types, value ranges, null rates, category sets — and run them as checkpoints at pipeline boundaries: on ingest, after feature engineering, before training. In a pipeline you can drive those expectation suites from pytest so a validation failure fails the build like any other test. The principle is fail-fast: bad data caught at ingest is cheap; bad data discovered as a degraded model in production is expensive.

Q2.6 — Beyond accuracy, how do you test model behavior?

Say it like this:

I borrow the CheckList framework — invariance, directional, and minimum-functionality tests. An invariance test applies a label-preserving perturbation, like swapping a synonym or a name, and asserts the prediction doesn't change; a directional test makes a change that should move the output a known way — adding strongly negative text shouldn't make sentiment more positive; and an MFT is a simple hand-built case the model must get right. These catch brittleness and bias that aggregate accuracy hides — critical for an e-commerce model touching real listings and buyers. It's the difference between "the model scores 0.9" and "the model behaves sensibly on the cases I care about."

Q2.7 — How do you keep tests fast enough for CI?

Say it like this:

Tier them. Cheap deterministic unit tests run on every push and must finish in seconds — so I use tiny synthetic fixtures, not real datasets, and mock external services. The slow stuff — full pipeline runs, training smoke tests, large data validation — runs pre-merge or nightly, not on every commit. For model tests I use a "tiny model, tiny data" pattern: two layers, ten samples, one step — enough to prove the code path works without paying for real training. I also cache dependencies and Docker layers in CI so setup isn't the bottleneck.

Connect it to my work (Testing): My routing work is a natural testing story — a router is a decision function, so I can unit-test it hard: given a query with known difficulty, does it route to the right tier? On MERA I care about 87.3% router-decision accuracy at 51.8% serving cost, so I frame router evaluation as behavioral testing — invariance (paraphrase the query, routing shouldn't flip) and directional (a harder query should escalate to the stronger model). And benchmarks like TwinRouterBench are essentially regression-test suites for routing quality: they let me prove a change didn't quietly degrade decisions before it ships.

TOPIC 3 — CI/CD for ML

Q3.1 — What does CI/CD look like for ML, and how's it different?

Say it like this:

Standard CI/CD versions and tests code; ML has to version and test the trifecta — code, data, and model — as one unit, because a result is only reproducible if all three are pinned. So my pipeline, triggered on a git push through something like GitHub Actions, runs lint and unit tests, then data validation, then optionally a training smoke test, and gates a model artifact into a registry. The extra machinery is versioning: Git for code, DVC for data and large artifacts, and MLflow for experiment tracking and the model registry. The goal is that any past model can be rebuilt exactly from a commit hash plus a data version plus the recorded hyperparameters.

Q3.2 — Data and models are too big for Git. How do you version them?

Say it like this:

DVC is the standard answer — it stores small pointer files in Git that reference the large data or model blobs in remote object storage like S3. So git checkout of an old commit plus dvc pull reconstructs the exact dataset and model that commit was built with — Git-like versioning without bloating the repo. For experiments and lineage I pair it with MLflow, which logs parameters, metrics, and artifacts per run and gives a registry with stage transitions, staging to production. Together they make a model traceable back to the code, data, and config that produced it.

Q3.3 — How do you make an ML result reproducible?

Say it like this:

Pin all four layers: code via git commit, data via a DVC hash, environment via a pinned Docker image, and randomness via fixed seeds. Then log the run — hyperparameters, metrics, git SHA, data version — in MLflow or Weights & Biases so the record stays queryable. The test I hold myself to: someone else, months later, can go from a commit hash to a bit-similar model without asking me anything. True bit-exactness on GPU is hard, so in practice I aim for pinned seeds plus deterministic-algorithm flags and accept tight tolerances.

Q3.4 — What runs in a GitHub Actions workflow for an ML repo?

Say it like this:

A YAML workflow triggered on push or PR. Cheap stages first — checkout, restore cached deps, lint, and fast unit tests — these gate every commit. Then data validation and a short training or inference smoke test on a fixture, so I catch breakage without a full training run in CI. On merge to main I'd build and push the Docker image, register the model artifact, and optionally kick off a real training or eval job on a GPU runner. The philosophy is fail-fast and cheap-first: order stages so the quickest checks reject bad changes before you spend on expensive ones.

Q3.5 — Would you actually train a model inside CI?

Say it like this:

Not the full run — GPU minutes are expensive and CI should be fast. In CI I run a training smoke test: tiny model, tiny data, one or two steps, asserting the loss is finite and the pipeline produces an artifact — that proves the code path without the cost. Real training I trigger as a separate job on dedicated GPU infrastructure, orchestrated but not blocking the merge. There's a pattern called CML — Continuous Machine Learning — that posts training metrics back as a comment on the PR, which keeps a human in the loop on model quality.

Q3.6 — How do you deploy a new model version safely?

Say it like this:

The model is an artifact promoted through the registry — staging, then production — with the deployment being an image plus a model version, so I can roll back by re-pointing to the previous version. The safe rollout patterns are canary — route a small percentage of traffic to the new model and watch metrics — and shadow or blue-green, running the new model alongside the old on live traffic without serving its output, and comparing. I'll be honest that I understand these patterns more than I've operated them at scale; my hands-on strength is the model efficiency that makes a rollout cheaper, not the production canary tooling itself. As an offline gate I'd also require the new model to beat the incumbent on a held-out eval before it's even eligible to deploy.

(For the full canary / live-drift / on-call answer, see "The Honest MLOps-Gap Script".)

Q3.7 — Where do MLflow and DVC actually fit together?

Say it like this:

Think of it as: DVC versions the inputs, MLflow tracks the process and outputs. DVC pins datasets and large artifacts to storage with Git-tracked pointers; MLflow logs each experiment's params, metrics, and resulting model, and hosts the registry that serving pulls from. A clean setup wires them so an MLflow run records which DVC data version it consumed — now lineage is complete end to end. Starting fresh I'd reach for MLflow first for tracking and registry, and add DVC once the datasets change often enough that versioning them by hand hurts.

Connect it to my work (CI/CD): Reproducibility is where my research habits pay off — my routing and compression results only mean something if they reproduce, so pinning seeds, data versions, and configs is already how I run experiments; MLflow and DVC are just the productionized version of that discipline. Benchmarks like TwinRouterBench are effectively CI gates for a router — a new routing policy has to clear the benchmark before it ships, same as a model has to beat the incumbent on held-out eval. And my Diffusion Weights work — accelerating generative training ~3.2x — is a CI story too: faster training means the retrain-and-evaluate loop turns over cheaper and more often.

TOPIC 4 — Distributed & Efficient Training (Infra Optimization)

This is the deep-probe zone and your edge. Derive the numbers, don't recite them.

Killer numbers to have cold:

Fact Value
Mixed-precision Adam memory 18 bytes / param
7B model, states only ~126 GB (7B × 18)
ZeRO paper figure 16 B/param (assumes fp16 gradients, not fp32)
Activation checkpointing cost ~+33% compute (one extra forward)
Good large-scale MFU ~40–50%
FlashAttention memory O(N²) → O(N)

Q4.1 — Explain the memory footprint of training a large model.

Say it like this:

Four buckets — model weights, gradients, optimizer states, and activations. For mixed-precision training with Adam, the per-parameter cost derives cleanly: 2 bytes for the bf16 weights used in the forward and backward, 4 bytes for the fp32 master copy of the weights, 4 bytes for the fp32 gradient, and 8 bytes for Adam's two states — momentum and variance — in fp32. That's 2 + 4 + 4 + 8 = 18 bytes per parameter before activations. So a 7B model is ~126 GB just for states, which is exactly why it doesn't fit on one GPU and you need sharding.

Component Precision Bytes/param
Weights (compute copy) bf16 2
Master weights fp32 4
Gradients fp32 4
Adam momentum fp32 4
Adam variance fp32 4
Total 18

Trap (if they push): the original ZeRO paper quotes 16 bytes/param because it assumes fp16 gradients rather than fp32 — the 16-vs-18 difference is entirely the gradient precision. Flag that and you show you actually understand the derivation.

Q4.2 — The fifth bucket is activations — why do they matter and how do you control them?

Say it like this:

Activations are the intermediate tensors saved during the forward pass so the backward pass can compute gradients, and they scale with batch size and sequence length — so for long-context or large-batch training they can dominate, often more than the parameters. The main lever is gradient (activation) checkpointing: you discard most activations in the forward pass and recompute them in the backward pass, trading compute for memory. It costs roughly one extra forward pass — about +33% compute — in exchange for a large memory saving that lets you fit bigger batches or models.

Back-of-envelope (have the sizing ready): activations scale as roughly layers × batch × seq × hidden × bytes — the intermediate tensors stacked across the network. The Megatron per-transformer-layer form makes it concrete: s·b·h·(34 + 5·a·s/h) bytes without checkpointing, where s=seq, b=batch, h=hidden, a=attention heads. The 5·a·s/h term is the attention score matrix — it grows with sequence length, which is why long context is what blows up activation memory. Full checkpointing collapses that to storing just each layer's input, roughly 2·s·b·h·layers bytes (the ×2 is bf16's 2 bytes/element), and recomputes the rest.

Trap: checkpointing actually lowers your reported MFU, because those recomputation FLOPs are real work the GPU does but they don't count as "useful model FLOPs." Interviewers love that subtlety.

Q4.3 — DDP vs FSDP — what's the difference and when do you use each?

Say it like this:

DDP — Distributed Data Parallel — replicates the full model and optimizer state on every GPU, splits the data across them, and does an AllReduce to average gradients each step; simple and fast, but only works if the whole model fits on one GPU. FSDP — Fully Sharded Data Parallel — shards the parameters, gradients, and optimizer states across GPUs, and gathers each layer's parameters just-in-time for its forward/backward, then frees them, so it fits models far bigger than one GPU. The key framing is that FSDP trades communication for memory — you pay extra all-gather and reduce-scatter traffic to avoid holding the full model. So DDP when the model fits and you want throughput; FSDP when it doesn't and memory is the binding constraint.

Q4.4 — Explain DeepSpeed ZeRO stages 1, 2, and 3.

Say it like this:

ZeRO progressively shards the three redundant pieces of state across data-parallel GPUs. Stage 1 shards the optimizer states; Stage 2 adds gradients; Stage 3 adds the parameters themselves — so each stage saves more memory at the cost of more communication. The clean way to remember it: 1 = optimizer, 2 = + gradients, 3 = + parameters. And PyTorch FSDP full-shard is essentially equivalent to ZeRO-3 — same idea, different framework lineage. In practice you pick the lowest stage that makes your model fit, because higher stages add communication overhead.

Stage Shards Mnemonic
ZeRO-1 optimizer states optimizer
ZeRO-2 + gradients + gradients
ZeRO-3 + parameters + params (≈ FSDP full-shard)

Q4.5 — Tensor, pipeline, data, and sequence parallelism — and when do you combine them?

Say it like this:

Data parallelism replicates the model and splits the batch — the default, easiest, scales throughput. Tensor parallelism splits individual layers' matrices across GPUs, each doing part of a matmul, so it's communication-heavy and you keep it within a fast NVLink node.

Pipeline parallelism splits the model by layers across GPUs and streams micro-batches through to keep them busy, minimizing the pipeline "bubble." Sequence parallelism shards along the sequence dimension for very long contexts.

For the biggest models you combine them — 3D parallelism, data + tensor + pipeline — mapping tensor-parallel inside a node, pipeline across nodes, and data-parallel on top. The mental rule: chatty parallelism, tensor, goes where the interconnect is fastest.

Q4.6 — bf16 vs fp16 for mixed precision — does it matter?

Say it like this:

Yes, and it's a favorite gotcha. bf16 has the same 8-bit exponent as fp32, so it keeps fp32's dynamic range while giving up mantissa precision — that means gradients rarely underflow and you don't need loss scaling. fp16 has a much narrower exponent range, so small gradients underflow to zero and you must use a GradScaler, or loss scaling, to shift them into representable range. On modern accelerators — A100, H100 — bf16 is the default for training precisely because it's more numerically robust with less babysitting. Either way you keep an fp32 master copy of the weights and do the optimizer update in fp32 for stability.

Trap: if you say fp16 and forget the GradScaler, your training silently diverges from gradient underflow. bf16 is the safer default because it removes that failure mode.

Q4.7 — What is MFU and why do we care?

Say it like this:

MFU — Model FLOPs Utilization — is the fraction of the GPU's theoretical peak FLOPs your training actually converts into useful model computation. It's the honest efficiency metric: ~40–50% MFU on large-scale training is considered good, and it maps directly to how much of your very expensive GPU time isn't wasted. It's distinct from HFU — Hardware FLOPs Utilization — which counts recomputation FLOPs, so activation checkpointing raises HFU but lowers MFU, because the recompute is hardware work that isn't "model" work. I care about MFU because it's the training-side analogue of the serving-cost efficiency I optimize for — same result, less compute burned.

Q4.8 — Warmup and cosine decay — why, and what changes for continued pretraining?

Say it like this:

Warmup — a short linear ramp over roughly 0.5–2% of steps — lets the optimizer's momentum and variance estimates stabilize before you apply a large learning rate, which prevents the early divergence you get from big updates on a cold optimizer. Then cosine decay smoothly anneals the LR down, often to about 10% of peak, which empirically converges better than a flat rate. The important nuance for continued pretraining — exactly what eBay does with e-Llama — is you restart a short warmup and use a lower peak LR, because you're adapting an already-trained model and a full-size LR would blow away what it learned. So the warmup there is about re-stabilizing the optimizer on the new data distribution, not training from scratch.

Q4.9 — Gradient accumulation — what is it and when do you reach for it?

Say it like this:

Gradient accumulation lets you simulate a large batch on limited memory by running several micro-batches forward/backward, summing their gradients, and only stepping the optimizer after K of them. Done right, accumulating K micro-batches on one GPU is mathematically equivalent to a K-times-larger batch, or to data-parallel training across K GPUs. I reach for it when I need a large effective batch for stability but the batch won't fit in memory — it trades wall-clock time for memory.

for i, batch in enumerate(loader):
    loss = model(batch) / K          # divide by K → average, not sum
    loss.backward()                  # accumulate grads
    if (i + 1) % K == 0:
        optimizer.step(); optimizer.zero_grad()

Trap: you must divide the loss by K so the accumulated gradient is the average, not the sum — otherwise your effective learning rate is K times too big.

Q4.10 — What is FlashAttention and why did it matter?

Say it like this:

Standard attention materializes the full N×N score matrix in GPU HBM, so memory and bandwidth blow up as O(N²) with sequence length.

FlashAttention is a fused, IO-aware kernel that tiles the computation into blocks that fit in fast on-chip SRAM and uses an online softmax to compute the normalized output block-by-block. It never writes the full matrix to HBM — memory drops from O(N²) to O(N), and it's faster because it moves bandwidth-bound work off HBM.

FlashAttention-3 goes further on Hopper by overlapping data movement (TMA) with the matmuls and using low precision, hitting ~75% of peak. The upshot: it's what makes long-context training and inference tractable, and it's essentially free, accuracy-preserving speedup.

Q4.11 — How do you checkpoint and resume a long training run robustly?

Say it like this:

You periodically save the full training state — not just model weights, but optimizer states, the LR-scheduler position, the RNG states, and the step count — so a resume continues exactly where it left off rather than restarting the schedule. On a multi-day run this is non-negotiable because hardware fails; you want to lose minutes, not days. For sharded training — FSDP or ZeRO — checkpointing is distributed, each rank saves its shard, and you want asynchronous saving so you're not stalling all the GPUs while writing to disk. A good sanity check: resume from a checkpoint and confirm the loss curve continues smoothly instead of spiking, which tells you the optimizer and RNG state were actually restored.
