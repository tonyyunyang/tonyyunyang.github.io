Live-Coding Playbook

The core of Interview 1's coding half: how to narrate while you code, how to recover when you stall, and the ranked pattern table to drill. Read this the week before, then re-skim the morning of. ML primitives (softmax, attention, metrics, BPE) live in the companion ML From Scratch (Python) doc — this one is DS&A + narration.

The one job: make your thinking audible

Communication is the #1 graded criterion — above speed, above a perfectly optimal solution. A correct answer explained poorly scores below a partial answer explained clearly. Treat the interviewer as a pair-programming partner you're thinking out loud with, not an examiner you're performing for.

Your specific risk profile: you know the algorithms and you're AI-native, so the two things that are actually rusty are (a) raw unaided typing/recall with no autocomplete, and (b) narrating out loud while your hands move. This playbook drills exactly those two.

Time budget for a 60-min round, one main problem: ~8 min talking before you type · ~25 min coding · ~10 min testing/reviewing · buffer for follow-ups. Do not touch the keyboard for the first several minutes — silent-then-deliberate is itself a positive signal.

1. The UMPIRE narration protocol

Use UMPIRE as your spine: Understand → Match → Plan → Implement → Review → Evaluate. Below is the exact spoken script, beat by beat, with the literal phrases to say.

Beat 0 — Frame it (10 seconds, before anything)

Say it like this:

"Before I write anything, I'll make sure I understand the problem, walk a small example, state my approach and its complexity, and only then code. I'll think out loud the whole way — please stop me anytime."

This one sentence tells them you have a process and buys you permission to be deliberate instead of panic-typing.

Beat 1 — UNDERSTAND: clarify inputs, constraints, edge cases (2–4 min)

Restate the problem in your own words, then ask targeted questions. Write one or two concrete input→output examples in a comment as you go.

Say it like this:

"Let me restate: I'm given ___ and I need to return ___. Is that right?" "What are the input sizes — n up to 10³, 10⁶, 10⁹? That tells me what complexity I need to target." "Can the input be empty? Duplicates? Negatives? Non-ASCII for strings?" "Is it sorted? Can I mutate it in place, or must I keep it intact?" "If there are ties or multiple valid answers, is any one fine, or is there a tiebreak?" "What should I return for the degenerate case — empty input, or k > n?"

Name the edge cases out loud even if you don't ask about them — it shows you see them: "I'll keep empty-input and single-element cases in mind as I go." Input size is load-bearing: say what it implies. "n up to 10⁶ rules out O(n²), so I'm aiming for O(n) or O(n log n)."

Beat 2 — MATCH: name the pattern (30 seconds)

Say what family the problem is in before you plan. This is where the pattern table in §3 pays off.

Say it like this:

"This looks like a hashmap-for-counting problem." / "…a two-pointer on a sorted array." / "…a BFS on a grid."

Beat 3 — PLAN: state the approach AND its complexity before typing (2–3 min)

Always mention the brute force first, then the improvement — it shows range. Commit to a target time and space complexity before you write a line.

Say it like this:

"Brute force is O(n²): check every pair. I think I can do better — a hashmap gives O(n) time for O(n) space. Let me confirm the idea before coding." "My plan, three steps: (1) build a count map, (2) one pass checking each element's complement, (3) return the indices." "If two approaches tie, I'll say so and pick: a heap is O(n log k) and simpler to get right; a bucket sort is O(n). I'll start with the heap and mention the bucket optimization at the end."

Then get a nod:

"Does that approach sound reasonable to start?"

If they hesitate, you just saved yourself 20 minutes. Inviting a check here lets them redirect you cheaply, before you've written code.

Beat 4 — IMPLEMENT: code in small, verbalized steps (bulk of the time)

Narrate the intent of each block — say why, then type. Do not narrate keystrokes.

  • Announce structure first: "I'll write a helper for ___, then the main loop."
  • Per block, state intent then type: "Now I iterate the array, and for each element compute its complement and check the map."
  • Name variables meaningfully and say the meaning: "I'll call this seen — a dict from value to index."
  • Handle edge cases in the code and point at them: "I'm guarding the empty case here so we return early."
  • If you defer something, flag it — don't silently omit: "I'll stub input validation for now and come back if there's time — flagging it so we both know."
  • Fill silence with the plan, not apologies: "Okay, complement check done; next I store the index."

Beat 5 — REVIEW: hand-trace an example + edge cases, out loud (~8–10 min, do NOT skip)

Even with no run button, trace the code by hand like a debugger. This is where strong candidates separate from average ones.

Say it like this:

"Let me trace nums=[2,7,11], target=9. i=0: seen empty, store {2:0}. i=1: complement 2 is in seen → return [0,1]. Correct."

Then run the edge cases you named in Beat 1:

"Empty array → loop never runs → return []. Single element → no pair → []. Duplicates [3,3], target 6 → first 3 stored, second finds the complement → [0,1]. Good."

If you find a bug mid-trace, say it and fix it — catching your own bug out loud is a positive signal:

"Wait — I store after checking, so [3,3] works. If I'd stored before checking, an element would match itself. Good that the order is right."

Beat 6 — EVALUATE: 3-sentence complexity wrap-up + one improvement (2–3 min)

Volunteer space complexity without being asked. Use this three-sentence structure:

  1. Time: "One pass over n elements, each map op O(1) average → O(n) time."
  2. Space: "The map holds up to n entries → O(n) space."
  3. Tradeoff (only if one exists): "I traded O(n) space for speed. If memory were tight and the array were sorted, two pointers would be O(1) space but O(n log n) if I have to sort first."

Then proactively propose an improvement:

"Given more time I'd add input validation and unit tests for the empty and single-element cases. For very large n I'd think about streaming rather than holding everything in memory."

2. Recovery & hints playbook

The interviewer expects to hint — how you stall and how you take a hint are both graded. The failure mode isn't being stuck; it's going silent or ignoring the nudge.

When you're stuck — never go silent

  • Downgrade deliberately to a working baseline (the single most valuable move): "I don't see the O(n) trick yet, so let me get a correct brute-force O(n²) working first, then optimize — a working solution beats a stuck-perfect one."
  • Externalize the search out loud: "The bottleneck is the repeated lookup. What structure gives O(1) lookup? A hashmap — let me switch to that."
  • Narrate exactly where you're blocked: "I can tell whether a pair exists, but I'm not tracking indices yet. Let me change the value I store from the number to its index."
  • Buy time honestly, then actually think: "Give me twenty seconds to think." Said once, this reads as composure, not weakness.

When given a hint — treat it as gold

A hint is information, not a penalty. Reading one fast and correctly is a stronger signal than never needing one.

  1. Acknowledge + paraphrase so they know it landed: "Thanks — so you're nudging me toward whether the array is sorted. If it is, I can drop the hashmap and use two pointers for O(1) space. Let me take that."
  2. Integrate, don't discard everything: "Good point — I'll keep my loop structure but swap the inner scan for a binary search."
  3. Extend past the hint to show you can build on it: "…and once I'm using two pointers here, the same idea generalizes to the 3Sum variant."

Trap: politely nodding at a hint and then continuing down your original path. Interviewers read that as not listening. Visibly change course.

Think out loud without rambling

There's a difference between narrating and filling air. The rule: narrate decisions and intent, not keystrokes, and not a monologue of every possibility.

  • Do: state the why before a block, name tradeoffs, announce the next step. "Next I need O(1) lookup, so I'll reach for a set."
  • Don't: read your own code back verbatim, or spiral through five half-ideas. If you need to genuinely deliberate, say "let me think for twenty seconds" and go quiet on purpose — a signposted silence is calm; an unexplained one is panic.
  • If you notice yourself rambling, land the plane: "Short version: I'll go with the hashmap. Coding it now."

When you spot your own bug

Say it plainly and re-trace — self-correction is a plus, not an admission of failure:

"I think I have an off-by-one — the loop should stop at n-1, not n. Let me fix and re-trace the last example."

Don't panic-rewrite the whole function when one test fails. Read the failure, form a hypothesis, isolate it (check boundaries first), fix, re-run the trace. Narrate the hypothesis.

Highest-value skeletons — have these in your fingers:

# Hashmap — Two Sum: check complement BEFORE inserting so you never match self
def two_sum(nums: list[int], target: int) -> list[int]:
    seen = {}                                   # value -> index
    for i, x in enumerate(nums):
        if target - x in seen:
            return [seen[target - x], i]
        seen[x] = i
    return []

# Prefix sum + count-map — count contiguous subarrays summing to k
def subarray_sum_k(nums: list[int], k: int) -> int:
    from collections import defaultdict
    seen, running, count = defaultdict(int), 0, 0
    seen[0] = 1                                 # empty prefix — the load-bearing init
    for x in nums:
        running += x
        count += seen[running - k]              # prefixes that close a subarray = k
        seen[running] += 1
    return count
# Gotcha: forget seen[0]=1 and you miss every subarray that starts at index 0.

# Two pointers — on a sorted array, move the side that improves the target
def two_sum_sorted(a: list[int], target: int) -> list[int]:
    lo, hi = 0, len(a) - 1
    while lo < hi:
        s = a[lo] + a[hi]
        if s == target: return [lo, hi]
        if s < target:  lo += 1                 # need bigger → move left up
        else:           hi -= 1                 # need smaller → move right down
    return []

# Sliding window — longest substring without repeats
def longest_unique(s: str) -> int:
    last = {}                                   # char -> last index seen
    left = best = 0
    for right, c in enumerate(s):
        if c in last and last[c] >= left:
            left = last[c] + 1                  # jump past the repeat
        last[c] = right
        best = max(best, right - left + 1)
    return best

# Tree DFS — Path Sum: does a root→leaf path sum to target?
def has_path_sum(node, target: int) -> bool:
    if not node: return False
    if not node.left and not node.right:        # leaf
        return node.val == target
    rest = target - node.val
    return has_path_sum(node.left, rest) or has_path_sum(node.right, rest)

# Tree BFS — level order with a deque
from collections import deque
def level_order(root) -> list[list[int]]:
    out, q = [], deque([root] if root else [])
    while q:
        level = []
        for _ in range(len(q)):                 # snapshot this level's size
            n = q.popleft(); level.append(n.val)
            if n.left:  q.append(n.left)
            if n.right: q.append(n.right)
        out.append(level)
    return out

Skeletons worth memorizing:

# Binary search on the ANSWER — Koko: smallest eating speed to finish in h hours
import math
def min_speed(piles: list[int], h: int) -> int:
    def hours(speed): return sum(math.ceil(p / speed) for p in piles)
    lo, hi = 1, max(piles)
    while lo < hi:                              # find leftmost feasible speed
        mid = (lo + hi) // 2
        if hours(mid) <= h: hi = mid            # feasible → try slower
        else:               lo = mid + 1
    return lo

# Graph BFS flood-fill — Number of Islands
def num_islands(grid: list[list[str]]) -> int:
    if not grid: return 0
    R, C, count = len(grid), len(grid[0]), 0
    def sink(r, c):
        q = deque([(r, c)]); grid[r][c] = '0'
        while q:
            i, j = q.popleft()
            for di, dj in ((1,0),(-1,0),(0,1),(0,-1)):
                ni, nj = i+di, j+dj
                if 0 <= ni < R and 0 <= nj < C and grid[ni][nj] == '1':
                    grid[ni][nj] = '0'; q.append((ni, nj))
    for r in range(R):
        for c in range(C):
            if grid[r][c] == '1':
                count += 1; sink(r, c)
    return count

# Topological sort — Course Schedule (Kahn's BFS); cycle iff not all nodes processed
def can_finish(n: int, prereqs: list[list[int]]) -> bool:
    from collections import defaultdict
    graph, indeg = defaultdict(list), [0] * n
    for course, need in prereqs:
        graph[need].append(course); indeg[course] += 1
    q = deque(i for i in range(n) if indeg[i] == 0)
    done = 0
    while q:
        u = q.popleft(); done += 1
        for v in graph[u]:
            indeg[v] -= 1
            if indeg[v] == 0: q.append(v)
    return done == n

# Top-k with a size-k min-heap — kth largest
import heapq
def kth_largest(nums: list[int], k: int) -> int:
    h = []
    for x in nums:
        heapq.heappush(h, x)
        if len(h) > k: heapq.heappop(h)         # drop the smallest → heap holds k largest
    return h[0]

# Backtracking — subsets: choose / recurse / undo
def subsets(nums: list[int]) -> list[list[int]]:
    res, path = [], []
    def bt(start):
        res.append(path[:])                     # copy the current subset
        for i in range(start, len(nums)):
            path.append(nums[i])
            bt(i + 1)                            # advance so we don't reuse
            path.pop()                           # undo
    bt(0)
    return res

# 1-D DP — Word Break: can s be segmented into dictionary words?
def word_break(s: str, words: list[str]) -> bool:
    wordset = set(words)
    dp = [True] + [False] * len(s)              # dp[i] = s[:i] is segmentable
    for i in range(1, len(s) + 1):
        for j in range(i):
            if dp[j] and s[j:i] in wordset:
                dp[i] = True; break
    return dp[len(s)]

For DP specifically, state the recurrence before you type: "Let dp[i] be whether the first i characters split cleanly. dp[i] is true if some j < i has dp[j] true and s[j:i] is in the dictionary. Base case dp[0] = True." Saying the recurrence out loud is half the score on a DP question.

4. Debug this DSA code

The mirror of the ML doc's "broken model" round: sometimes they hand you working-looking DS&A code that's subtly wrong. Spot by symptom, not by re-reading line-by-line — narrate the symptom, form a hypothesis, check boundaries first. These classics cover most planted bugs.

Bug 1 — Off-by-one loop bound

for i in range(len(a)):
    if a[i] + a[i + 1] == target: ...   # BUG: a[i+1] runs off the end

Symptom: IndexError on the last iteration, or a silent miss/overcount at the boundary. Any time you index i+1 inside the loop, the bound must be range(len(a) - 1). First check on a boundary crash: what's the largest index you touch?

Bug 2 — Missing seen[0] = 1 in a prefix-sum count

seen, running, count = defaultdict(int), 0, 0
# BUG: never seed seen[0] = 1
for x in nums:
    running += x
    count += seen[running - k]
    seen[running] += 1

Symptom: Count is low by exactly the number of qualifying subarrays that start at index 0. The empty prefix (sum 0) must be pre-seeded. Sanity check: nums=[3], k=3 should return 1, not 0.

Bug 3 — Storing the value, not the index, in Two Sum

if target - x in seen:
    return [seen[target - x], i]
seen[x] = x                             # BUG: stored the value, wanted the index

Symptom: Returns numbers where indices were asked for ([7, 1] instead of [0, 1]). The map's key is the value you look up; the value is the index you return. Trace one example and read what you actually return.

Bug 4 — Mutating a list while iterating it

for x in items:
    if is_stale(x): items.remove(x)     # BUG: mutating under the iterator

Symptom: Elements get skipped — removing shifts everything left, so the iterator jumps past the next item. Iterate a copy (for x in items[:]) or build a new list by comprehension.

Bug 5 — pop(0) as a BFS queue

q = [start]
while q:
    node = q.pop(0)                     # BUG: O(n) shift every dequeue

Symptom: Correct answers, but O(n²) and it times out on large inputs. list.pop(0) shifts every remaining element. Use collections.deque with popleft() for O(1). Say this out loud even if it "works" on the toy case — naming the complexity trap is the signal.

5. One ML-flavored warm-up (narrate the intent)

For an ML-research team the DS&A problem sometimes wears an ML costume — a set/hashmap dressed as a data-pipeline step. Same patterns, but narrate the ML intent, not just the mechanics: say why the structure matters for a model. Two that land well:

# Feature hashing — map sparse text tokens to a fixed-width vector (the "hashing trick").
# ML intent: bounded memory, no vocab lookup, collisions traded for O(1) featurization.
def hash_features(tokens: list[str], dim: int) -> list[float]:
    vec = [0.0] * dim
    for tok in tokens:
        h = hash(tok)
        vec[h % dim] += 1.0                     # signed variant: use (h >> 1 & 1) for ±1
    return vec

# Dedup a stream of search queries — a set is the exact O(1)-membership answer.
# ML intent: strip near-duplicate queries before they skew training-set frequencies.
def unique_queries(stream) -> list[str]:
    seen, out = set(), []
    for q in stream:
        key = q.strip().lower()                 # normalize before dedup
        if key not in seen:
            seen.add(key); out.append(q)
    return out

Say it like this: "This is a hashmap/set problem underneath — I'll use feature hashing so featurization is O(1) with bounded memory, and I'll flag that collisions are the cost. For a truly massive stream where even the set won't fit, I'd swap in a Bloom filter and name the false-positive tradeoff."

Python fluency signals (reach for these)

Naming the right standard-library tool reads as production habit. Quick reference — details and gotchas are in the Unaided Practice Regimen companion; the load-bearing ones for a live round:

Tool Use it for One-liner
collections.Counter frequency counts, top-k Counter(s).most_common(k)
collections.defaultdict adjacency lists, grouping defaultdict(list) — no "key exists?" checks
collections.deque BFS queue, sliding window O(1) append/popleft (a list's pop(0) is O(n) — say this)
heapq top-k, k-th, merge-k, median min-heap; push -x for max-heap; push tuples (priority, item)
bisect binary search on a sorted list bisect_left / insort in O(log n)
functools.cache memoize recursion (DP) one decorator turns a recursion into top-down DP

Cheap seniority signals while coding: type hints on the signature (def two_sum(nums: list[int], target: int) -> list[int]:), guard clauses / early returns for edge cases up top, enumerate instead of range(len(...)), tuple unpacking (a, b = b, a + b), and "".join(parts) instead of += in a loop (which is O(n²) on strings — worth naming).

Trap: the mutable-default-argument bug — def f(x, acc=[]) creates the list once at def time and shares it across calls. If you ever need a default list/dict, write acc=None then acc = acc if acc is not None else [] (not acc or [], which also wipes a deliberately-passed empty list). Being able to explain this on demand is a classic Python probe.

The one-line summary to internalize

Correct-but-silent scores below partial-but-clear. Frame → clarify → state approach and complexity → code in narrated steps → hand-trace → wrap complexity in three sentences → propose one improvement. When stuck, secure a working baseline and say where you're blocked. When hinted, paraphrase it back and change course visibly.
