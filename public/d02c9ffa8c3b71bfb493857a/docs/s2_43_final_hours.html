Final-Hours Cheatsheet

The one screen to read in the ~4 minutes before you walk into each round. Recall only — everything here is proven elsewhere in the pack.

THE ONE RULE

  • Never invent a specific. No number, name, date, threshold, or deployment status you can't source. If unsure, say the range or say "I'd have to check" — caveat-first beats confident-wrong.
  • Caveat-first, then pivot. Own the edge in one clause, then move to what you do know cold. Underclaiming credibly beats overclaiming; this loop deep-probes.
  • Derive, don't recite. If they push past a memorized fact, fall back to first principles out loud.

SAY-IT-COLD — my own numbers (one line each)

  • LLM Router — >90% cost reduction at matched ("iso") quality.
  • MERA — 87.3% router-decision accuracy at 51.8% serving cost. (never say "task accuracy")
  • TwinRouterBench — 53%, trained router vs unrouted Opus, matched resolve, dynamic track.
  • Diffusion Weights — ~3.2x wall-clock fine-tune speedup (generative training).
  • Pruning nnU-Net — >80% weights removed at proxy Dice >0.95, unstructured, no measured speedup. MIDL, first author.
  • Eye-Tracking VR emotion dataset — first author, 240 Hz gaze, 7 Ekman emotions, 26 participants; shared on request; public artifact = the open-source Unity tool.

THE HONEST-GAP ONE-LINER (say it once, clean)

"The efficiency side I've owned end to end — routing, pruning, generative training acceleration — so inference-optimization and cost levers are second nature. What I haven't personally run is a large-scale production on-call rotation with live canary and real-time drift monitoring; my deployments have been research and shadow-eval scale. But I know exactly how that machinery works — shadow → canary → A/B with automated rollback, drift on both input and decision distributions, proxy metrics while labels lag — and it's the fastest gap I'd close."

CODING — narrate-while-you-code (one beat per line)

  1. Restate the problem + confirm 1-2 examples and edge cases out loud.
  2. Brute force first — say it and its complexity, don't code it.
  3. Optimized plan — name the pattern + target O() before typing.
  4. Narrate as you type — one sentence per block: what it does, why.
  5. Dry-run on a tiny example, tracing state.
  6. State time/space complexity unprompted.
  7. Test edges: empty, single, duplicates, negatives, overflow.

From-scratch primitives to have cold (NumPy, out loud):

  • Stable softmax + cross-entropy — subtract row max (overflow); +1e-12 inside log (log 0).
  • Scaled dot-product attention — QKᵀ/√d, causal mask = -inf on future, softmax, ×V.
  • Precision / recall / F1 + AUC — guard divide-by-zero; AUC = P(pos ranked above neg).
  • BPE merge step — greedily merge most-frequent adjacent pair; order matters.
  • Welford mean/var + z-score — one-pass stable; it's what LayerNorm does inside.
  • kNN / k-means / logreg+GD — guard empty cluster; BCE gradient is (p − y).

Trap: don't burn clock on "let me recall the formula." If it stalls, derive the shape from first principles while narrating.

ENGINEERING — cheat blocks

Training memory (mixed-precision Adam)

2 (bf16 weight) + 4 (fp32 master) + 4 (fp32 grad) + 8 (Adam m,v) = 18 bytes/param
7B → ~126 GB of state (before activations) → why it won't fit one GPU

ZeRO paper quotes 16 because it assumes fp16 grads — the 16-vs-18 gap is only gradient precision. Flag it if pushed. Fifth bucket = activations (scale with batch×seq); tame with activation checkpointing (+~33% compute, lowers MFU).

KV-cache memory (know cold)

KV bytes = 2 × layers × kv_heads × head_dim × seq × batch × bytes_per_elem
           K,V ↑        ↑ use kv_heads (GQA), NOT n_heads              2=bf16 / 1=int8 / 0.5=int4
  • Llama-3-70B, bf16, 4K, batch 1 → ~1.34 GB/seq; batch 32 → ~43 GB.
  • Grows linearly in seq × batch (long context + concurrency = the killers).

    GQA trap: using n_heads instead of kv_heads is an 8x error (Llama-2-70B: 8 vs 64).

Quantization (fast recall)

Precision vs BF16 Quality Use when
FP8 ~1.5x, ½ mem ~0.6 pt loss Default on Hopper/Blackwell; quality-critical
INT8 modest over FP8 comparable Rarely the clear winner on modern GPUs
INT4-AWQ ~2.7x <2 pt MMLU, −8 pt HumanEval VRAM-bound; not code/math/reasoning

AWQ < GPTQ perplexity at INT4 (protects salient weights). FP8 first, INT4 only when memory forces it.

Drift — PSI thresholds

  • < 0.1 stable · 0.1–0.2 moderate, watch · > 0.2 significant, act.
  • PSI/KS on inputs = no labels needed. KS is hypersensitive at millions of rows — lean on effect size. Data drift → maybe retrain; concept drift → target moved, rethink model.

Inference cost ladder (cheapest win first)

quantize (FP8) → continuous batching + PagedAttention (40%→90% util) → prefix cache → speculative decode → ROUTE

The biggest lever is often not touching the model — route the easy query to the small model. That's my edge. Spec-decode is a latency win, not throughput (can hurt under high batch load).

SYSTEM DESIGN — 8-step spine

Clarify → Metrics → Data/Features → Model → Serving → Experiment → Monitor → Trade-offs.

  • Lead with a business metric (GMV / CTR / conversion / latency SLA) — then ML metric → offline proxy. Never open with architecture.
  • Latency-budget breakdown, never one global number. Search E2E ~500 ms; the ranking model itself ~120–200 ms. Split the budget across retrieval → features → model → post.
  • Trade-off at every layer; end scaling answers with "what breaks first + cheapest lever."

    Trap #1 silent-fail: thinking like a researcher, not a production engineer. Say cost, SLO, rollout, monitoring.

WALKTHROUGH — "Cost-Efficient LLM Routing at Scale" (8 beats, ~12 min)

  1. Bottom-line first — the result, then the story (>90% cost cut at iso quality).
  2. Problem — serving big models on every query is the cost floor; most queries don't need it.
  3. Scope — non-goals up front (no tail-latency phase 1); prove cost/quality frontier before productizing.
  4. Approach — benchmark → simple router → skill-adaptive; each phase gated on a measured result.
  5. Key decision under uncertainty — learned router over cascade (escalation-tax reasoning); kept cascade as fallback.
  6. Results — quantify everything: MERA 87.3% decision acc @ 51.8% cost; TwinRouterBench 53%.
  7. Retrospective — build the online/shadow eval harness earlier; offline-first hid calibration under real traffic.
  8. Scale to eBay — sub-10 ms shardable router, drift on the decision mix, retrain cadence, SLA knob; map onto LiLiuM/e-Llama zoo.

STAR timing: Situation 20% · Task 10% · Action 60% · Result 10%. Say "I", not "we". Quantify every result.

Confirm before the room: deployment status (offline vs live), exact thresholds, stakeholder/timeline specifics — flag, don't invent.

eBay — why + values

Why: eBay already lives the systems-race thesis — LiLiuM's custom tokenizer (~34% faster generation), e-Llama (continued pretraining at 10% LR to avoid forgetting), a fine-tuned Gemma3-4B (F1 50.5) beating a zero-shot 27B (44.8) at ~3.8x faster. That small-specialized-beats-big-general economics is exactly what I publish on; I'd put a router in front of that fleet.

5 values: Customer Focus · Innovate Boldly · Be For Everyone · Deliver With Impact · Act With Integrity. (Pin one quantified STAR story to each; use "Deliver With Impact" for the routing numbers, "Act With Integrity" for the honest-gap.)
